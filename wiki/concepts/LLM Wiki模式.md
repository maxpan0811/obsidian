# LLM Wiki 模式

一种利用 LLM 长期维护知识库的模式，由 Andrej Karpathy 在 2026 年 4 月提出。核心思想是让 LLM 从被动的"检索员"变为主动的"知识编译与维护者"。

本 wiki 即基于此模式运作。

## RAG vs LLM Wiki

| | RAG | LLM Wiki |
|--|:---:|:--------:|
| 查询方式 | 每次从原始文档检索 | 从已编译的 wiki 查询 |
| 知识积累 | 无，每次都从零开始 | 持续累积，交叉引用已建立 |
| 维护成本 | 低（无需维护） | 近零（由 LLM 维护） |
| 知识复利 | 无 | 每次摄入和查询都在丰富 wiki |
| 类比 | 每次出门现查地图 | 助手每天更新你的私人地图 |

## 三层架构

```
raw/          ← 用户维护，只读不修改
  ├── 文章、PDF、笔记、数据文件
  └── 不可变，作为事实来源
  
wiki/         ← LLM 全权维护
  ├── 来源摘要、概念页、实体页
  ├── 分析页、产品页、功能页
  ├── 索引（index.md）
  └── 内部链接形成知识网络

schema/       ← 用户 + LLM 共同演进
  └── CLAUDE.md / AGENTS.md（规则与工作流）
```

## 三种核心操作

### Ingest（摄入）

新资料进来时的标准流程：阅读 → 讨论 → 写来源摘要 → 更新相关概念/实体页 → 更新索引 → 记录日志。一份资料可能触及 10-15 个 wiki 页面。

### Query（查询）

查询时先读索引找到相关页面，再综合回答。高质量回答可以归档回 wiki 成为新页面，让探索也产生复利。

### Lint（健康检查）

定期检查：页面间矛盾、过时声明、孤立页面、缺失交叉引用、术语不一致、需要补充的数据空白。

## 关键理念

> 人类放弃维护知识库，是因为维护成本增长快于知识价值。LLM 改变了这个等式。

LLM Wiki 的精髓不在于技术，而在于**重新分配了人类和 LLM 的工作**：
- **人类负责**：策划资料、指导分析、提出好问题、思考意义
- **LLM 负责**：总结、交叉引用、归档、记账——所有维护工作

## 四层知识价值链

> 来源：[[sources/从会话管理到知识库-四层进阶指南|墨染四层进阶指南]]

层与层之间严格依赖，多数"AI 记忆"方案试图从存跳到用，跳过中间层导致检索出来的东西读不懂、信不过、用不上。

| 层 | 核心问题 | 检验标准 |
|----|----------|----------|
| 存得下 | 数据可见、格式可读 | — |
| 读得懂 | 打开文件几秒内知道讲了什么 | 5秒内判断相关性 |
| 找得到 | 两三百个文件里直接命中 | 不记关键词，3步以内找到 |
| 炼得出 | 从对话流提取独立可复用知识单元 | 搜到卡片直接引用，无需翻原对话 |
| 用得上 | 知识回流到工作流 | 过去一月实际引用过；AI上下文未随档案线性膨胀 |

### 三类值得提炼的卡片

- **Decision**：选了什么、为什么、**否决了什么**（否决理由是最高价值信息）
- **Pattern**：反复出现的有效做法，可迁移
- **Pitfall**：踩过的坑、反直觉行为

卡片质量分级：AI 提取默认 `unreviewed`，人工确认后标 `verified`。

### AI 上下文分区加载

避免档案随时间线性膨胀、无关内容抢占上下文的问题：

| 区域 | 内容 | 时机 |
|------|------|------|
| **热区** | 项目状态+最近决策+待办+已否决方案 | 每次启动必读，≤300行 |
| **温区** | 完整档案+行为规则 | 按需加载 |
| **冷区** | 历史对话原文 | 精确检索时才动用 |

热区不是档案缩写，是 briefing：五行讲清状态、关键决策、下一步、已排除方向。

### 三条跨层设计原则

1. **今天做比明天做又准又省** — 上下文热的时候生成摘要/卡片最准
2. **否决记录和执行记录一样重要** — 没有否决理由，几个月后必然重走弯路
3. **简单方案优先，阈值到了再升级** — MOC < 200篇时优于 Dataview；全文搜索 < 50会话时够用
