### 捷径辅助工具包，让捷径制作更简单

**@Minja：**捷径制作是一件好玩的事。相对于编程，做捷径更像是用种类有限的乐高积木搭出厉害的家伙。

![](.evernote-content/500C9DC8-9C22-4841-B452-55E100C6CA9F/7D3AAB55-6FE5-4917-A1D8-03F422148E9A.gif)

乐高搭的差分机，可以算高次方程

在制作中，我发现有些操作，经常重复；还有些操作，已经超出了捷径的能力范围。我逐渐摸出了 3 类辅助工具，让捷径制作更简单：

1. 用于调试的捷径
2. 高亮代码的捷径
3. 辅助处理代码的第三方工具

#### 调试工具箱

制作的捷径多了，我发现自己常常拿几个操作拼成临时捷径，用来生成数据，比如：

* 检查日期格式是否符合预期
* 把图片处理成 base64 编码 1
* 为特定捷径生成 URL Scheme
* ……

有段时间我是用完就删，次数多了，忽然想，为什么不把常用的操作汇集到一个捷径里呢？慢慢地，我就组合出一只捷径调试工具箱，覆盖了常见的调试需求。

**使用方式很直白，直接把需要查看或转换的素材填进开头的文本框** 2 **，然后选择一种转换方式，就能查看结果。有需要的话，还可以根据提示把运行结果拷贝到剪贴板。**

![](.evernote-content/500C9DC8-9C22-4841-B452-55E100C6CA9F/E0AF757A-70FD-484F-9EC5-DEEFF53C1D26.png)

覆盖了常见调试需求的捷径

[> 动作下载](https://www.icloud.com/shortcuts/bd1a101ba7f741d6a357a3cba9ec56f7)

整个捷径是由文本框、一大串列表和拷贝操作组成的，你可以在列表里继续增加自己需要的操作。

#### 两个代码美化捷径

注：复杂代码推荐使用下一节的 Kodex 和 Jayson，本节小工具用于临时「扫一眼」。

各种扒视频、扒图片的捷径，可以说是大家最喜闻乐见的。但是这些和网页交互的捷径，我不是很乐意亲自制作：分析网页数据是一件枯燥的工作，而且页面一旦升级改动，捷径可能就挂了。

![](.evernote-content/500C9DC8-9C22-4841-B452-55E100C6CA9F/D162E3C5-E854-4DB3-A08E-D2987396A7DF.jpg)

各种网页的捷径，用着爽，做得累

说分析网页繁琐，基本是因为捷径显示网页数据的样式太马虎：

1. 没有代码高亮
2. 代码挤成一团（主要是「获取网页内容」返回的 JSON 数据）

这样一来，就很难搞清数据的结构，更难找到我们想要的数据。

![](.evernote-content/500C9DC8-9C22-4841-B452-55E100C6CA9F/4F42AF75-0777-4EB6-88F2-32F17BBB15AB.jpg)

捷径自带的 JSON 数据显示方式非常原始

本来我是找在线服务「美化」代码的，但在 iPhone 上，这些网页本身的使用体验就够我喝一壶，于是寻思着用捷径本身来解决。思考的过程很简单：想呈现格式化或者染色过的代码（代码高亮），肯定避不开富文本、HTML，而捷径又支持在 HTML 里运行 JavaScript，那搞个插件就好了。

之前假期捣鼓了一下，两个捷径就出来了：

第一个，**代码高亮**：选择需要的编程语言、输入代码，即可显示高亮后的代码。这个捷径调用了 [highlight.js](https://highlightjs.org/)，支持 185 种语言（捷径制作中最常见的 HTML、JavaScript 自然不在话下）。

![](.evernote-content/500C9DC8-9C22-4841-B452-55E100C6CA9F/BED45AA5-F854-41AF-A0D7-CDF623D9F01E.png)

代码高亮

[> 动作下载](https://www.icloud.com/shortcuts/89936f0e2f1741cba861b8e1460534e3)

第二个，**JSON 美化**：本捷径调用了 JSON.stringify，将原本一团糟的 JSON 数据换行排好版，并且高亮显示出数据结构。如果你不喜欢我提供的配色，可以根据注释在开头的字典中配置自己想要的高亮方案（用颜色的英文名或色值）。

![](.evernote-content/500C9DC8-9C22-4841-B452-55E100C6CA9F/82B4A7E8-39BC-4779-BE81-6669A28ECC72.png)

#### 备用的第三方工具

前两类工具都是用捷径实现的，但在制作复杂捷径时，捷径 App 的一个天生弊端就要暴露出来：**你必须不断在调试用的捷径和正在制作的捷径之间来回跳**…… 这时候最好开个第三方 App，把调试工作转移进去，避免打搅捷径的正常制作。

我最常用的工具是：

* **RegEx Lab**：检查正则表达式。尝试用捷径匹配文本之前，先检查一遍会比较有底。
* **Kodex**：查看和编辑各种代码。主要用于看 JavaScript、HTML 这些移动端也可以运行的代码，还有些代码是 Shell 写长、通过捷径 SSH 到电脑上运行，比如远程关机、一键休眠，也可以用 Kodex 排查一下明显的问题。
* **Jayson**：查看 JSON 数据。很多网页传回来的数据都是 Jason 格式的，如果比较复杂，需要对照着数据不断修改捷径，我还是会用 Jayson。

![](.evernote-content/500C9DC8-9C22-4841-B452-55E100C6CA9F/7F9A1350-B98C-4A34-A854-671272173A96.jpg)

我的第三方备用工具

RegEx Lab 是一个轻量的正则表达式检查工具。正则表达式在复杂捷径的制作过程中甚为关键，**可以实现 `case` 语法** 3 ，《[如何用 RSS 实现个性化订阅 | 实用技巧](https://sspai.com/article/51956?series_id=70)》中我就用正则表达式把几百个操作简化到了十几个。不过正则表达式在捷径里修改、查看起来都不够直观，得写一次、运行一次、查看一次。稍微长一些的表达式，我就交给 RegEx Lab，可以边写边观察匹配情况，更加省力。

![](.evernote-content/500C9DC8-9C22-4841-B452-55E100C6CA9F/8D39FDD3-DD5C-4774-9D24-5C9C9C164BAC.gif)

RegEx Lab 可以边写边检查匹配到了哪些文本

Kodex 在 [奏折 03](https://sspai.com/article/52208?series_id=70) 里有过介绍。作为代码编辑器，Kodex 比较全能，捷径制作只是它的众多功能之一。简单的网页，可能直接用捷径扫一眼染色后的数据，就撸袖子开工了；复杂一些的网页，最好把代码复制到 Kodex 里面看（建议先 [在线排版](http://jsonviewer.stack.hu/) 一下）。如果有条件在 iPad 上制作捷径，一边是 Kodex 做参考，一侧开着捷径 App，会有一种手动作坊般的亲切。

![](.evernote-content/500C9DC8-9C22-4841-B452-55E100C6CA9F/BE690704-079D-4231-96CB-8540E70E3336.png)

一边是 Kodex ，一边是捷径 App

最后一个 JSON 数据检查工具 Jayson，在 [第 8 期奏折](https://sspai.com/article/52809?series_id=70) 中已经详细介绍过了，经常需要处理复杂 JSON 数据的读者可以备一个。

制作一个或大型、或复杂的捷径，不仅是为了一劳永逸，也是为了追求一种成就感。捷径的作用固然很有限，调试起来也不算方便，不能和正经的编程工具相比；但是这种用乐高积木搭建出高楼大厦的成就感，没亲历过的人是很难体会的。

### Setapp 新上架：一键网页转应用 Unite

![](.evernote-content/500C9DC8-9C22-4841-B452-55E100C6CA9F/6C4AEACD-7BD1-4213-BBA1-7DF0F38317B2.png)

**Unite**

[官网链接](https://www.bzgapps.com/unite)

#### 应用简介

**@沨沄极客：**Unite for macOS 是一款将网页生成 macOS 本地应用的小工具。它和 Chrome 的生成网页应用功能（PWA）类似，但两者有很多不同之处，Unite 采用了完全独立的设置、自定义和 Cookie，**所以每一个生成的应用都相当于是一个独立的浏览器**。

![](.evernote-content/500C9DC8-9C22-4841-B452-55E100C6CA9F/C7670B65-D831-43BF-BFC9-EFDDA2D347B7.png)  

生成一个 Power+ 本地应用

比如把 Power+ 的文章列表：`https://sspai.com/series/70/list` 作为一个单独的应用，Unite 能够自动抓取网页的图标。生成应用后只需登陆一次，以后就会自动登录，作为本地客户端使用。

![](.evernote-content/500C9DC8-9C22-4841-B452-55E100C6CA9F/D092345E-E0E8-4FF7-8B51-1B4C2F75D1DD.png)  

生成的效果，需要登录一次

更独特的是，Unite **支持非常多的自定义功能**，像是「窗口透明度、默认字体、创建浮动窗口」等，你可以对每一个网页应用分别设置。还**允许你将网页添加到状态栏，点击状态栏就可以呼出界面**。这个功能很适合 Twitter：`https://twitter.com`、微博 Lite：`https://m.weibo.cn` 等网页。

![](.evernote-content/500C9DC8-9C22-4841-B452-55E100C6CA9F/082D9B6E-F72A-40D4-A9BB-A31202023CD4.png)  

生成一个微博 Lite

为了使网页应用更接近本地客户端的体验，Unite 还提供了在通知中心推送、用户脚本、页面白名单等高级功能。使用时需要注意，**不要将网页命名为电脑中已有的应用**，这会将原有应用直接覆盖。

**@Fairyex：**如果你经常看 Vlog 的话，即使没听说过「Jump cut（跳接）」，肯定也看过下面这样的剪辑手法：

来源：carykh

简单来讲，跳接就是按照一定逻辑将不同时间的片段粗暴地拼在一起。在 Vlog 中常见的是将主角说话中比较长的停顿剪掉，以提升谈话的流畅性；另一种常见的场景则是定格视频拍摄。

专注于自己动手实现机器学习应用，解释背后原理的 YouTube 频道 carykh 最近上传了视频[「Automatic on-the-fly video editing tool!」](https://www.youtube.com/watch?v=DQ8orIurGxw)，介绍了如何利用机器学习来实现自动剪辑 Jump cut 视频，并把成果开源到了 [GitHub](https://github.com/carykh/jumpcutter) 上。

下载 jumpcutter 并在电脑上装好 Python3 环境，接着在应用目录调用终端（CMD），输入：

```
python jumpcutter.py -h
```

获取所有可以调整的参数，再使用命令：

```
python jumpcutter.py --input_file 要剪辑的视频位置 --output_file 剪好的视频存放位置 你想要的任何参数
```

### Procreate 4.3：增强独立工作的能力

![](.evernote-content/500C9DC8-9C22-4841-B452-55E100C6CA9F/B746D48D-D087-469D-96A3-0D5049B62A69.png)

**Procreate 4.3（2019–04–16）**

[下载链接](https://itunes.apple.com/cn/app/procreate/id425073498?mt=8)

**@Minja：**这是一次解决创作「最后两站路」痛点的更新。

Procreate 在绘画方面的地位无可争议，连手机版都可以拿下 App Store 的年度奖。但是创作者仍然很难离开 Photoshop，比如加文字，比如调光效，比如做动图…… 这些点睛的操作，都让人很难从 Ps 那儿断奶。

而这次更新，就着重在解决其中相当一部分问题：

* 支持导入字体
* 支持调整图层样式
* 支持导出 PDF 和 GIF 等更多格式
* ……

每一项，都让 Procreate **独立工作**的能力更强。

字体的重要性不言而喻，尤其是对于需要绘制海报、封面的画师而言，能够直接使用自己习惯的字体可以节省很多时间。字体导入和笔刷一样，直接 Drag & Drop 一下即可，完事儿还能在 Procreate 里慢慢调细节。

![](.evernote-content/500C9DC8-9C22-4841-B452-55E100C6CA9F/E75501AA-0EF1-4C23-B111-091D0E396848.png)

导入字体

图层样式（更暗、更亮、混合等等）则是渲染氛围的重要工具。单纯靠笔刷绘制的图片可能有点「素」，这时候调整一下光效就比较容易出效果。当然图层样式的用法远不止这点，总之需要「调味」的时候，就可以用上这套新功能。

GIF 格式的支持则大大方便了动图创作者。以往绘制静态图片原图后，免不了导入 Photoshop 制作 GIF。有不少画师从事表情包创作，这类轻量级的工作如果还要移到电脑上完成，实在很不自在。Procreate 现在不仅可以导动图，还可以调整简单的帧率、尺寸，足以应对复杂度要求不高的作品。

![](.evernote-content/500C9DC8-9C22-4841-B452-55E100C6CA9F/E1F40FA3-BCE2-49D8-A658-EDC888B2E912.gif)

导出动图（图是我画的）

此外还可以看到一批操作方式上的改进，比如**单图层显示模式**（长按锁定按钮即可开启），就能在图层叠嶂的画作中让人看清当前在画什么……

### Reeder 4 集成 Bionic Reading

**@Minja：**Reeder 4 的公测还在进行，这周又加了一个重磅功能：**[Bionic Reading](https://bionic-reading.com/en/)（BR）**，即根据词义自动加粗开头部分字母，以达到提升阅读速度的目的。

![](.evernote-content/500C9DC8-9C22-4841-B452-55E100C6CA9F/5519F80B-D390-4901-AC54-2B99FC397A17.png)

Reeder 4 的 Bionic Reading 的功能

据说这样加粗之后，看到前几个字母，大脑就会自动补全整个单词（当然得有一点点英文底子）。作为非英语母语使用者，我的确感觉阅读速读有提升，以往可能会想几秒某个词素是啥意思，开启 Bionic Reading 后就不会想太多；特别是读一些浅显的科技类文章，开了 Bionic Reading 后几乎可以「扫」过去。

除了集成在 Reeder 中，Bionic Reading 也有 Chrome 和 Firefox 的扩展插件，想体验的读者现在就可以试试。

![](.evernote-content/500C9DC8-9C22-4841-B452-55E100C6CA9F/5FAF405B-D208-49B8-A30A-010B2049B14A.png)

Bionic Reading 的浏览器插件

不过，我把 Bionic Reading 推荐给几位美国朋友后，他们似乎感觉没什么用处，反倒认为文章整体排版变差了，像快没墨的打印机一样断断续续…… 有意思的是，「后续」开发者也做了一个[中文版的「Bionic Reading」](https://m.weibo.cn/status/4362353674032211?)，结果不少操中文母语的网友表示提升不大，似乎和 Bionic Reading 那边的情况差不多。也许只在对一门语言不十分熟练时，Bionic Reading 这类技术的辅助效果更明显。

### Play 商店新 UI：底栏与侧栏菜单对比

**@Fairyex：**最近 XDA 上有人发帖[爆料](https://www.xda-developers.com/google-play-store-material-theme-redesign-apex-packages-android-q/)了 Google play 商店测试的新 UI：

![](.evernote-content/500C9DC8-9C22-4841-B452-55E100C6CA9F/5D1A8458-8CB7-4284-8D3C-873167AA7E64.jpg)

新 UI

Google 最近正在努力将自家所有的应用界面升级到 Material Design 2.0，但一直没有对最重要的应用商店动手，现在终于着手开始改变商店的 UI。除了界面变成现在流行的纯白以外，最明显的改变就是新 UI 添加了 Material Design 2.0 里面的底栏，但没有移除之前的侧栏菜单，所以现在 Play 商店**同时拥有底栏和侧栏汉堡菜单**。

![](.evernote-content/500C9DC8-9C22-4841-B452-55E100C6CA9F/138B7DEC-1257-4570-B5C2-6B0C0C4557B9.jpg)

底栏与侧栏菜单

有的人很讨厌应用底栏，觉得底栏不仅不好看还会让手机多一个下巴；有的人讨厌侧栏汉堡菜单，不能快速切换页面还很难划出来。

![](.evernote-content/500C9DC8-9C22-4841-B452-55E100C6CA9F/B220F7BA-06F6-4F33-99FC-02BA1AC849F6.jpg)

评论

而 Google 在 Material Design 2.0 中对这两样东西做了很好的平衡。设计规范规定底栏适合需要快速切换的分类，但页面下滑后要隐藏，避免了底栏用户界面的问题；汉堡菜单适合放各种不需要经常点击并且数量较多的分类，隐藏起来后整个应用看起来变得更「轻」。两样东西同时搭配使用才能达到信息量与美观的较好平衡，所以我们可以看到 Google 自家很多应用都是底栏与侧边菜单并存。

对比之前 Android 5.0 以下看起来像是工程机的 UI，现在的 Android 界面不仅变得更精致，而且形成了自己的风格，真正变得好看又好用。

### 将 iPad 作为外接显示器？其实是不想浪费这块好屏幕

**@文刀漢三：**9to5Mac 最近有一篇[报道](https://9to5mac.com/2019/04/16/mac-ipad-display-feature/)，称苹果将会在今年发布的 macOS 10.15 里推出类似 [Luna Display](https://sspai.com/post/52706) 的功能，让你可以将 iPad 变成 Mac 的外接显示屏。

这篇报道提到了很多实现的细节，比如当你将光标放在窗口的绿色按钮上时，会弹出两个选项，包括传统的全屏功能，以及将窗口发送到外接显示屏上。其中，外接显示屏可以选择 iPad 作为发送目的地。

少数派之前有介绍过 [Duet Display](https://sspai.com/post/27758) 和 [Luna Display](https://sspai.com/post/52706)，都可以实现类似的功能。很多人不懂为什么不买多一块真正的外接显示器，而是要想尽办法把 iPad 当成外接显示器呢？

其实他们搞错了这些工具的目标用户，这些工具要瞄准的不是那些长期需要开着多个窗口的人（这些人当然更建议买多一块显示器），而是那些需求比较轻量的人：

1. 偶尔有多窗口需求
2. 刚好有 Mac 和 iPad
3. **不想浪费 iPad 的这块好屏幕**

第三点尤为重要，因为 iPad（尤其是 iPad Pro）确实是一块非常出色的显示屏 —— 分辨率高、色域广、色准好。当手边有这么一块出色的显示屏时，你自然不想浪费它。为了这些偶尔出现的需求去买一块占地方的显示器，也显得不现实。

当然很多测评都有提到，这些第三方工具最大的问题仍是流畅性得不到保证，时常会有延迟掉帧的现象。此外，当你将一套原本用于键鼠交互的系统直接投屏到触屏设备上，两种交互方式杂糅在一起带来的不适感，是第三方厂商无法解决的，只能由苹果从系统层面上去进行优化。我作为一位同时拥有 MacBook Pro 和 iPad Pro 的用户，非常期待苹果在今年夏天推出的新功能。

### 网站：PWA 软件目录

**@Fairyex：**PWA（Progressive Web App，渐进式网页应用），是一个由 W3C 推出，Google 大力推广的网页转应用方式。核心目标是提升网页应用的性能，改善用户体验，并通过调用系统支持的 API 扩展网页应用的功能。

如果用 Chrome 打开一个支持 PWA 的网站（比如 Twitter），登录后就会弹出提示框问你是否选择把 Twitter 添加到桌面，确认后 Chrome 就会生成 Twitter 的网页应用并添加桌面图标。

![](.evernote-content/500C9DC8-9C22-4841-B452-55E100C6CA9F/534A94AA-B648-4CDD-896B-87599DBA7632.jpg)

甚至还能在系统应用管理中管理

比起单纯的网页版，PWA 应用的体验几乎与官方客户端一样，还有着不占内存、不后台运行、获取权限较少等优点。随着 PWA 应用与支持的平台（只要能装 Chrome 都支持）[越来越多](https://developers.google.com/web/progressive-web-apps/desktop)，如何找到好用的 PWA 应用也成了问题。

所以一些收录 PWA 应用的项目应运而生，我体验了 [PWA Directory](https://pwa-directory.appspot.com/)、[PWA Room](https://progressivewebapproom.com/)、[PWA Rocks](https://pwa.rocks/)、[Appscope](https://appsco.pe/)、[OutWeb](https://outweb.io/) 等 PWA 目录网站，其中一部分存在着分类不清晰、收录数量太少、没有中文 PWA 应用的缺点。而推荐大家使用的是 PWA Directory（收录较全）与 OutWeb（体验较好）。
