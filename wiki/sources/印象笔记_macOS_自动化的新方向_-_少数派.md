macOS 自动化的新方向
=============

macOS 的自动化环境已经不温不火了许多年，不仅如此，今年 Catalina 的发布还给一大批系统内置自动化脚本判了[死缓](https://developer.apple.com/documentation/xcode_release_notes/xcode_11_beta_7_release_notes?preferredLanguage=occ)，令人担心 macOS 自动化的未来走向。

与此同时，借助跨平台的 [Catalyst 框架](https://developer.apple.com/documentation/uikit/mac_catalyst)，iOS 端的自动化工具已陆续登上了 macOS 这片旧大陆。这些从 iOS 乔迁过来的时髦工具中，有已经上架的 [Drafts](https://apps.apple.com/cn/app/drafts/id1435957248?mt=12) 和 [Jayson](https://apps.apple.com/cn/app/jayson/id1468691718)，也有频频在开发者 Twitter 中露脸的用 Catalyst 框架实现的[第三方快捷指令](https://twitter.com/stroughtonsmith/status/1135956331360346113)。我们不禁想问，这些移动端的工具会给 macOS 带来哪些变化？macOS 需要这些 iOS 移民吗？

答案都是乐观的。在传统「脚本流」自动化发展停滞的背景下，iOS 新秀们带来了更加现代的**图形化**创作方式与更具**云属性**的自动化功能，将把 macOS 带入下一个自动化时代。

欢迎所有用户的图形化界面
------------

在不会代码的人眼中，「自动化」似乎是高级用户的专利。其实，每个人或多或少都有自动化的意识 ——Word 生成目录是自动化，PPT 批量转 PDF 是自动化，就连自动领红包的小插件，也是自动化 —— 只是，macOS 中的很多自动化需求要靠代码实现，因而将一群心有余的用户拒之门外。

在你学习某门编程语言之前，你会为了分享阅读清单而捣鼓一段代码吗？号称像自然语言一样易懂的 AppleScript 真的一看就懂？用 [Karabiner-Elements](https://pqrs.org/osx/karabiner/) 修改一枚按键的映射对你来说是不是也充满了挑战？如果你的回答是「不会」「不懂」「有挑战」，不必感到羞愧，绝大多数人都和你一样。

macOS 自动化的过去属于代码高手，而下一个时代将引入更多的图形化编辑方式，逐渐取消「代码」这一道门槛。

### 回顾前图形化时代：Automator

2005 年以前，macOS 的自动化几乎被脚本统治着。伴随着 [OS X Tiger](https://zh.wikipedia.org/wiki/Mac_OS_X_Tiger) 的发布，苹果的自动化项目经理 [Sal Soghoian](https://en.wikipedia.org/wiki/Sal_Soghoian) 交出了他的第一款作品：**Automator**，一款图形化的自动化工具。对于当时主要依赖代码完成自动化流程的 macOS 用户来说，Automator 诞生所带来的轰动绝不亚于近几年 Workflow 的问世。看看 Automator 和同门的 AppleScript，你更愿意使用何者来创作自动化动作？「常识」告诉我们，一般来说都是图形化的 Automator 更有吸引力。

![](.evernote-content/86047BE9-F4C6-40B3-92C6-B463CDA03DE7/05D04AF8-D557-449A-9E6A-69130F7829C8.png)

AppleScript 和 Automator

但是，图形化的吸引力到底在何处？也许换个切入面，弄清脚本的限制会更容易一些。脚本由文本组成，但文本却不是最易懂的编辑方式。一方面，文本比图形更加抽象，不够符合图像化思考的直觉；更重要的是，一段文本**处处可改**，从而使不熟悉编程的人**不知如何下手**。

![](.evernote-content/86047BE9-F4C6-40B3-92C6-B463CDA03DE7/1219EF75-5F9F-4886-85D4-779D3AF99AB5.png)

这段 Brainfuck 代码是打印「Hello Word」的意思

即便你完全不会写代码，也不用害怕承认这一点，因为专业人士和脚本高手看到上图的感受和门外汉是一样的：

> 这是什么鬼东西！

而不懂代码的自动化玩家看到脚本时，也是这样的反应。程序员眼中的代码，如同围棋选手眼里的棋局、外科医生眼中的病灶乃至商业大亨目中的经济情势类似，一定程度上都可以归为一种**模式**，相关领域的专业人士多多少少可以避开无关的部分，直达要处理的部分。可是在外行看来，任何代码仍然都是噩梦，因为即便把代码「模块化」，不懂的人也很容易误动已有的代码，只要不慎少复制一个括号、多打一个空格，整段代码往往就无法运行。于是，不难理解代码对于外行来说多么可怕：它们太容易被改动，让不熟悉代码的人去用脚本化的自动化工具，无异于让门外汉举着牛刀去做脑外科手术，所经之处难免血肉模糊。

总之，代码无论如何简化，不懂的人就是不懂，好比医生用速记符号写下的处方，彼之蜜糖，吾之砒霜，专业人士拍案而起，围观群众不知所以。WWDC 2018 上亮相的新版 Swift 编程语言也是如此，纵使苹果炫耀其如同搭积木一样的模块化编程能力，也没有解决入门难的根本问题。

Automator 图形化「编程」的妙处，就在于把编程中的**模式部分**抽离了出来，做成相对**固定的模块**，只留下少量几个参数供用户填空。「拖来拖去」只是图形化编辑在操作上的表现形式，在更深层次，图形化**简化乃至隐藏了无需改动的代码**。这样一来，就大大减少了「写错代码」的担忧。我曾制作过一个批量抓取网页图片的 Automator 动作，其实现方式异常简单：

* 添加「获取网页」「获取 URL」「过滤 URL」和「下载 URL」4 个模块；
* 填入想要抓取的图片格式。

仅此而已，就做出了一个批量抓图的动作。下图演示中我还通过另一个支持图形化编辑的工具 Keyboard Maestro 给 Automator 抓图动作做了一个快速启动菜单，实现方式可以参考《[为 Mac 浏览器定制鼠标手势操作](https://sspai.com/post/53825)》。

图形化编辑界面之于脚本编辑器，好比**一屉积木**之于**一张白纸**，后者纵使配上厚厚的说明书（技术文档），也很难激励一个新手创造出能用的自动化动作。除了天资过人的作家，没有人看到一张白纸就能文思泉涌 —— 可是面对一篮子积木，就算小孩也能搭出一些东西来。当一个工具让人产生「想摸摸看」的冲动时，它已经拥有了可贵的优势。

本节结束之前要提醒一下，我们没有否定代码（或者说脚本）的必要，只是在有更加简单的实现方案时，可以选择图形化。况且，Automator 以及后来带有图形化特点的工具（Hazel、Alfred 等）通常也支持 AppleScript 和其他脚本语言，并不耽误通过脚本实现更复杂的功能。

### 在图形化工具中写脚本的怪圈

macOS 从来不缺图形化工具，从较早的 Automator、Hazel、Keyboard Maestro，到后来居上的 Alfred，macOS 俨然成为一个百花齐放的图形化自动化生态圈。但是我们却能看到一个怪圈：这些图形化工具严重依赖脚本，乃至出现了**「在图形化工具中写脚本」**的怪圈。例如，作为 macOS 上图形化自动化工具开辟者的 Automator，其最好用的模块却是 Shell 脚本和 AppleScript 脚本模块，这简直是一种讽刺。

![](.evernote-content/86047BE9-F4C6-40B3-92C6-B463CDA03DE7/01A8B5E8-BF67-4EF1-8017-20DB72349834.png)

Automator 仍然非常依赖脚本

脚本功能强大是理所应当的，但这不意味着所有事情都要写脚本。macOS 的自动化工具界一直很热闹，理论上这些工具都是「图灵完备」的，即它们可以实现任何功能。但实际上，这些工具的图形部分往往**缺胳膊少腿**，缺少重要功能。以 Automator 为例，它甚至**没有 IF 判断**，你要是想为 6 个网站制作快捷搜索的动作，一般就要做 6 个大同小异的动作 —— 除非写脚本，才能获得完整的编程功能。

![](.evernote-content/86047BE9-F4C6-40B3-92C6-B463CDA03DE7/4FF3A7A7-85EE-43CB-A3C0-1C1344FB3FE5.png)

为了多种场景，必须制作多个类似的 Automator 动作

**不彻底的图形化**，把想用图形化的用户也**赶回了脚本阵营**，这就是 macOS 自动化至今仍然依赖脚本的重要原因。在当今，「支持脚本」四个大字等同于工具的**免责条款**，用户用不好只能怪他们自己水平不到家、不懂写代码，这是令人沮丧的。

### 后图形化时代的「编程」：快捷指令

要走出弃图形化不用而写脚本的怪圈，就需要一款图形化程度彻底的工具，而快捷指令就是这类工具中的标杆。

快捷指令的成熟，表现在它将代码中的**逻辑部分**也进行了图形化，令一根肠子通到底的 Automator 难以望其项背。快捷指令中的「如果」模块就令人印象深刻，默认情况下它由「如果」「否则」「结束」几个部分拼成，留给你打字的地方只有判断条件部分，模块本身完全不容许改写，让用户从一开始就不可能拼错代码（因为根本就不用写代码）。就算是健忘症「晚期患者」，在快捷指令里也没有机会拼错、漏写或者重复写 `end if` 或 `fi`（AppleScript 和 Shell 中表示结束「如果」的代码。更进一步，需要使用多个 IF 的时候，你也没有机会把它们穿插造成错误的机会，因为图形化的模块根本就没办法穿插！

![](.evernote-content/86047BE9-F4C6-40B3-92C6-B463CDA03DE7/916082E6-C322-4CEB-930B-34416449558C.png)

快捷指令图形化界面留下的「犯错空间」并不多

稍有编程经历的读者可能会问，IF 只能照顾到「是」和「否」两种场景，那么多场景（Case）怎么应对呢？结合快捷指令自带的列表、文本合并和正则匹配模块，Hum 和我已经实现了 Case 语法。在《[如何用 RSS 实现个性化订阅](https://sspai.com/article/51956)》一文中，就用了前述一套模块组合照顾到了多个网站的链接格式。这里不展开解释细节，只是说明一下快捷指令的潜力。

组合现有图形化模块满足多场景

这种用有限零件实现高级功能的过程也是充满乐趣的，也让我回忆起最原始的自动化意识：我们在知道「自动化」的概念前就已经开始使用自动化技巧了。我在读小学时，班上曾有同学在一款横版闯关游戏中发现了刷弹药的 bug：只需持续攻击某个建筑，就会不断掉出弹药。于是此君置一叠硬币于表示攻击的 `J` 键上，随后撒手不管、让游戏角色自动攻击，不费吹灰之力就刷了个饱。

用硬币按住键盘按键，便是一种原始的、借助硬件的自动化实现方式。今日的图形化给了我们一个轻松实现这些小技巧的途径，那些乍现的灵感不会被「需要代码」的门槛吓退。

通过图形化 API 模块走向云端
----------------

在上一节中，我们主要讨论了一些自动化工具内部或者和本地应用的交互。在这些讨论范围内，图形化设计已经发挥了开辟性的作用，帮用户敲开了自动化第一扇门。

在自动化中，还有一个更为复杂的领域 ——API（Application Programming Interface，应用程序接口），通过它可以和在线服务联动，实现本地难以实现的功能，例如把文章存到指定的 Evernote 笔记本，或者借助云端的机器学习程序快速压缩海量图片。

当图形化设计和 API 结合起来，编辑动作的难度就能明显下降，同时解决动作分享、隐私保护方面的难题。

注：本文所指 API 皆为自动化工具内置的、图形化的、现成的 API 模块，至于通过 Get Content of URL 或者编写一段 curl 脚本来和 API 交互的方法，就不属于本文讨论的情况。

### API vs. 本地应用

最早让我意识到 API 重要性的是互联网自动化工具 **IFTTT**。它像是一个「接线生」，负责通过 API 联通各个服务。API 最明显的特点就是 **可全程后台执行**，不需要打开应用。我曾经帮 Hum 改进过一个 Evernote 剪藏的快捷指令动作，它可以将网页上选中的好句带上自定义样式摘抄到指定的 Evernote 账户，而根本不需要打开 Evernote 的本地应用 —— 实际上，就算没有装应用也不会影响 API 的执行。

![](.evernote-content/86047BE9-F4C6-40B3-92C6-B463CDA03DE7/A2597118-7F8A-448E-AEDD-16F31B9E72AA.png)

借助 API，从 Instapaper 摘录到 Evernote

![](.evernote-content/86047BE9-F4C6-40B3-92C6-B463CDA03DE7/25EBEEE7-5BA5-4809-922A-56C6F5EC81FB.png)

不过，API 毕竟只是一个零件，它们也需要本地自动化功能的支持才能充分发挥实力。IFTTT 就让我感到力不从心：其本地功能匮乏，自身不能对数据做太多处理。我在《[在线阅读系列（三）：将 Instapaper 摘抄导入 Evernote](https://sspai.com/post/57548)》一文中分享过一套自定义摘抄样式的思路，由于 IFTTT 的功能几乎完全受制于 API 提供者，我不能直接在其流程中配置个性主题文件，只好另行桥接了一个自动化工具（Zapier），冗长的成品固然可用，但是制作、修改和维护都不容易。

![](.evernote-content/86047BE9-F4C6-40B3-92C6-B463CDA03DE7/9187B79B-37B2-41AD-9902-052291B4E070.png)

通过 IFTTT 和 Zapier 协同制作的冗长自动摘抄流程

而介于 IFTTT 这种纯在线 API 集成工具和本地自动化工具之间的，是一批 iOS 上的自动化明星。同样在阅读摘抄的系列文章中，我还分享过两个利用快捷指令实现的摘抄动作，它们能够实现的样式就丰富多了。

现在把目光转向 macOS，却难以找到拥有类似 API 的工具。对于当前的 macOS 而言，**API 的集成**是亟需填补的空缺。

### API 是「套壳」时代的自动化必备品

上一小节中我们讨论了 API 的重要性。这一节我们将面对一些严肃的现实状况，谈谈 API 对于当前 macOS 环境的必要性。

近几年，越来越多的工具转向了使用全平台通用的开发框架，比如 Todoist 和 Slack 的 macOS 版就是「网页套壳」工具。这些「套壳」工具都有一个特点，不支持 macOS 现有的原生自动化，既**不提供 Automator 模块**，也**不支持 AppleScript 脚本**，难以融入自动化工作流。下图是 Slack 在 AppleScript 词典中的查询结果，除了所有应用都有的通用部分，Slack 不提供任何自动化支持。

![](.evernote-content/86047BE9-F4C6-40B3-92C6-B463CDA03DE7/7357E50D-4120-45B1-A2CD-0571A06CBEB5.png)

Slack for Mac 几乎没有留下什么自动化的空间

「套壳」工具的涌现很大程度上是因为开发成本高，毕竟开发一款 macOS 原生应用可不是轻松的事情。可以预见，未来专门为 macOS 打造的原生应用只会越来越少，原先的「脚本流」自动化方案只会越来越行不通。

这种趋势下，API 是更经济、更可行的自动化选择。实际上，前文提到的 Todoist 就提供了 API，而前不久登上 macOS 的草稿工具 Drafts 便集成了这一 API，不依靠 Todoist 客户端便能够快速创建任务。我前年也借用 Slack 的 API 做了一个文章搬运工具，把网站后台的投稿搬到 Slack 工作群中供同事审核。

![](.evernote-content/86047BE9-F4C6-40B3-92C6-B463CDA03DE7/A77E714B-EA2B-49D2-999B-3AD19E241ADC.png)

通过 Slack API 投票的几个动作

美中不足的是，前述两个动作仍然是借助脚本实现的（相比之下，Drafts 的 Todoist 账户数据和脚本是分开的，比所有数据全部明文存储在本地的 Slack 动作要更安全），编写起来并不轻松。而快捷指令中，所有的 API 也实现了图形化，不说的话可能不会发现它们和其他模块有什么不同（除了需要联网……）。由于本文在探讨本地自动化时已经分析了图形化的优势，快捷指令的图形化 API 模块在方便、可视化方面的优势便不再赘述，希望深入了解的读者可以阅读 @文刀漢三 的《[快捷指令的第三方工具支持现状](https://sspai.com/post/57778)》。下面，我们把目光转向图形化在 API 安全性上的意义。

### 通过图形化解决 API 安全顾虑

作为在线服务，API 在带来便利的同时拥有很强的**专属属性**，往往要求用户注册并登录账户才能够使用其功能。而登录账户就涉及安全问题：本地存储密码有泄漏隐患，分享动作时有夹带私人信息的风险。早些年的快捷指令也面对类似的安全问题，我们下载一些动作后，偶尔就会看到类似「请填入你的 token」的导入提示问题。

![](.evernote-content/86047BE9-F4C6-40B3-92C6-B463CDA03DE7/F7BA7458-853E-4090-8537-9A85B7ABE389.png)

快捷指令的导入问题

为了避免泄漏或误用别人的账户，动作创作者需要抹掉才能放心分享，使用者也需要检查动作以免其中夹有「私货」。这好像朋友管你借一本书，你一面满口答应，一面拼命擦掉上面的页边读书笔记，检查一遍确保不会泄漏隐私才把书交出去；或者站在借书者的角度，为了安全竟然要把书脊扒开看看有没有窃听器（相当于动作里的「私货」）。这样的分享体验是大打折扣的，我自己在 GitHub 上分享过上百个动作，每个动作都要留个心眼「洗」一遍才敢分享出去，徒增工作量。

iOS 上那些图形化的 API 模块正好能够对症下药。以快捷指令为例，其内置的 API 模块都存在着**两层授权**，从而保证数据安全：

1. **第一层授权在服务和快捷指令应用之间。**在线服务只认当前这台设备上安装的快捷指令，即便原模原样把动作分享出去，别人也需要重新授权才能使用，不用担心泄漏账户信息。类似的授权方式在草稿工具 Drafts 中出现过，但是在 macOS 中很少见，至少主流的几款自动化工具似乎没有。
2. **第二层授权在快捷指令应用和每个动作之间**。这层授权机制避免了一人授权、鸡犬升天的情况，你可以掌控每一个动作的权限状况 —— 比如，从网上下载了一个安全性待确认的动作，就可以暂时禁止它访问某个 API（比如，不准访问 Evernote，避免误删、误改数据），运行无误后再开放。

![](.evernote-content/86047BE9-F4C6-40B3-92C6-B463CDA03DE7/BFEA67C1-D67E-48A9-B072-5CC473587308.png)

每个快捷指令都有 API 权限的开关

借助图形化，这些 API 设计都得以固化，既不用你操心如何设置，也不必担心别人恶意修改。快捷指令的两层授权设计更是将动作中的 API 信息洗的一干二净，可以让人没有任何顾虑地按下分享按钮。对于有心和他人交流自动化动作的玩家而言，图形化 API 称得上是一种安全保障。

小结
--

传统的 macOS 自动化主要由高级玩家统治，难以成为大众解决问题的手边工具箱。Catalina 的发布并没有否定脚本的价值，进阶用户也可以自行安装、坚守脚本；相比之下，iOS 反哺过来的「图形流」在编辑、分享上都有不容忽视的优势，能够帮助有心提高效率但是缺乏代码基础的用户踢开临门一脚，更可能引领 macOS 自动化的发展。

在到达 Catalina 之前，自动化工匠们已经在苹果这位巨人的花园中嬉闹多年；基于 Catalyst 的新工具的出现意味着巨人的苏醒，相应的变化或许会让一大批根基不稳的工具轰然落幕，但是有着悠久沉淀的工具也将和快捷指令、Drafts 为代表的新工具们共存、历久弥新，满足不同用户、在不用场景下各自施展拳脚。
