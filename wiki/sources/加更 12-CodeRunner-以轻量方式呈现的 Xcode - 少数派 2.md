CodeRunner 初体验
--------------

在我第一次听到「CodeRunner」这个词时，我一度以为是 VS Code 中知名的由国人开发的插件 [Code Runner](https://sspai.com/link?target=https%3A%2F%2Fmarketplace.visualstudio.com%2Fitems%3FitemName%3Dformulahendry.code-runner)，因为二者不仅在名称上有些重合之外，就连核心功能也是十分类似，都能让我们快速运行代码。

但当我分别查看了二者的发布日志时才发现，前者似乎比后者出现得更早，并且是作为一个 App 或者说**轻量的文本编辑器**而存在。

虽然我们可以完全将 CodeRunner 作为像 Windows 的记事本或 macOS 上文本编辑一样的替代品用于文本编辑工作，但它除了文本编辑之外实际对编写代码有着更多的支持。

可既然作为可用于编写代码的文本编辑器，那么自然不可避免地就会要和同样是可编写代码的、且目前主流或知名的文本编辑器相比较，如 VS Code、Atom、Sublime 等。

但在比较之前，先让我好好聊聊对于 CodeRunner 的第一印象和使用体验。

### 外观

在我第一次尝试用 macOS 自带的 Xcode 学习 Swift 语言时，我就对 Xcode 留下并不太好的印象，跟我印象中所使用的 Pycharm、Eclipse 这类 IDE 的操作逻辑并不相同，对于新手来说感觉就像是一台功能复杂的机器，一时半会不知道从哪开始启动。但谁叫 Apple 这家公司有着深厚的艺术基因，尽管使用时的操作逻辑让人难以习惯，但从视觉上又总觉得感觉这是个精致的艺术品，无论是 UI 还是字体都让人感到舒适，以至于我曾有段时间想将 Xcode 的 UI 主题和字体都复刻到了经常使用的 Pycharm 上。

但 Xcode 终究只是一个适用于 Object-C、Swift 等这类和苹果公司软件开发相关的编程语言，并不能支持其他编程语言。但直到我看到了 CodeRunner，我一度以为这是个「Xcode Lite」，毕竟它的 UI/UX 外观设计简直和 Xcode 太像了。

![](/images/file-generic.png)

左：CodeRunner 右：Xcode

然而 CodeRunner 在界面设计上并没有像 Xcode 那样复杂，整个应用的窗口布局反而是相对简单，没有冗余其他元素从而能让人拥有较为纯粹的沉浸式体验。不过简单也就意味着 CodeRunner 可能不一定有 Xcode 那样特性丰富、功能强大，CodeRunner 的定位主要还是一款通用性的编程编辑器。

### 开箱即用

CodeRunner 的基础功能和其他文本编辑器的基础功能没有较大区别，同样可以用它来记录普通文字；但它的核心功能在于，它内置了对许多编程语言的语法支持（Syntax Mode），并且还提供了特定语言的代码文件的运行语句，我们可以无需借助其他插件来实现。

比如下面这一段代码，当我们电脑里已经存在 Python 解释器时，只需要将 CodeRunner 的语言模式改成 Python 再点击运行按钮即可运行：

import os import unittest class TestFileSplit(unittest.TestCase): files = ["file.py", "abc.txt"] names = [] exts = [] def setUp(self) -> None: self.names = ["file", "abc"] self.exts = [".py", ".txt"] def tearDown(self) -> None: self.names = [] self.exts = [] def test\_os\_split(self): tmp\_names = [] tmp\_exts = [] for file in self.files: name, ext = os.path.splitext(file) tmp\_names.append(name) tmp\_exts.append(ext) self.assertListEqual(tmp\_names, self.names) self.assertListEqual(tmp\_exts, self.exts) if \_\_name\_\_ == '\_\_main\_\_': unittest.main()

![](/images/file-generic.png)

这也是其在官网上重点宣传的「Out-the-box」开箱即用亮点。

CodeRunner 本身支持超过 25 种编程语言，基本能覆盖大多数市面上主流或是较为常见的编程语言，甚至对于某些编程语言还提供了 IDE 级别的体验。

![](/images/file-generic.png)

但需要注意的是，虽然 CodeRunner 主打的是开箱即用，但在实际使用中并非如此；因为即便是 CodeRunner 已经为我们内置了对编程语言的语法支持以及运行代码所需的命令语句，但最终是否能真正运行程序还是取决于**你电脑中是否已经存在对应编程语言的解释器或编译器**，否则 CodeRunner 会报错说不存在相应的解释器或编译器，或是询问是否需要安装。

比如 macOS 里并没有内置 Lua，所以当我们简单运行以下代码时就会看到 CodeRunner 抛出的错误：

function greet() print("hello, SSPAI ") end greet()

![](/images/file-generic.png)

因为 CodeRunner 默认会根据你选择的编程语言而使用相应的执行命令，而这个在运行时被执行的命令本质上和我们在终端 Terminal 中运行代码文件的命令别无二致。例如我们在 CodeRunner 中运行 Python 代码时，本质上就等于：

python

所以 CodeRunner 即便是声明主打着开箱即用，在我们使用它编写相应编程语言的代码时还是最好保证我们的电脑里已经事先安装了对应编程语言的解释器或编译器。

默认情况下，CodeRunner 的运行模式是以 Console 控制台的形式进行，实际就相当于我们在终端命令行中以命令行的方式进行操作；但对于前端的使用者来说，CodeRunner 也支持了 Web 的展现形式，即将目前的 Console 形式转换成浏览器页面的形式，以便能更好地直接查看 HTML、CSS 或 JavaScript 的效果，而不需要我们额外打开一个浏览器。

![](/images/file-generic.png)

### 配置设置

CodeRunner 的设置内容并不多，因为除了编程语言的设置之外，其他项比如通用性设置、文本编辑设置、键位绑定等项的默认配置项已经能满足基本使用；并且只要是我们能够设置的部分，都可配置的地方都能通过图形界面化进行操作。

![](/images/file-generic.png)

比如我们可以根据自己的需要去调整文本编辑设置中的字体和大小、缩进以及补全。

![](/images/file-generic.png)

如果我们不是想要快速验证想法、代码逻辑或是算法解法等事情，而是想编写更复杂的项目代码，那么 CodeRunner 在编程语言设置项上的默认设置可能并不能满足你的要求，你仍然需要自己去配置诸如虚拟环境、不同版本的解释器或编译器等步骤。

同时，由于 CodeRunner 并不支持中文汉化，换言之所有配置项均为英文，因此对于英文水平不太高的使用者来说可能是个不小的阻碍，需要借助额外的翻译软件去知道待配置项的意思，才能进一步决定是否修改。

但总体来说，CodeRunner 简约的配置项可能也和其主打的开箱即用特性相吻合，避免使用者在使用过程中陷入到繁琐的配置中，而是专注于即刻使用。

思索 | 和「Big Brother」们的较量
-----------------------

在编辑器的红海里竞争异常激烈，前有 Vi/Vim、GNU Emacs 等上古利器，后有 Sublime Text、Atom 和 VS Code 等后起之秀。基本上目前还能被叫得出名字并且被人广泛使用的编辑器，往往都是经过了无数开发者捶打历练的。

抛开远古的老大哥不谈，对于 CodeRunner 来说，和它最为近似且用于比较的可能就是 Sublime、Atom 和 VS Code 了。那么同样作为编辑器 CodeRunner 能有什么胜过前面「三大将」的地方吗？

因此，我从一个开发者的角度出发，主要选取了以下几个在使用编辑器时可能比较关心的方面来看看 CodeRunner 的核心竞争力如何：

* 快速上手
* 可配置化
* 插件生态
* 跨平台
* 编码体验

### 快速上手

当我们在使用或操作一个产品或软件的时候，作为消费者以及用户的我们最关心的往往不是它们具备哪些很酷的特性，甚至是功能有多么强大；最关心的反而是我们能不能快速上手使用它们（通过本能直觉），或是通过内置的引导教程了解该使用它们（被引导）。因此「能否让人快速上手使用」是反映一个产品或软件是否对用户友好的直接表现。

从我目前的体验来看，VS Code 和 CodeRunner 在这一轮比较中可以说是直接胜出，但 CodeRunner 相比于 VS Code 可能就稍逊一筹。

VS Code 和 CodeRunner 能够胜出的原因有所不同，前者拥有良好的使用引导，能够帮助初次使用者快速上手其最核心的功能部分；后者则是足够简洁，并且由于具备了开箱即用的特性，能够让使用者根据不同的文件模式去运行，因为它足够符合「小白」的直觉。

![](.evernote-content/1F4CB73E-6DF0-4DB8-B084-7431EEEB9530/6E921ADF-BBF4-40AE-B2E1-2D9CA0075A8D.gif)

但 Atom 和 Sublime 在快速上手这方面的用户友好程度较低。Atom 在打开之后会进入到一个简单的「Welcome」界面，但最终初次使用者需要通过其文档来逐渐对它了解；而 Sublime 可以说简约到了极致，没有任何的使用引导或提示，打开了之后就是一个纯粹的编辑框。

![](/images/file-generic.png)

CodeRunner 开箱即用的特性，能够基于用户电脑里已有的特定编程语言的编译器或解释器而直接运行对应的代码文件，这是 Atom、Sublime、VS Code 所没有的，因为这三者都需要额外通过第三方插件去实现。

### 可配置化

对于开发者来说，无法让自己去定制化的东西通常很难受到青睐。就像是「一千个观众眼中有一千个哈姆雷特」一样，不同的编辑器在不同的开发者手中会被逐渐调教成适合自己的模式。因此，对于编辑器来说如果配置项过于死板，除非本身设计足够优秀或是在特定领域完全独占，否则只要还存在其他同类更好的竞品，基本上就可以宣告该编辑器注定是属于少部分人了。

实际上大部分编辑器都提供了基本的配置项，但 CodeRunner 相比于 Sublime、Atom 和 VS Code 来说在可配置化项较差，后三者都能通过开发者最为熟悉的 JSON 格式来对编辑器中的选项进行配置。

通过 JSON 配置的好处，一方面在于使用者能够将其备份在特定的网盘或者 Github 仓库中，避免更换设备或重新安装时配置丢失的问题；另一方面在于，因为 JSON 格式不属于特定操作系统的文件格式，而是一种交换数据的通用文件格式，当编辑器支持跨平台并且我们需要在不同平台上使用编辑器时，通过 JSON 文件能够帮我们最大程度保持我们使用编辑器的模式统一。

尽管 CodeRunner 缺少了 JSON 配置项，但好在提供了设置界面，让使用者可以在设置界面中对 CodeRunner 的默认配置进行微调。和 CodeRunner 类似，VS Code 和 Atom 在提供了 JSON 配置的同时，也提供了对应的设置界面。

但 VS Code 更为贴心，提供了配置搜索的功能，不管是 VS Code 本身的配置还是第三方插件的配置，我们能够通过精确或模糊搜索找到具体的配置项进行设置，这一点比 Atom 做得更好。因为如果你要在 Atom 修改对应插件的设置，就必须先找到对应的插件，然后再点击设置按钮进入到配置项中配置，这种方式随着插件逐渐增多且缺乏配置搜索功能而显得十分繁琐。

![](/images/file-generic.png)

最为极端是 Sublime，它没有提供任何设置界面，用户无法通过鼠标点选来调整，而是完完全全以最纯粹的 JSON 方式来设置所有配置项，这就意味着使用者想要配置就必须先阅读 Sublime 的相关文档了解具体的配置项。

![](/images/file-generic.png)

综合来看，VS Code 在可配置化上可以说是完全胜出，不仅配置项丰富以及对应的配置搜索功能，而且提供了设置界面以及 JSON 设置两种形式，既能减少一般使用者的使用成本又能满足喜欢「把控一切」的 Geek 的需要。而 Atom 虽然稍逊 VS Code 一筹，但相比于 CodeRunner 而言提供了可自定义的 JSON 配置文件，能够方便用户备份或是跨平台使用。

### 插件生态

在如今趋向于开发者共同参与贡献或建设的「开源」潮流愈加明显的当代，衡量某个技术事物（无论是 App、编程语言还是技术框架等）是否成功的主要特征之一就是看是否受到了一定量开发者的簇拥。因为开发者不同于一般的用户，他们是一群特殊的、富有创造力的用户。这点往往在插件（Plugins）或者是扩展（Extensions）上体现得更为直观。

为什么这么说？因为一个技术事物不可能去实现或覆盖所有的功能，毕竟当它服务的不仅仅只是一小撮人时，众口难调的情况就会愈发明显；那么此时最好的做法就是存在一定的弹性且可扩展空间，将其交给最富有创造力的程序员群体们去实现属于符合他们真正需要的功能，从而做到「千人千面」。

所以无论是开源的 Atom、VS Code 还是 Eclipse，还是可能需要付费的 Sublime、Idea、Visual Studio，都支持开发者们使用甚至开发插件以扩展 IDE 额外的功能。

**但很不幸的是，到目前为止尚未看到 CodeRunner 存在任何支持插件或扩展的设置或模块**。

这就意味着你在使用 CodeRunner 的过程中，有可能面临 CodeRunner 功能缺失的情况；但这缺失的部分，哪怕你是一个拥有「造轮子」能力的开发者，CodeRunner 也没有给你留下任何去弥补的余地。

同时，即便是 CodeRunner 本身就实现了一些特定的功能，但个人或小团体的能力总不如「众人拾柴火焰高」那般来得壮大。因此，CodeRunner 的一些小毛病会在这一部分中会被尽可能地放大。

因为我是一个有 Vim 键位操作习惯的使用者，所以无论是使用文本编辑器也好，还是使用 IDE 也好，为了能让我双手保持在键盘上，我往往都尽可能会使用快捷键或设置 Vim 键位映射模式来写文字或是代码。当我看到 CodeRunner 支持 Vim 的键位映射时一度乐开了花，还以为在 XCode 上不支持的东西却在 CodeRunner 上被支持了。

但在实际使用中，CodeRunner 的 Vim 模式可以说是**灾难级表现**。我就通过几个我经常性的 Vim 操作进行测试：

1. 在代码末尾，按下两次 G 键后跳转到代码起始行
2. 进入到 normal 模式后，按下 Ctrl + v 并结合 K 键和 J 键来进行纵向的块级元素选择
3. 使用 Shift + V 选择对应行内容后并按下 Shift + D 进行删除

在上述的三个操作中，CodeRunner 连最基本的按下两次 G 键跳转代码起始行都未曾支持，同时也无法进行块级元素选择，甚至是在我进行第三步操作时如果操作速度比较快会将待删除行和下一行内容一起删除！

![](.evernote-content/1F4CB73E-6DF0-4DB8-B084-7431EEEB9530/D675594C-402B-4EF3-A382-2B7FFE2EB5FB.gif)

最开始我曾怀疑是否是我的输入法存在了干扰，但即便我将输入法切换至 macOS 的原始英文输入法，以上存在的问题依然能够复现。于是我又进一步在我 Terminal 中原生的 Vim 进行测试，发现就算我开着搜狗输入法并将其切换至英文模式，它也同样能很好地实现我上述的三个操作。切换到 VS Code 中通过插件进行测试也依然准确无误。

根据测试结果来看，尚不得知 CodeRunner 的 Vim 键位映射是如何实现，但 CodeRunner 的 Vim 模式甚至还比不上一般的 Vim 模拟操作插件。这也说明如果一个软件或是 App 不存在扩展的部分，那么就需要等待软件或 App 的作者去开发实现，但这个实现是否做到了可用性的地步仍然是个未知数。

撇开 CodeRunner 不谈，在插件下载和使用方面 VS Code 的体验又比 Atom 和 Sublime 更好，因为 VS Code 的插件模块位置显眼又提供模糊搜索和筛选器，并且依托于发行商微软在国内的也有对应的镜像，即便是没有内置一些插件，都可以通过插件模块去下载获得更好的体验；而 Atom 和 Sublime 的插件安装选项层级较深，并且存在网络限制问题，因此这两个编辑器的使用者可能要花不少的时间在前期安装插件、调试、配置这些中间过程上。

CodeRunner 在插件生态这方面可以说是以失败告终，因为在这个无论是开源还是付费的编辑器都有插件生态的时代，缺少了对插件或扩展的支持就可能意味着，在编辑器的红海里注定属于格格不入的「异类」。

### 跨平台

在现在互联网发展愈发蓬勃的年代里，无论我们是作为一个为软件或应用付费的消费者还是将软件或应用作为生产力的使用者，我们都尽可能地希望我们使用的软件不会受限于使用的平台或是操作系统。

这一点对于既是使用者又是开发者的人来说尤为重要，因为对于开发者来说如果一个软件或应用既不是平台或系统独占，又不是缺乏替代品，那么如果不具备跨平台这一特性，会不得不让人在使用之前再三考虑，以免在迁移个人数据时花费大量的时间成本。同时，个人在一个软件或应用上花费的学习成本，不亚于完全学习一门新的技能。

**然而，在这一点上，CodeRunner 毫无疑问又败了**，因为 CodeRunner 只能够在 macOS 上使用。但作为同样是商业付费的 Sublime，都完全支持 Windows、macOS 和 Linux 三个操作系统，配置文件可以通过 JSON 的形式保存并通过借助相应的插件和 Github 这样的托管平台去同步，完全做到了跨平台和无缝衔接。

![](/images/file-generic.png)

哪怕就连微软引以为傲的 Visual Studio（注意，不是 VS Code）这一有着「宇宙第一 IDE」美称且最开始只能在 Windows 上使用的 IDE，都推出 mac 版本，为的就是借助通过跨平台来完善自己的生态（当然好不好用再另说）。

![](/images/file-generic.png)

更不用说是像 VS Code、Atom 这样的开源编辑器了。

尽管像 CodeRunner 这样只支持 macOS 平台或 iOS 平台的软件或应用还有很多，但同类竞品如果从用户数的层面来比较，跨平台的软件或应用永远是比非跨平台的软件或应用要多得多，使用的人多也就意味着也会更有利于开发者去完善自己的软件或应用。这也就是为什么在开源社区开源了的技术会得到快速发展的原因，因为在开源社区的人他们既是技术的使用者，又会在条件成熟时成为帮助该技术进一步改善的开发者。

### 编码体验

无论编辑器有着再怎么多的特性，最终都还是要以实际的编码体验为主。

和其他三个编辑器不同的是，CodeRunner 不需要依赖于插件去实现某些功能，它主要还是提供了一站式的使用体验。而其他三个编辑器，尽管在我们安装时就已经为我们预装了一些插件，但如果想要让实际的使用体验更加好，仍需要我们去寻找高质量的第三方插件。

需要注意的是，因为编辑器就算是有再多的第三方插件也很难达到 80% 的 IDE 使用体验，毕竟编辑器的通用性可能更强，而 IDE 更偏向于实际的编程工程或项目，二者并不处于同等的比较层次，因此很难以 IDE 的角度去审视编辑器。

#### 操作逻辑

在操作的逻辑上 CodeRunner 和其他编辑器有着最大的一个区别就是它没有所谓的「Project」工程概念，或许说它默认的工作区就是当前用户的系统根目录。所以当我们进入到 CodeRunner 时，会在左侧看到用户根目录下的所有文件，并且只要是可见的文件夹，你都可以无限的点击下去。

![](.evernote-content/1F4CB73E-6DF0-4DB8-B084-7431EEEB9530/4C363067-B661-4669-82BD-7C63C4D8A837.gif)

这和一般的树形目录结构可展开的操作逻辑有所不同，它更像是 N 级界面下沉的方式，当我们双击某个文件夹时会直接跳转到对应文件夹的内部，而此时的工作区就是这个文件夹。但它这种下沉操作的缺陷在于，如果你有一个工程或项目下有许多子工程或项目，那么当你需要在这些子工程或项目切换中跳转时，下沉式的操作就会变得异常繁琐，你需要在目录间不断回退或前进，原不如树形目录结构来得清晰。

CodeRunner 本身可能更像是一个即开即用的编辑器，刻意弱化了工程项目的概念，对于单文件或单脚本来说或许更好，因为在这样的形式里我们不需要考虑目录或文件之间存在的链路关系。可实际上我们更多的会是以工程或项目来组织我们的代码，但 CodeRunner 并没有提供大部分编辑器或是 IDE 所具备的「New Project」或「Open Project」的选项，所以只要当工程或项目的文件形成一定的规模时，CodeRunner 下沉式的操作逻辑弊端就会显得一览无遗。

#### 代码补全

编辑器和 IDE 最大的一个差异在于：编辑器的代码补全往往都不是那么的智能。这点当许多工程师从 IDE 转入到 VS Code 这样编辑器时在使用中最明显的感受，因为即便我们依托于插件来帮助我们提高编辑器本身自带的代码补全功能，但仍然是达不到 IDE 的高度。

在 CodeRunner 的官方介绍页面中可以看到，其主打的开箱即用之外，还宣称其拥有着最好的代码补全功能（CodeRunner's code completion is the best you'll find in any IDE.）。但事实上真的如此吗？

我在实际测试的过程中发现，像 Go 语言都没能很好的补全，即便我是手动导入官方的标准库，但依然没办法补全相应的 API 方法，只能人为手动去补全。完全达不到 CodeRunner 在官网上所宣称的那样是「IDE-level」。

![](.evernote-content/1F4CB73E-6DF0-4DB8-B084-7431EEEB9530/4E13EDAA-73FA-424A-AE45-D5653A4FF827.gif)

但最让我感到奇怪的是，在 macOS 上天生就自带的 Swift，竟然在 CodeRunner 里连最基本补全都没有，而在 Xcode 自带的 Playground 中却是没有任何问题。比如简单的一行「Hello, world」：

var str = "Hello, playground" print(str) str.append("\nHello, SSPAI") print(str)

这里我仅仅只是声明了一个名为 str 的字符串变量，并且调用其 append 方法在原有字符串上追加新的内容。但也就仅仅是调用这个方法都无法弹出正确的补全提示，更不用说是自己编写封装的类方法了。

![](.evernote-content/1F4CB73E-6DF0-4DB8-B084-7431EEEB9530/0C10B7A2-DDDF-4FA0-968F-195648A3ADEA.gif)

当然，像 Python、Java、PHP、JavaScript 等编程语言在**不引入第三方依赖库**的前提下，在 CodeRunner 上基本都达到了及格线或中上程度，但即便是这样也存在瑕疵。比如我使用 Python 目前流行的 Type hint 写法，然后调用对应的方法时 CodeRunner 似乎就变得不那么智能：

message: str = "Hello, world"

这一带类型注解的写法目前在脚本语言里十分流行，大热的 TypeScript 就是在 JavaScript 的基础上引入了类型系统，使得动态脚本语言也具备了像静态编译语言那样的类型检查功能，减少 BUG 的产生。Python 也自 PEP484 提案之后逐渐支持上述写法，但它并不是会被解释器强制约束，而是像 Python [官方文档](https://sspai.com/link?target=https%3A%2F%2Fdocs.python.org%2Fzh-cn%2F3%2Flibrary%2Ftyping.html) 说的那样，它只是为了第三方工具能够提供更好的体验，比如类型检查器（Linter）、IDE 等。

在我使用 IDE 或者 VS Code 这样的编辑器进行编码时，如果类型补全提示不那么智能时我就会通过类型注解来帮助这些工具改善提示没生效的问题。但在 CodeRunner 中编写 Python 代码时加上了类型注解之后似乎 CodeRunner 的代码补全提示就完全失效了，取而代之的反而是和该类型无关的其他类型方法。

![](.evernote-content/1F4CB73E-6DF0-4DB8-B084-7431EEEB9530/240841C1-288F-4CB5-B4BD-77C300C49201.gif)

然而无论是需要安装插件的 VS Code、Atom 和 Sublime，还是 Pycharm 这样的 IDE，都能很好地支持 Python 的类型注解写法。

定论 | CodeRunner 的好与坏
--------------------

编辑器之争一直以来都是个能引发无数开发者争议的话题，每个人心目中都有一个「The Best」候选。通过文中的一些介绍，也可以看到 CodeRunner 相比于目前主流编辑器而言的一些优势之处：相对容易上手、配置简单、无需额外的插件支持等。它在一些**快速验证想法**或是**编写简单的脚本**这类场景上，确实会相对易用一些，因为我们无需还要指定对应的配置然后再编写代码，而是直接开箱即用。

可和目前主流的编辑器或 IDE 比起来它也许并不能胜任较为**工程化或项目化**的任务，甚至可能就不如 IDE 来得更加全面；不仅如此，CodeRunner 虽然也在官网上宣称了对某些语言提供 IDE 级别的补全功能，但在实际使用上也效果确实十分一般，甚至也出现较为差的情况。

但 CodeRunner 和目前被广泛使用的编辑器们最大的差异还是在于**它并不支持插件或扩展**，无论是已经有了年头的 Vi/Vim 和 GNU Emacs，还是现在主流的编辑器或 IDE，没有一个不是支持用户或社区生态插件或扩展的；加上它仅仅只适合于 macOS 平台，因此对于使用者来说除了要额外投入物质金钱之外，更多的还需要考虑到行为习惯、配置迁移等成本。
