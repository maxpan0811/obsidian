抬高 iOS 设计「地板」的图标字体：SF Symbols
=============================

| 本文为付费栏目文章，您已订阅，可阅读全文 |

引言
--

图标是界面的重要构成元素。恰当地在设计中使用图标，不仅能让应用更加简洁美观，也有助于用户快速理解和高效使用各种功能。

图标也是统一视觉风格、强化平台辨识度的重要工具。很多知名应用在改进设计时都把图标作为重点，就是考虑到其对于构建品牌形象的作用。

![](.evernote-content/2BF90509-6625-4BF5-BB59-3CBF732E5FDE/00F77C5D-2816-4D83-A6B3-E38FB13E6EC0.png)

Uber 和 Spotify 的图标设计体现出很强的品牌辨识度

然而，一向以设计见长的苹果，近年来在图标方面的表现并不出色。iOS 7 以来，苹果的图标设计风格似乎一直处于不断的摸索和调整中，经历了从方正到圆润、纤细到醒目的渐变。结果，系统各处散落着不同风格的图标，在让用户困惑的同时，也让开发者感到无所适从、缺少指引。

![](.evernote-content/2BF90509-6625-4BF5-BB59-3CBF732E5FDE/B2BFB9C8-86C8-48FB-9368-86E437BAEFB5.png)

iOS 中的「分享」按钮有不下八种版本（来源：Benjamin Mayo）

苹果此前为应用开发提供的设计资源也比较有限。很多第三方开发者、尤其是独立开发者没有经过专业的设计训练，也缺少足够的精力和资源放在图标设计上。为此，包括 Android 和 Windows 在内的平台都提供了符合各自设计标准的图标库，供开发者设计界面时从中选用，而 iOS/macOS 平台却一直缺乏这样的官方资源。这显然不利于提高平台应用的整体水平和风格一致性。

今年，这一现状有望得到改观。六月举办的 WWDC 上，苹果推出了 [SF Symbols](https://developer.apple.com/design/human-interface-guidelines/sf-symbols/overview/)—— 一个规模庞大、与系统紧密整合的图标资源库。SF Symbols 的内容不仅包括系统服务、应用功能，还包括天气、手势、物件等，能在很大程度上满足开发的常见需要。

![](.evernote-content/2BF90509-6625-4BF5-BB59-3CBF732E5FDE/EA37E868-99B3-42B4-A232-96400104C383.png)

SF Symbols（来源：苹果）

图标与字体的结合
--------

不过，SF Symbols 的最大亮点并不在于提供了一套丰富的图标，而在于它的实现方式 —— 直接把图标整合进了 iOS 和 macOS 的系统字体 San Francisco。苹果在宣传是也着重强调了 SF Symbol 的「文字」特性。用它自己的话来说，「SF Symbols 引入了一套完整的、基于矢量的图标库，你可以凭借其与周围文本的自动对齐和对多字重、多尺寸的支持，将其整合到自己的应用中，从而简化用户界面的布局。」

那么，苹果为什么选择将图标资源放在字体中？这种做法为什么具有苹果宣称的优点？

以字体的形式提供图标（一般称为「图标字体」，icon font）实际上已经是一种流行的做法，在网页环境中的应用尤其广泛。或许你没有意识到，我们日常使用的网站上的很多图标，都是以字体的形式提供的。

![](.evernote-content/2BF90509-6625-4BF5-BB59-3CBF732E5FDE/003FE367-E9C7-4EF1-811C-7F85792D0625.png)

以字体形式提供的图标

这项技术的原理也不复杂。字体文件可以看作是一系列带有编号的图案的集合。在显示文字时，软件通过编号（例如 Unicode 编码）从字体文件中定位到所需的图案，进而将其绘制到屏幕上。因此，只要将图标设计以同样的方式编号并打包成一个字体文件，就可以通过与显示文字相同的方式显示图标。

在过去，iOS 应用原生支持的图标资源格式主要是 PNG 这种位图格式。与之相比，图标字体具有一些先天优势。最明显的一点在于，**字体是矢量的，用它做图标意味着放大缩小不会导致失真。**这是应对当前设备尺寸多元化、动态布局流行的开发环境所亟需的。

![](.evernote-content/2BF90509-6625-4BF5-BB59-3CBF732E5FDE/583550E8-4248-4565-9D94-14B7BCB87B84.png)

矢量的图标字体可以不失真地适应各种尺寸

其次，**图标字体具有文本的属性，可以像普通字符一样调整字号、字重、颜色、对齐方式等。**因为图标在界面中很少孤立出现，而是要与其他图标和文本搭配，图标字体的文本特性能使其更方便地融入到周围的元素中。

![](.evernote-content/2BF90509-6625-4BF5-BB59-3CBF732E5FDE/C887550C-2BB7-4371-B8FB-545C7B60B742.png)

图标字体的文本属性在与其他文字搭配时显得非常方便

此外，每个位图图标都需要一个单独的资源文件，而图标字体可以在一个文件中打包设计中用到的所有图标，这对**提高调用效率和性能都有帮助。**

（**注：**值得指出的是，苹果内部负责该项目的工程师明确[表示](https://twitter.com/Inferis/status/1136902323609858048)，SF Symbols 并不是普通的图标字体，而更接近于存储在字体文件中的图片。且与 emoji 那种在字体中存储位图信息的做法不同，SF Symbols 在字体中仍然是矢量的，只有在显示时才会通过某种机制转化为图片。不过，这并不影响 SF Symbols 在用途和功能上与其他图标字体有可类比之处，故本文在讨论时没有刻意区分。）

后来居上
----

在 SF Symbols 以前，市场上已经有很多套图标字体方案，其中不少还是免费的。例如，知名的第三方图标字体集 Font Awesome 已经有七年的更新历史。类似的项目还包括 [IcoMoon](https://icomoon.io/) 和 [Friconix](https://friconix.com/) 等。

![](.evernote-content/2BF90509-6625-4BF5-BB59-3CBF732E5FDE/0CDDFB7F-3F09-4D8F-8A64-A731AA019221.png)

流行的第三方图标字体集 Font Awesome

为操作系统提供图标资源也不是苹果的独创。上世纪九十年代，Windows 就内置了三套名为 [Wingdings](https://en.wikipedia.org/wiki/Wingdings) 的装饰字体，某种程度上可以看作图标字体的先驱。现代的 Windows 在默认界面字体 Segoe UI 中整合了称为 [Segoe MDL2](https://docs.microsoft.com/en-us/windows/uwp/design/style/segoe-ui-symbol-font) 的图标资源，SF Symbols 的功能和用法都与其有类似之处。Google 也为 Android 系统的 Material 设计系统提供了配套的[图标字体资源](http://google.github.io/material-design-icons/#icon-font-for-the-web)。

![](.evernote-content/2BF90509-6625-4BF5-BB59-3CBF732E5FDE/130441C9-365C-40C8-B5A5-6649ECE499A2.png)

微软为 Windows 准备的 Segoe MDL2

但与这些先行者相比，SF Symbols 仍然体现出了不少优势。

**首先是数量齐全且用途灵活。**目前版本的 SF Symbols 包含 1584 个不同图标，日后应该还会扩充，显著多于 Segoe MDL2（1072 个）和 Material 图标（932 个）的数量。不仅如此，每个图标都有多达 27 个变种，包括大、中、小三个尺寸和从极细（ultralight）到极粗（black）九种字重（即粗细程度），能够满足多种场景下的需要。如果既有的设计不能满足需要，开发者还可以通过模版自行修改、制作图标，然后导入到 Xcode 中，并获得与原有图标一致的丰富功能。

![](.evernote-content/2BF90509-6625-4BF5-BB59-3CBF732E5FDE/5C3A4A50-0321-4746-94D8-33FAEBAFA895.png)

SF Symbols 的丰富变化（来源：苹果）

**其次是与系统的整合格外紧密。**大多数图标字体都会给每个图标起一个名称，开发者进而通过这些名称调用需要的图标，但这么做的前提是要先从外部加载图标字体资源。SF Symbols 中的每个图标也有一个系统名称（system name），但因为它们是直接整合在系统默认字体里的，无需额外步骤就可以简单地通过形如

```
UIImage(systemName: "circle")
```

的语法来调用。又因为这些名称属于一个独立的命名空间（`systemName`），它们始终不会和自定义图标冲突，即使名称重复。

![](.evernote-content/2BF90509-6625-4BF5-BB59-3CBF732E5FDE/ECA9E816-94DE-4018-B87F-2B3AEDAA1B70.png)

SF Symbols 在调用时受到特别的对待（来源：苹果）

调整图标外观的方式也很灵活。一方面，如果 SF Symbols 被用作按钮图标，其外观会根据设备的类型、屏幕的方向、按钮的作用等因素而自动变化。另一方面，苹果还提供了专门的接口，用于像调整文本一样调整图标的磅数、字重、样式，或者指定某个视图下的默认图标样式等。这些功能是其他图标字体中鲜见的。

**第三，SF Symbols 充分考虑了图标与文本的配合。**在使用普通图标时，开发者需要计算并指定图标的尺寸、和文本的距离等参数，这个过程繁琐且容易出错。而使用 SF Symbols 则只要像对待文字一样为图标设置一个「字号」即可。这个字号并不代表图标实际占据的尺寸，而是系统根据配合的文本自动缩放到的最佳尺寸。（指定实际尺寸仍然是可行的，但苹果明确[不鼓励](https://developer.apple.com/videos/play/wwdc2019-206/?time=1199)这么做。）

![](.evernote-content/2BF90509-6625-4BF5-BB59-3CBF732E5FDE/C47526C7-4C1E-474E-AD02-F6AADFCE5484.png)

SF Symbols 的「字号」并不代表实际显示的尺寸（来源：苹果）

为了保证不同形状的图标都能和谐地与文字对齐，苹果还针对性地调整了每个图标的基线（baseline）。在排版中，基线指小写字母 x 底部所在的直线，是字符之间相互对齐的参考线。显然，如果笼统地认为图标的基线都是其下边缘，它们在对齐时就会因为尺寸不同而显得忽高忽低。苹果考虑到了这个问题，按照不同图标的形状和特征设置了合理的基线位置，并允许调整自制图标的基线。

![](.evernote-content/2BF90509-6625-4BF5-BB59-3CBF732E5FDE/016DCE1A-7169-45C2-B33E-23B5FC443FD8.png)

苹果为 SF Symbols 中的图标按照形态特征设定了合适的基线

除了当作按钮图标使用，SF Symbols 也可以插入到文本中，让界面中的文字描述更加简洁生动。这时，图标会自动适应周围文字的尺寸、字重和颜色。

还有一个小细节：SF Symbols 中的数字和字母都使用了 SF Round 字体（即系统自带「备忘录」应用的默认字体），进一步促进了与系统字体的风格一致。

实现原理与官方应用
---------

如上所述，SF Symbols 是整合在 iOS 和 macOS 的默认界面字体 San Francisco 中的，在新版系统下不需要加载额外的资源就可以调用。与其他很多图标字体的做法类似，图标的编码位于 Unicode 的 Supplemental Private Use Area-B 区块（U+100000 以后）中。

![](.evernote-content/2BF90509-6625-4BF5-BB59-3CBF732E5FDE/E0007B64-C0C5-4D04-B440-7514CD059ED9.png)

SF Symbols 在 SF 字体中的编码位置

SF Symbols 的各种变体 —— 包括大、小两种额外尺寸、圆形 / 方形边框、实心颜色等 —— 则存储在更靠后的、未定义的码位上。它们通过 OpenType 字体的替代样式集（alternative stylistic set）功能与原始尺寸的图标相关联。

![](.evernote-content/2BF90509-6625-4BF5-BB59-3CBF732E5FDE/EEAB5AA2-7D31-45C6-A5E5-DA65B01C8222.png)

SF Symbols 通过 OpenType 特性实现多尺寸支持

在支持调整 OpenType 特性的软件（包括系统内置的「文本编辑」应用）中输入并选中一个 SF Symbols 图标，然后打开字符选项面板，就可以测试这些变体。

![](.evernote-content/2BF90509-6625-4BF5-BB59-3CBF732E5FDE/1A9EC63F-B0EB-46A1-AF64-DED5F3CCC85C.gif)

在文本编辑应用中测试 SF Symbols 的变种

为了方便开发者查阅和使用 SF Symbols 资源，苹果还专门制作了一个[同名应用](https://developer.apple.com/design/downloads/SF-Symbols.dmg)（链接为官方安装包）。打开后，用户可以以网格或列表形式浏览 SF Symbols 提供的图标，在不同字重间切换，或者通过名称搜索特定图标。找到所需的图标后，可以按 ⌘C 复制这个字符本身，或者按 ⇧⌘C 复制它的系统名称以便在 Xcode 中调用。

![](.evernote-content/2BF90509-6625-4BF5-BB59-3CBF732E5FDE/BD82B8A9-BEAA-4B52-893E-923A5B3E6108.png)

SF Symbols 应用（来源：苹果）

SF Symbols 应用的另一个重要作用在于导出图标的模板以供自行修改。选中一个图标后按 ⌘E，就可以导出它的 SVG 模板。用矢量编辑工具打开后，可以看到一个包含 27 种不同尺寸、字重变种的矩阵。新版的 Xcode 可以直接读取这种模板，在其基础上修改并导入后的图标，能获得与自带图标一致的功能。如果开发者没有精力为同一个图标制作全部 27 个变种，也可以先只制作 regular 字重、medium 尺寸这一项，然后根据应用的需求补充其他用得上的变种。

![](.evernote-content/2BF90509-6625-4BF5-BB59-3CBF732E5FDE/96CFB74D-CE70-4223-A212-6F69F8D434F1.png)

Xcode 可以识别通过 SF Symbols 应用导出并修改的图标模板

对于日常用户而言，SF Symbols 应用的用途则比较有限。尽管其中一些图标或许可以在文档和幻灯片中使用，但根据测试，SF Symbols 在 Word 等第三方软件中兼容性不佳，只有手动将字体改为 SF 系列字体才能显示出来。另外，由于这套图标是嵌入在 SF 字体中的，严格来说其用途也必须遵守 SF 字体许可证的要求 —— 仅限用于苹果平台软件的界面设计。

SF Symbols 的不足
--------------

尽管 SF Symbols 已经显示出很大的潜力，但在目前阶段还称不上完善。专业用户已经从不同角度对 SF Symbols 的实用性和质量提出了疑问。

**首先，SF Symbols 提供的接口只适用于 UIKit。**这意味着只有 iOS 应用、以及通过 Catalyst 技术反向移植到 macOS 上的应用才能享用 SF Symbols 的便利，而使用传统的 AppKit 开发的原生 macOS 应用则被排除在外。这种厚此薄彼的做法似乎有违苹果重新重视 Mac 平台的承诺。

**其次，苹果对 SF Symbols 的使用范围施加了不甚合理的限制。**SF Symbols 中有多达 83 个图标被留作所谓[「特定用途」](https://developer.apple.com/design/human-interface-guidelines/sf-symbols/overview/#symbols-for-use-as-is)（symbols for use as-is），苹果规定它们只能用于指代对应的苹果产品或功能，不得用于其他目的，也不允许导出修改。这些特定用途图标中，有一些的确是苹果服务的专用图案，例如 AirPlay、Face ID 等，对它们的使用加以限制是可以理解的。但另外一些图标 —— 例如 16 个含有「听筒」造型的图标和 8 个含有「信封」造型的图标 —— 则与苹果产品没有必然关系，且本身就可以被解读为多种含义，为它们「钦点」特定用途就显得很不合理了。更何况，苹果自己似乎也完全没有遵守这些特定用途的意思：为 iBooks 预留的「书本」图标在 Safari 里被堂而皇之地当作「书签」图标使用；WWDC 上的一些讲座中，苹果工程师也把「信封」图标挪用到了登录界面上。

![](.evernote-content/2BF90509-6625-4BF5-BB59-3CBF732E5FDE/293DDE97-7BE2-478B-A736-39FA0970D287.png)

SF Symbols 中的「特定用途」图标

**SF Symbols 的设计也存在一些不合理之处。**例如，有设计师[指出](https://twitter.com/eli_schiff/status/1135655105758240768) SF Symbols 图标的边角处刻意留出了细微的「缺口」。这种称为 [ink trap](https://en.wikipedia.org/wiki/Ink_trap)（油墨扩张弥补）的设计一般是用于书面印刷的，其目的是为了缓解小字号印刷时，由于油墨扩散导致的文字边缘模糊。通过在文字的边角处预先「挖」出一小部分，油墨在扩散时就会自然地填进这些缺口，最终呈现出一个完整的形状。然而，SF Symbols 显然在绝大多数场景下都是用于屏幕显示的，为其加入一个印刷时才需要的妥协设计，似乎是弊大于利。

![](.evernote-content/2BF90509-6625-4BF5-BB59-3CBF732E5FDE/5C6E91FF-18E3-407C-8DB2-0F2594E61AC5.png)

SF Symbols 中的 ink traps

**另外，SF Symbols 在目前的测试版系统中还出现了明显的显示模糊问题。**在图标设计中，一项重要的原则是图标的边缘应当对齐到像素（pixel perfect）。否则，由于屏幕没有办法显示半个像素，系统就只能通过次像素渲染等方式显示出一个近似效果，这种妥协必然导致失真和模糊。

然而，目前 SF Symbols 中很多图案的设计都没有对齐到像素。更何况，由于 SF Symbols 的理念是动态地调整尺寸和位置来配合文本，它从性质上就是难以做到像素对齐的。在前三个版本的 iOS 13 测试版中，系统 bug 还会导致没有对齐到像素的图标边缘被切割掉，进一步劣化了显示效果。Beta 4 在一定程度上[改善](https://twitter.com/marcedwards/status/1151723058907848705)了图标模糊的问题，但仍然没有恢复到 iOS 12 时的锐利程度。正是因为这个问题，知名工具 iStat Menu 的开发者对 SF Symbols 的可用性[持怀疑态度](https://twitter.com/marcedwards/status/1137140209911455744)，并表示暂时不会考虑在开发中使用这套图标。

![](.evernote-content/2BF90509-6625-4BF5-BB59-3CBF732E5FDE/8CE67B84-B03B-4448-A73F-AEBCC8DAF2B7.png)

SF Symbols 相比于旧版 iOS 图标显得模糊（来源：Marc Edwards (@marcedwards)）

结语
--

整体观察今年 WWDC 上发布的设计相关功能，一个共同的主题是将很多原本基于数值的配置**语义化**。设置颜色、字号的方法不再是填写具体的数值，而是[声明元素的用途](https://developer.apple.com/design/human-interface-guidelines/ios/visual-design/color/#system-colors)；界面代码的写法不再是逐步控制界面的构建（指令式），而是直接[描述界面上有什么元素](https://developer.apple.com/tutorials/swiftui)（声明式）。类似地，SF Symbols 将添加图标从「设计 —— 导入 —— 引用」的多步流程简化为了一条指令：你只用告诉系统某处需要一个表达某种含义的图标就行了。通过将更多的实现细节留给系统，这些功能实际上强化了苹果对第三方应用界面的控制权：如果它日后再次改变系统的图标、配色设计，通过上述新方法调用系统资源的应用也将自动换上新外观，而无需等待开发者手动适配。因此，进一步统一和规范苹果平台应用的风格，是这些变化背后共同的目的。

![](.evernote-content/2BF90509-6625-4BF5-BB59-3CBF732E5FDE/F9E4A063-5420-4836-ABE1-49E5E875AA54.png)

SF Symbols 的加入让 iOS 13 的界面更加易用

但无论苹果的动机为何，SF Symbols 的出现仍然对于开发者和用户有很大的积极意义。SF Symbols 的设计固然称不上完美，但作为系统内置的设计资源，这套图标的意义并不在于设定 iOS 应用的「天花板」，而在于抬高了「地板」：精力有限的开发者可以借此更轻松地做出质量过关、接近原生的界面设计，而有实力、有资源的应用仍然可以自由探索更新颖、更美观的方案。用户进而可以从整体提升了的应用设计水平中获益。当然，SF Symbols 能在多大程度上满足开发者的需求、目前发现的问题能否得到改进，还要等到新系统发布后再进一步观察。
