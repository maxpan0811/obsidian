## 思路：为什么选择 Notion 做会议记录？

在 Notion 之前，我们使用的是 Trello，每次开会时大家会一起打开 Trello，对当周的选题、素材进行讨论。讨论完毕后，将卡片拖动到代表不同进度的列表中，再将选题卡片分配到人。

![](attachments/8e4535a92640cf42.jpg)

原先 Trello 的流程

Trello 的看板形式原本是非常适合这个流程的。但是随着时间的推移，数据量逐渐增多时，我们发现 Trello 开始变得不那么适合这项工作了。问题主要体现在下面的三个方面 ——

1. **维度太少**：Trello 的每张卡片中，可以包含「成员、描述、附件、标签」等基础数据，但我们无法为每张卡片自定义更多数据，比如只有到期日而没有起始日，只有成员但没有成员职责等。
2. **记录不便**：开会不可避免地会发生讨论，这些讨论有时候需要被记录下来。而 Trello 是一个卡片式的工具，在讨论中需要同时用到多个卡片时，或者讨论内容与卡片无关时，Trello 便无法处理，需要另外一个工具记录灵感。
3. **难以整理**：看板的形式注定难以管理大量的卡片，而选题往往是从大量卡片中选用的。当积累大量等待处理却又无法归档的卡片时，就需要使用更多的分类来解决目前的问题。时间一久，看板就会变得臃肿。

我们希望能够以更系统的方式对每一次有价值的讨论进行记录，保证复盘时都有据可循，便于及时更新选题进度，避免囤积过期选题。

在这样的要求下，我们需要的就不只是一个「可以记录」的地方，而是需要一个能够聚合多种内容的空间，能够面向会议流程，将所有需要用到的数据串联起来，让与会人员高效地处理每一件需要处理的任务。

在记录讨论这方面，最好的方式自然是文本形式，而在处理数据方面，更好的方法却是表格形式。我们曾用两种方式来解决这两种形式之间的冲突，比如同时使用 Quip 和 Trello，分别用于记录会议流程和处理卡片数据。但效果并不理想，割裂感强、需要来回切换、难以对应数据。

![](attachments/811ad8dd6e2e5296.png)

需要在两个应用之间切换

经过对比，我们最终发现，能够**将文本和表格形式做到无缝融合**的最佳工具就是 Notion。我们可以用 Notion 同时完成这两项任务，还可以解决上面提到的 Trello 的问题，实现了一加一大于二的效果。

## 会议流程

会议是一个流程化的工作，会有具体的议题、需要共同处理的数据、对关键任务进行决策。

我们设计了一个流程，来舒适地完成会议记录这项工作。你可以打开上面的会议模板，看看我们是如何通过这一流程实现会议记录的。

### 会议准备

Power+ 的选题会中，每次会议都需要完成多个固定的议题：

1. 更新历史选题进度
2. 讨论新的选题、素材、新闻
3. 对未来几周的文章进行排期
4. 对上周文章进行点评、互评
5. 讨论读者反馈意见、选题建议
6. 其他事务

这些议题是我们每周都需要讨论的内容，为了不在会议期间浪费过多的时间，肯定不能两手空空地开会。每个人都应该提前准备需要讨论的内容。

#### 需要准备的数据

根据会议的议程，我们需要下面这些数据，有了这些数据，才能进行任务的剖析、分配、推进：

1. 选题箱
2. 知识库
3. 文章排期
4. 读者反馈
5. 标签系统

![](attachments/7b8cea58540d1ee6.png)

原始数据库

其中，选题箱承担了前两项议程，文章排期承担了第三、第四项议程，读者反馈承担了第五项议程。知识库则是用于存储值得收藏选题的。这些数据库就组成了我们开会需要讨论的原始数据。

#### 收集和添加原始数据

原先通过 Trello 开会时，我们会在 RSS 中进行大量阅读，从中筛选出可能成为选题的新闻、消息、素材，添加特定标签收藏到 Pocket，再由 IFTTT 将选题自动发送到 Trello。这样就能实现添加选题流程的自动化。

由于 Notion 至今还未支持 API 和自动化，我们目前的方案是保留原有的 Pocket + IFTTT + Trello 的方法实现自动化添加选题，在每次开会前由 @Fairyex 通过 Tasker 将存储在 Trello 的选题导出为 JSON 格式，再以半自动的方式导入到 Notion 中。这一方法需要等待 Notion 官方提供 API 的支持才能继续优化。

如果将来 Notion 能够直接支持 API，这一个步骤就可以一次性省去 IFTTT、Trello、Tasker，实现从 Pocket 到 Notion 的一步到位式选题添加，进一步减少手动操作。

![](attachments/4796863ec0d6036a.png)

半自动添加选题的流程图

就目前而言，仍然需要我们动手做的事情，无非是像以前一样为选题素材打上标签，只是多了一步「会议前导入选题」的步骤，已经算是足够简便了。

其他需要会前收集的数据，也就只有「读者反馈」和「选题建议」了，由于这两者都是来自读者自发提供的，分散在各种平台不受我们控制，所以无法做到完全自动化。但我们也想了一些办法，让这些数据的收集变得尽可能地简便。所以我们在 Slack 群中设有 `#选题建议` 频道，我们每周都会对读者提出的建议进行讨论。通过设立相对集中的反馈渠道，可以有效减少我们在收集数据时的工作量。

完成了数据收集工作，接下来那就让我们开始会议吧。

### 开始会议

会议开始之后，我们会按照上面提到的议程依次讨论。

![](attachments/2d905b0cd16d409a.png)

由于模板中已经完成了自动化的筛选，在导入选题的同时，就可以在会议记录中查看和编辑这些选题（关于如何制作这种形式的模板，文章后半部分的「流程总结」中会详细提及）。

#### 处理历史和新鲜选题

这是我们的会议模板，在真实工作环境下，每张表格中都会有几十行的信息，长度是这张图片的几倍。

![](attachments/0407563f5080ec00.png)

Power+ 会议记录分为这几个部分，其中是精简过的演示数据

首先是处理历史选题，大部分素材都被用于每周五的奏折，因此许多素材在第二周就会立即失效，我们会优先查看被标记为「奏折素材」的选题，将它们的进度调整为「已完成」，这样可以及时将有时效性的选题素材进行归档。如果暂时素材没有被使用，我们会将其标记为「选题预备」，并将其保留一段时间。为了保证页面不被过多的预备选题占用，超过一个月的选题也会被暂时隐藏，也不会显示在这里。当有朝一日需要用到时，我们可以直接跳转到 选题箱页面进行查看。

接下来是讨论新的选题，一个选题应当具备很多要素，最基本的有标题、摘要、相关链接，然后需要有选题添加者、选题负责人，在讨论之后，应该明确选题的用途，并及时更新进度。

如果可以，最好能为每一个选题打上一些标签，以便我们对选题的方向和类别进行分析，也方便会后分配任务。

![](attachments/8938fd7e047e0be2.png)

会议期间打上标签

我们还在会议模板中放置了 知识库、 文章排期这两个页面的跳转链接。分别用于存放「无法成为选题但值得收藏的选题」和「可以立即准备完成文章的选题」，在经过讨论之后，我们会把符合这两个条件的选题 选题箱中拖走，进入写稿分配流程，这里与本文关系不大，暂且不提。

我们每周还会对上周文章进行评分，并评选出一篇最佳文章。由于涉及到计算规则和分值权重，公开分享的模板中没有加入这部分内容。

![](attachments/7c29c145a33c6c18.png)

互相点评上周文章

#### 用单独页面记录重点

当会议进行到重点讨论阶段时，我们往往会发现一些内容需要记录更详细的内容。

我们曾尝试在石墨文档中以表格的形式讨论选题，发现纯表格无法像 Trello 的卡片一样展开，因此在纯表格中，很难为一行信息添加更详细的备注。

而 Notion 的 Database 就非常独特，除了具备表格的功能，还有一些有意思的特性。比如每一行都能展开为单独的页面，所以 Notion 中就不存在这样的问题。如果一个选题需要进行更详细的讨论，可以像打开一张卡片一样，直接打开表格中的一个条目，将一行数据转为一个 Page，在这个页面中添加详细信息，同时不影响之前的数据。被转为 Page 的行会多一个图标，表示可以打开。

![](attachments/275984472038381c.png)

在表格中添加 Page 记录更多内容

可别小看这个小小的功能，它不仅具备了表格的优点，还能像卡片一样被展开。这能让每一个条目都承载大量的信息，解决了文章开头提到的「记录不便」问题。

### 会后检查

在会议结束之后，我们需要将讨论的内容付诸实践。由于素材内容比较零碎，通常是由 @文刀 完整安排一次任务，我们在选题库中检查自己的任务。

在会议期间分配了每个人的任务，那么在会议结束之后，如何查看属于自己的任务呢？Notion 给出的答案是多视图功能。

![](attachments/d7cafe77c87c987a.png)

我仍然可以选择用看板模式

Notion 的 Database 支持多视图功能（View），这些视图能够保持各自的样式、筛选、排序。这样我们就能为每个人分配一个视图，比如在 Minja 的专属视图下，Minja 可以看到分配给他的所有任务，当他对这一视图中的排序、样式进行改动时，其他人的视图不会发生变动。如果 Minja 对内容进行了改动，那么所有人看到的内容都会有所改变。所以 Minja 可以以他喜欢的形式（比如看板）组织自己的视图，并及时更新任务的进度让其他人知晓。

这样一来，每个人都能够通过自己喜欢的视图来管理任务。

![](attachments/bf3baf59673a66db.png)

甚至是以日历视图

## 流程总结

其实在这套工作流程的背后，还有许多大家没有看到的地方。最为复杂的就是上面的会议模板，看上去只是一页会议记录，但在实际使用过程中，一套初期搭建的模板总会遇到一些问题。我们根据使用中遇到的问题不断优化，最终稳定在一个可以长期使用的版本。

下面就来讲讲会议模板中还有哪些值得注意的要点。

### 实现内容聚合

如果每次开会都需要在这些页面中进行切换，其麻烦程度并不比使用两个工具低，如何将它们聚合起来就成了一个难题。

好在 Notion 的一大特性就是可以帮我们聚合信息，**它能在单个页面中聚合多个表格内容**。大家在上面看到的表格，并不是存放在某一份单独的会议记录中的，而是我们充分利用了 Notion 的特点，将多个表格中的数据集合到会议模板中。实现的方法是利用 Notion 中的 **Linked Database** （连接数据库）功能。

Linked Database 的作用是在文章内引用另一个大型表格的数据，当你在文章中更新表格内容时，另一个大型表格中的数据也会被更新。这和网站内插入一个 Google Sheets 页面有些类似。

![](attachments/c6c94b4f8fde99fc.png)

Linked Database 相当于关联了大表的一部分

通过这种方式引用的表格，既能让与会人员快速更新数据，又能通过文字记录会议流程。而且在未来回顾时不必再单独打开会议记录查看。

### 准备会议模板

了解了 Linked Database 的用法，事情就变得轻松许多。接下来就可以将文本部分和表格部分进行结合。

![](attachments/91b64e5d374afb48.png)

一份会议记录由多个 Linked Database 组成

Notion 的文本内容都是记录在 Page 中的，我们新建一个 Page，预先设计一版会议记录模板。这个模板中以议题流程作为小标题，方便我们在会议过程中进行实时记录。然后将它和其他数据库进行关联，但是如果直接关联，会让整个表格的所有数据出现在 Page 中，不能这么做。

我们要做的就是**在这些表格中各取一小块需要用的内容**（比如本周需要讨论的内容），再通过 Linked Database 功能，整合到一个单独的 Page 上。具体操作起来，就需要事先对表格内容进行自动筛选和排序，筛选出每次会议需要用到的内容。

比如我们需要讨论历史选题进度、新的选题时，需要用到的内容都来自 选题箱，在筛选历史选题时就需要筛选出「本月内、一周前、非归档、非过期」的选题；在筛选新的选题时就需要筛选出「添加时间为本周」的选题。

![](attachments/084cd8ca07efc6f9.png)

对当周选题进行筛选

排序也是根据会议流程指定的，比如我们习惯依次发言，引用表格就会按照「选题负责人、选题添加人、选题用途」等次序依次进行排序。这样每个人可以将注意力集中在自己需要负责的内容上。

![](attachments/f958c6eab1194ddf.png)

对筛选结果进行排序

制作完一次模板后，将来就再也不需要这么复杂了。每次会议前只需要将这份模板复制一份，修改日期，其中的内容就会立即更新为我们需要的内容，十分省时省力。

在开会时，每个人都只需打开这份会议记录，与会人员可以在同一个视图中看到所有开会所需的内容，这些内容会根据负责人进行排序方便讨论，非常符合会议流程。我们要做的就是记录下自己的那部分内容，会后自然就能形成一份完整的会议记录了。

### 打上统一标签

「打标签」这件事似乎可以在任意一款表格工具中轻松实现，但 Notion 的特别之处在于它能在 Database 之间、数据之间实现很强的关联性。除了 Linked Database，Notion 的表格内部提供了 Relation（关联）功能，能够引用其他表格中的某列数据。

一个最典型的例子就是共用标签系统。

![](attachments/e58a77e8e52aa587.png)

有上百个标签的标签库

简单的少量标签可以通过多选框的形式实现，但如果你有成百上千个标签需要在**多个表格中共用**，那可能需要引入一个外部的标签表格。你可以通过表格的 Relation 功能，将另一个表格的某一列引用到这里，这一列的数据可以在标签系统中进行数据统计、单独分析，甚至为每个标签加上一个 Emoji 让它们更美观。这样，我们就可以在「选题、排期、知识库」等多个库中为素材打上标签。

这个技巧在《[Notion 进阶技巧：学会使用「组合技」 | 工作日志](https://sspai.com/post/52769)》的「高级功能二：Relation 关联」部分有详细的描述，这里就不再赘述。

长此以往，我们就可以根据标签的数据，看看文章的类型、选题角度是否需要注意和修正。还可以结合网站后台的数据一起进行分析，来调整未来的选题方向。

## 总结

得益于 Notion 的 Linked Database 功能，我们可以在 会议记录中引用 选题箱、 读者反馈等其他表格中的内容。同时由于会议记录本身是文字形式，我们可以在任意位置加入会议中的额外讨论和灵感，让整个会议变成流程式的结构。

整个流程其实并不复杂，建立会议记录模板看似麻烦，但恰恰是能够为后期节省大量时间的做法。如果感兴趣，可以在这里查看完整的会议记录和演示数据库，将它们完整保存到自己的 Workspace 中。
