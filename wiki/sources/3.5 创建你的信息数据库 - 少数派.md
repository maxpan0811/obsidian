# 3.5 创建你的信息数据库 - 少数派

---

3.5 创建你的信息数据库
=============

你好，我是 Louiscard，这是你打造信息管理系统的第 14 天，今天我来跟你聊聊如何构建你的信息管理数据库。

在整个教程中，这是我投入最多时间的一部分，也是我认为含金量最高的内容，想明白这件事更是我从信息管理入门到进阶的重要里程碑。

「知识管理」的困境
---------

相对于信息管理，我在初期更多使用「知识管理」这个概念。在「知识管理」的框架下，我更多聚焦于「文章」或者「课程」这类完整的信息组块，在一开始更是 All in「印象笔记」，在我眼里最小的信息单位就是一篇篇剪藏的文章。于是，我疯狂囤积优质文章，积累了成百上千，最终达到了近万条笔记。

这样的信息输入帮助我建立了最初的专业认知，然而这种粗放的信息搬运工作，在实践中，效果总是差强人意。比如我在工作生活中需要查找调取信息，比如撰写文章或准备分享的时候，虽然搜索关键词能在笔记本中找到几篇质量不错的相关文章，但这些文章并不能给我立竿见影的支持，大多数还需要耐着性子从头读到尾，有时候感觉这和直接在互联网搜索差别也不大，帮助有限。

然而，具体怎么做才能提升信息的使用效率，这个问题困扰了我相当长的时间，直到我听到了 API 这个概念。

信息管理的 API 接口
------------

有一天我在[《湾区日报》](https://wanqu.co/)偶然读到这样一段话：

> 不只是面试，在很多社交场合都需要介绍自己，介绍自己做过的事，要会讲故事、让人有兴趣听下去。平时要准备一些小段的、不同角度的自我介绍，要用的时候就像调 API 一样方便。
>
> 名人在电视上接受采访，好像都能即兴演讲、侃侃而谈；都是平时有准备、有练习、不断 A / B testing、不断迭代的，最后总会在合适的时候呈现出合适的版本的说辞。

仿佛被闪电击中了一般，这不就是我理想中的信息使用效果么！于是我赶紧查了查 API 的概念，Wikipedia（维基百科）是这么介绍的：

> API（Application Programming Interface，应用程序编程接口）是一些预先定义的函数，目的是提供应用程序与开发人员基于某软件或硬件得以访问一组例程的能力，而又无需访问源码，或理解内部工作机制的细节。

听起来有点复杂，其实非常简单，说白了就是预先设定好信息输入、输出规则，能够提供稳定的信息调取能力，满足相应的信息需求，类似我们在手机中点击时钟获取时间信息，点击天气获得天气信息一样。

于是我开始琢磨，怎么样才能让个人的信息管理系统实现 API 接口的这种效果，在需要信息的时候直接调取，而不再需要花费额外的时间重复阅读呢？

顺着 API 的概念往上追溯，找到了解决的最终答案：「数据库」。你也许也在大学的计算机课上学习过 Access 数据库的知识，但是大概率像我一样毕业之后就再也没再打开过。

然而，要想实现 API 的能力，背后其实就靠这一张张结构化的数据表格构成的数据库发挥作用。

![](.evernote-content/3270CDD3-244B-47DA-9C9C-1EF4E7F03351/A8931686-8B18-409D-A868-BE0CD03F42BE.0)

数据库的信息的特点是「结构化」，也就是说存储的信息不是一篇篇非结构化的文章，而是一条条准确可被直接调取的信息。

拿上面的「自我介绍」来举例子，原先我积累的信息都是类似下面这些经验分享的文章。

* 如何做一个生动的自我介绍；
* 会讲故事才能让别人记住你；
* 自我介绍的 5 个套路。

但如果实现 API 的效果，搭建「自我介绍」数据库，里面记录更应该是：

![](.evernote-content/3270CDD3-244B-47DA-9C9C-1EF4E7F03351/BA865ADD-CCD6-498D-AF62-99AF87852F08.0)

这样无论是严肃场合、轻松场合、书面介绍、1 分钟版本、25 分钟版本，甚至只需要介绍你的工作、你的作品、你的爱好，都可以应对自如，轻松、快速、精准、愉悦。

基于这样的思路，我开始着手建立自己的各类信息数据库：CRM、工作流、运营工作案例、个人 Wiki 等等，在工作、学习和生活中给我带来了巨大的帮助，效果显著。

具体如何构建自己的信息数据库呢？下面我介绍几条经验：

1. 以真实的高频信息需求为基础
----------------

在信息管理的初期，非常容易陷入到一种追求信息的完备性的状态，看到的各种信息都觉得对自己有用，不管什么时候能用得到，现保存下来再说。在这个阶段，更多处于一种盲目的信息输入状态，不那么在意需求，低着头提升信息吞吐量，占有原始资料。

一旦要建立更高效的信息数据库，对信息的加工成本要多出很多，因此需要优先满足更高频的信息需求。

我最早建立的数据库是记录「我最喜爱的 App」，因为我在很多地方做完分享之后，总会有朋友让我推荐生产力工具。一开始我就是直接把常用的工具打字敲给对方，后来问的人实在太多，而且还得补充解释半天。

于是，我创建了「我喜爱的工具清单」，把我在用的工具和在线服务全部整理下来，随手更新。

![](.evernote-content/3270CDD3-244B-47DA-9C9C-1EF4E7F03351/BB3BA42E-7C75-41FB-A1C0-35CCC43558A2.0)

从此以后，再有人问推荐工具，直接甩一个链接过去就好，效率大幅提高。

还有一种常见的数据库是「CRM 客户管理系统」，我虽然不用管理客户，但是经常遇到想要联系一个朋友，死活想不起对方的名字，只能一个一个关键词去尝试，在几千个微信联系人列表里翻，其实一个数据库轻松搞定：

![](.evernote-content/3270CDD3-244B-47DA-9C9C-1EF4E7F03351/560CEB47-FF25-4521-B01C-B2B56F85BB26.0)

熟悉我的人都知道我是一个极端的系统爱好者，给自己搭建了很多系统解决具体的问题，以前的时候一直没有一个统一的地方管理，现在我有了自己的「My Workflow」数据库，里面是我大大小小的工作流，有制定每日计划、准备一场旅行、每周检查回顾等等，这是这些系统不断提升自己的生产力，让我能够像打造产品一样打造自己。

![](.evernote-content/3270CDD3-244B-47DA-9C9C-1EF4E7F03351/6B4BD5A7-E6D2-4EF9-9D84-8F6F75DCBE70.0)

除此之外，我还建立了：

* 段子素材；
* 金句素材；
* 故事素材；
* 要听的音乐清单；
* 要看的电影清单；
* 约饭的朋友清单；
* 等等。

这些数据库更多是 List 或 SOP（即 Standard operating procedure，固定操作流程）类型，除此之外我还建立了「Knowledge Cards」数据库。

![](.evernote-content/3270CDD3-244B-47DA-9C9C-1EF4E7F03351/5E638F42-6EB1-4F31-A83B-82A0B3F34D1E.0)

用来存放「知识卡片」，可能是一个概念、一个学说、一个大人物、一个观点等，他们相对于大纲，更加扁平地保存在数据库中。

你有哪些高频的信息需求，不妨优先建立相关的数据库，着手维护起来吧。

2. 设定更小的信息单位
------------

前面我提到自己在很长时间主要聚焦在「文章」这样的信息单位，觉得收集了几百上千篇高质量文章是一件非常厉害的成就。当我开始建立自己的信息数据库之后，最大的变化就是有一种「看山不是山」的奇妙感受。

我依然会用 DEVONthink、印象笔记、OneNote 这样的工具收集优质的文章，但是在我的眼中，这些文章往往不再是一个整体，而是一个个题目、一条条案例、一个个故事、一个个概念、一个个思维方式、一条条实验数据构成的素材合集。

很多朋友特别喜欢总结书记、演讲、文章的结构，甚至还会绘制精美的思维导图，我则刚好相反，最近越来越习惯将内容直接切片，将一条条知识碎片纳入自己的数据库中。

* 读完一本书，得到 5 个概念和 10 个模版；
* 读完一篇文章，得到 3 个故事和 2 组数据；
* 听完一节课，得到了一句巴菲特关于定投的观点；

获取处理这些作品的目的是：

筛选出对自己最有用的信息，将其保存到自己的信息数据库中。

这里特别提醒你注意，在信息数据库中，保存的信息单位越大，处理的成本越低，信息调取的成本越高。

在我的信息管理 1.0 时代，我在印象笔记里面保存了大概上百篇人人都是产品经理、三节课、极客时间关于运营的文章，但是遇到什么具体的问题，一个关键词往往搜出十几篇甚至几十篇文章，真正的帮助并不大。

在我的信息管理 2.0 时代，我在数据库中新建了「产品运营案例库」，从此之后，我从文章、课程、Newsletter 看到的各种案例，都会保存到这个数据库中，贴上活动的题目、类型和原始链接、截图。

![](.evernote-content/3270CDD3-244B-47DA-9C9C-1EF4E7F03351/116F59F6-1D65-4EAC-96A5-ED1F5E96E5B8.0)

于是，当我在工作中遇到具体任务，在这这个数据库中搜索筛选，往往能得到可以直接参考的案例。

当然你需要结合自己的产品进行优化调整，但是有了这个数据库，不管遇到什么工作心里都不会太慌张。你还能够不断收集补充这个数据库的内容，针对性地对这些案例进行分析学习。

这么调整之后，最明显的一个变化就是信息的基础单位变得非常小，从一篇篇文章，变成几句话、几个字、一张图、一个链接，很可能是清单的一个条目，SOP 的一个步骤。信息的单位虽然小了，信息的实用性却得到大幅提高，真的像 API 接口一样，随时调取，随时使用。

3. 与笔记工具、大纲工具的配合
----------------

最后，我来说说信息数据库和我们在前面介绍的笔记工具、大纲工具的关系，以及如何与这些工具相互配合。

事实上，它们之间的关系确实有点复杂，在流程上和定位上还有重叠，所以我希望在这里给你讲清楚他们之间的差异性。

首先，我们先抛开工具，就像前面提到的，在信息管理系统中，它们最大的不同就是存储的基础单位，在信息处理流程中，往往还会与信息的加工程度相关。

不管工具是什么，Excel、Word、Evernote、Notion、DEVONthink、1Password、幕布、OmniOutliner 等等，他们都是一种信息数据库，对存储的信息类型有自己的偏好，会针对某一类信息提供更好的支持。

我们在收集篇介绍过，这里侧重于知识类信息，对「数据库类型」做个补充说明：在一个高效的信息管理体系中，至少要有这样的几个数据库：

### 数据库 1: 原始信息数据库

这个数据库能够低成本快速保存各种信息的原始数据，不管是什么内容，微博、图片、PDF、微信文章、邮件等等，都可以快速保存，你可以在这里对相关内容进行阅读处理，从中标记提取重要信息。Evernote、有道云笔记、OneNote，以及我在使用的 DEVONthink 都是第一梯队的选手。

### 数据库 2: 清单类数据库

我在前面提到的类似 CRM、金句素材、出差准备 SOP、工具 List 等等，这些信息应该有一个专门存储的数据库，这个数据库应该和上面的原始数据库有所区隔。

有两种选择，如果你是印象笔记或 DEVONthink 的重度用户，可以在印象笔记中利用标签的方式实现。比如，我新建了一条选购礼物的清单笔记，为了和那些「给你推荐 22 个朋友过生日的礼物」的文章参考类信息区隔，我给这篇笔记贴上了 list 标签。

![](.evernote-content/3270CDD3-244B-47DA-9C9C-1EF4E7F03351/7C413F06-BA2F-4B02-A87B-BD6FF97894D3.0)

当然使用 Notion 这样的具有类似 Excel 筛选功能的数据表格的工具，类似 Airtable、Coda 等工具，体验会更好。它们提供更强大的筛选、标签、链接功能，在这类信息的管理上具有非常明显的优势。

### 数据库 3: 大纲数据库

前面提到的大纲其实也是数据库的一种，而且是一种比「清单类数据库」加工程度更高的数据库。清单数据库往往保存比较扁平的信息组块，但是一个新的信息要想进入到大纲中，它不仅需要更大程度的加工，还需要一个「位置」属性，从能纳入到树状知识库中。

在理想情况下，一篇文章先被完整地存储到印象笔记中，你按照第二部分介绍的信息处理的方式进行阅读学习，从里面 Highlight 出的增量信息有 2 个新概念，3 个案例，于是你把概念直接记录到诸如「大纲 - 产品运营」的文件中，3 个案例记录到 Notion 的案例库中，其中一个案例非常精彩，比大纲中支撑这个点的案例还要精妙，于是你把这个案例在 Notion 的页面的链接贴到大纲中，至此完成一次信息从收集到整理的全过程。

某一个主题的内容积累到一定程度，需要将全文保存的情况会越来越少，更多时候只需要将少量有价值的内容摘录到相应的数据库中。不过，在日常的信息处理中，本来觉得就一两条有价值的信息，读了一会发现信息密度巨大，再将全文保存下来，精读处理的情况也常有发生。

最后的话
----

很早以前，团队的一个朋友总是开玩笑地说我折腾工具有点「奇技淫巧」，当时总觉得不服气，实际上在那时候确实陷入到一种「把手段当目的，为了使用工具而使用工具」的误区。

直到有一天，我突然意识到，所有的信息管理最终服务的都是是「使用信息」的时刻，而不是我收集多少信息、整理得多么有序，把分类做得多好，把标签贴得多全面。

某种意义上讲，构建自己的信息数据库已经不再是单纯的信息输入和整理，而是一种初级阶段地信息输出，在需要的时候，这些信息组块，可以被你直接调取使用，放到文章或者演讲中。

理论如此，但是真正要实践，还有很多问题和挑战，不过，我们已经迈出了信息管理 1.0 到 2.0 的第一步，希望对你有启发，如果你对此有更多想法，欢迎留言告诉我。
