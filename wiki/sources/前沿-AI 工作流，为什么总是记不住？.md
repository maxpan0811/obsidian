# 前沿｜AI 工作流，为什么总是记不住？

---

来源：[打开原文](https://mp.weixin.qq.com/s/qjiH1oT0HhqASrLP8Xd2Sw)

![](.evernote-content/0DBFA54A-11FD-486E-A218-DC5298438D23/4C84CD50-AD4A-40BB-B9A2-6C9281C5F32E.png)

前沿｜AI 工作流，为什么总是记不住？
-------------------

很多人用 AI 做研究、写代码、写文章，都会遇到同一个小麻烦：这次好不容易把资料、链接、背景和判断喂给模型，下一次又要从头来一遍。

问题不在于工具不够聪明。ChatGPT 可以给快速答案，Claude Code 和 Codex 可以帮你改代码，NotebookLM 可以消化一批文档。但只要任务变成长期项目，资料就开始散落：Obsidian、Readwise、Notion、Google Drive、GitHub、会议纪要、浏览器收藏夹。模型每次都能临时理解，却很难把这次工作变成下一次的资产。

AI Engineer 这期公开视频里，Decoding AI 的 Paul Iusztin 和 Towards AI 的 Louis-François Bouchard 展示了一套 AI Research OS，也就是面向个人和团队研究工作的 AI 研究操作系统。它没有再造一个聊天入口，而是把“笔记、资料、来源、比较、概念和追问”整理成 agent（智能体）可以反复调用的长期记忆。

如果这个方向成立，AI 工作流的重点会从“上下文窗口还能塞多少”转向另一件事：哪些东西值得留下来，以及怎样让它们在下一次工作里继续发挥作用。

收藏夹不是记忆
-------

Paul 开场给了一个很具体的数字：他的 Obsidian 里有 5000 多条笔记，Readwise 里还有 5000 多条，Notion 和 Google Drive 里也散着不少内容，而且每个月还会新增约 250 个文件。

这听起来像一个很强的 second brain（第二大脑），但真正工作时并不轻松。

当他要写一篇文章、启动一个项目、研究一个代码库或做一个功能时，最想要的是把相关的高信号笔记捞出来。现实往往相反：X 上收藏过的帖子、读过的文章、看过的视频、存过的 GitHub 仓库，都在某个地方，但很难在需要的时候自动浮上来。

很多人的知识管理问题也是这样。资料越多，不一定越聪明；资料没有进入工作流，只是把“以后可能有用”变成了一个更大的仓库。

![](.evernote-content/0DBFA54A-11FD-486E-A218-DC5298438D23/3740CFFD-B389-4B2F-8156-12D8BC15077A.png)

这也是 AI 工具容易制造错觉的地方。

一次对话里，模型看起来什么都懂；对话结束以后，这些理解就被压回历史记录。下次再做相近任务，仍然要重新贴链接、重建背景、解释项目、说明偏好。

Paul 和 Louis-François 提出的核心判断很朴素：上下文窗口不能同时当数据库、文件系统、记忆和推理空间。它可以承载当下任务，但不适合独自承担长期积累。

什么时候该用 ChatGPT，什么时候需要 Research OS
---------------------------------

这期视频里最实用的部分，是先把工具场景分清楚。并不是所有任务都值得搭一套复杂系统。

如果只是查一个事实、问一个概念、写一小段代码，Google、ChatGPT、Claude、Codex 都够用。一次性任务最重要的是快，不值得为它建立长期结构。

如果是一个会反复推进的项目，情况就变了。

比如你在长期写一个技术系列，做一门课程，研究一个新方向，或者持续迭代一套内部 AI 工作流。你希望新资料能留下来，旧判断能被继承，下一次 agent 能读到上一次研究产生的结构。空白会话不适合承接这种工作。

NotebookLM 能处理一批资料，但不够可控，也不够 agent-native。这里的 agent-native，可以理解为从设计上就让 agent 读取、调用、更新和继续工作的能力。传统 RAG（检索增强生成）和向量数据库在生产系统里很强，但对个人和小团队来说又太重，也不够容易手工检查。

AI Research OS 卡在中间：它不追求漂亮 UI，也不急着变成一个 SaaS。Markdown、YAML 和文件夹被用来搭一个工作层，人能看，agent 也能用。

![](.evernote-content/0DBFA54A-11FD-486E-A218-DC5298438D23/60BA88A3-98D1-404B-82DD-3B986915BBFA.png)

这点对企业 AI 负责人也有启发。

很多公司一开始会问：“我们是不是该上知识库？”这个问题还不够具体。更该问的是：哪些知识会继续参与工作？会议纪要、客户反馈、竞品分析、合规规则、代码库说明，如果只是存起来，价值有限；它们必须能被下一次任务调用、比较、更新和追溯。

这套系统怎么从 V1 长到 V3
----------------

Paul 讲了三个版本，值得单独拿出来看。

V1 是一个比较典型的 deep research（深度研究）流程。用户给一个主题，再手动给几个高质量链接。系统会把主题拆成多个问题，让 agent 反复查询、收集链接、排序来源，再抓取高价值资料，最后生成一份静态 Markdown 研究文档。

这个版本能用。Paul 说他们曾靠它快速生成课程内容。但它的问题也明显：每次输出都是一份静态文件，研究做完就结束了。下次有新问题，还是要重新跑一遍。

V2 把输入从“公开网络”扩展到个人 second brain。系统不再只问 Google，也会查 Obsidian、Readwise、NotebookLM、GitHub 等来源。对个人来说，这一步很关键，因为它开始把公开资料和自己的历史判断接起来。

但 V2 仍然没有解决“会不会长期增长”的问题。资料更新后要重新跑，问题变了也要重新跑，token 成本和时间成本都会变高。

V3 才是这套系统真正有意思的地方。它不再只生成一份研究文档，而是形成三层结构：raw files、index.yaml 和 wiki layer。

![](.evernote-content/0DBFA54A-11FD-486E-A218-DC5298438D23/AF78B1FE-15D8-4744-8842-4D23E37F216B.png)

Raw files 是原始资料，尽量不动。Index.yaml 是目录和索引，记录每个来源的链接、摘要、主题、元数据和关联关系。Wiki layer 是可增长的衍生层，里面有概念解释、实体说明、对比笔记、来源摘要、开放问题和项目判断。

这就把一次性研究，变成了一个会继续长的项目记忆。

原始资料不动，Wiki 可以生长
----------------

V3 的工程取舍很值得学。

系统不会让 LLM 每次都重读所有原始资料。它先读 index.yaml，找到相关来源；再读来源摘要和 wiki 里的衍生笔记；只有在需要核对细节时，才回到 raw files。

这看起来只是节省 token，实际更像一种记忆分层。

人类工作也差不多。你不会每次写文章都重新读完所有书，而是先看自己的目录、摘录、主题笔记和对比判断。只有碰到关键事实，才回到原文。

![](.evernote-content/0DBFA54A-11FD-486E-A218-DC5298438D23/8A14885B-FA91-447A-A024-DAF7C7E315FB.png)

这个 wiki 不只在 ingestion（资料导入）时生成。每一次追问、比较、总结和修正，都可以产生新的概念、笔记或对比页。问题被记录下来，新的工作也会继续反哺项目记忆。

这也是它和普通笔记库的区别。

普通笔记库更像存档。Research OS 更像一个项目工作台：资料进来以后，会被整理成 agent 能读的结构；agent 工作以后，又会把新的结构留下来。

这里面还有一个很克制的原则：个人 second brain 本身不被 LLM 直接改写。系统更像从大库里切出一个项目范围，再在项目里生成可操作的 wiki。原始资料保持稳定，项目记忆负责生长。

这对企业场景也很重要。公司知识库不能被 agent 随意重写，但项目空间可以被持续加工。原始记录、可审计索引和衍生判断，最好分层保存。

它给 AI 工作者的提醒
------------

这套 AI Research OS 目前仍然是 builder workflow，也就是更适合开发者和 AI 工程学习者的工作流。它跑在终端里，依赖 Claude Code、Codex、skills、Git 和一些脚本，没有成熟产品那种顺滑体验。

但它给出的方向，比工具本身更重要。

第一，不要把所有东西都塞进上下文窗口。

上下文窗口适合当前推理，不适合当长期数据库。真正值得留下来的东西，应该进入文件系统、索引和项目 wiki。

第二，长期项目需要自己的记忆边界。

不需要把整个个人知识库都塞进模型。更可控的做法，是围绕一个项目，把最相关的来源、摘要、概念、比较和开放问题收进来。边界越清楚，agent 越容易工作。

第三，来源必须能追溯。

AI 工作流最怕“看起来像总结，实际上不知道来自哪里”。Research OS 的 raw files、index.yaml 和 source summaries，本质上是在给 agent 留证据路径。

第四，不要只追求自动化，也要追求可检查。

向量数据库和复杂 RAG 很强，但如果日常工作者看不懂、改不了、查不出来源，就很难成为个人和小团队的稳定工作台。Markdown 和 YAML 不性感，却足够透明。

![](.evernote-content/0DBFA54A-11FD-486E-A218-DC5298438D23/5085B3AE-93B9-4466-AD58-63382563FE69.png)

第五，记忆也要考虑成本。

视频里演示了不同研究强度：轻量模式可能只跑一轮、三次查询；更深的模式可以跑多轮、多查询。不是所有任务都值得重研究。会判断“这次该轻，还是该深”，本身就是 AI 工作流的一部分。

最后
--

过去一年，很多 AI 工具都在抢“当下这一轮对话”。谁回答得快，谁写代码强，谁上下文更长，谁就更容易吸引用户。

高频使用以后，另一个问题会冒出来：我不只想要这一次答案，我还想让这次工作成为下一次的起点。

这就是 Research OS 这类系统的意义。

它不一定会变成大众产品，也不一定适合每个普通用户。但它提醒了一个很重要的方向：AI 工作流的竞争，不只在模型回答质量，也在记忆结构。

能长期积累的团队，会越来越少从零开始。它们的 agent 不只是聪明一点，而是知道更多过去发生过什么、为什么这样判断、哪些来源可信、哪些问题还没解决。

AI 时代的第二大脑，如果只是收藏夹，价值会被高估；如果能变成 agent 可以调用的项目记忆，它才真正开始复利。
