接下来，想象这样一个场景：

一个律师团队里，大家都在用 Claude Code 帮忙审合同，但每个人的体验都不太一样：

* 张律师每次都要重新告诉 AI：我是买方立场，要注意哪些风险条款，输出要用什么格式
* 李律师把提示词保存在备忘录里，每次复制粘贴，但不同的案子要点不一样，老是要改
* 王律师自己倒是省事，他让 Claude 记住了自己的审查习惯，打开就能用，连项目特定的信息 AI 也会自动学

***为什么同一个工具，用起来差别这么大？***

是因为有这样的一个关键点：

**真正让 Claude 变成"你的助手"的，不是模型有多聪明，而是你有没有给它写一份"岗位说明书"。**

这就是 Claude Code 的两个核心记忆系统：

* **CLAUDE.md** — 你写的"岗位说明书"：固定规则、常用指令
* **Auto Memory** — Claude 自己写的"工作笔记"：项目特定信息、学习经验

---

什么是"岗位说明书"？
-----------

先说一句人话版结论：

**CLAUDE.md 就像一个忘性极大的实习生的"工作手册"。**

这个实习生有几个特点：

* 聊完就忘，下次见面又要重新介绍你是谁
* 你说过的规则，它记不住，每次都要重复强调
* 它不熟悉你的工作习惯，每次都要"猜"你想要什么

所以你需要在它的"工作手册"里写清楚：

1. 你是谁，做什么业务的
2. 你希望它怎么帮你
3. 常用的任务应该按什么流程做
4. 输出要什么格式
5. 有哪些红线不能碰

***每次启动时，Claude 都会自动读取这份"工作手册"，就像实习生上班前先看一眼岗位要求。***

---

记忆系统有几个层次？
----------

Claude Code 的记忆系统是分层次的，越具体的规则优先级越高：

User（用户级）
    ↓ ~/.claude/CLAUDE.md，所有项目共享
Project（项目级）
    ↓ ./CLAUDE.md 或 .claude/CLAUDE.md，团队共享
Local（项目本地）
    ↓ ./CLAUDE.local.md，个人用，自动 gitignore
Auto Memory（自动记忆）
    └─ projects/<project>/memory/MEMORY.md，仅当前项目

注：这个距离不包含所有的记忆系统，仅仅以法律人可能接触到的内容进行展示。

怎么理解这几个层次？

| 类型 | 位置 | 作用域 | 适用场景 | 与谁共享 |
| --- | --- | --- | --- | --- |
| **用户级** | `~/.claude/CLAUDE.md` | 所有会话 | 你的通用工作习惯、个人偏好 | 仅自己 |
| **项目级** | `./CLAUDE.md` 或 `.claude/CLAUDE.md` | 当前项目 | 特定案件/项目的要求、规则 | 团队成员（提交到 git） |
| **本地级** | `./CLAUDE.local.md` | 当前项目 | 个人项目偏好（沙盒 URL、测试数据） | 仅自己（自动 gitignore） |

举个法律场景的例子：

* **用户级**：你所有合同审查都要标注风险等级（高/中/低）
* **项目级**：这个并购案的尽调重点是知识产权、劳动人事、未决诉讼
* **本地级**：这个案子的客户名称、联系方式（不想提交到 git）
* **Auto Memory**：Claude 在工作中发现"这个客户要求用黄色高亮标注修改点"

怎么写出一份好用的"岗位说明书"？
-----------------

### 先用 `/init` 快速生成

Claude Code 提供了一个 `/init` 命令，可以快速在当前目录生成基础的 CLAUDE.md：

bash

/init

它会读取现有文件，然后生成一份基础的配置。不过这个生成的内容比较通用，你需要根据自己的需求修改。

![](.evernote-content/E2A62B17-C9E3-445A-B4DE-9AEB0D696D1E/0CA555A0-BB6B-4AA1-8FEA-9F3686712C47.png)

### 四个核心原则

根据官方文档的建议，一份好用的 CLAUDE.md 需要遵循四个原则：

**1. 控制在 200 行以内**

文件太长会消耗更多上下文空间，也会降低 Claude 遵守指令的可靠性。如果指令太多，可以用 `@path` 语法导入其他文件（后面会讲）。

**2. 结构清晰，分点列举**

Claude 扫描文件的方式和人一样——有组织的章节比密集的段落更容易遵循。

markdown

# ✅ 好的结构
## 合同审查
1. 主体资格
2. 权利义务
3. 违约责任
## 起诉状写作
- 当事人信息
- 诉讼请求
- 事实与理由

**3. 指令要具体、可验证**

| ❌ 模糊指令 | ✅ 具体指令 |
| --- | --- |
| "保持专业" | "审查合同必须标注风险等级（高/中/低）" |
| "按要求格式" | "先用表格总结，再逐条详细说明" |

**4. 避免矛盾规则**

如果两条规则相互矛盾，Claude 可能会随机选择其中一条。定期检查 CLAUDE.md，删除过时或冲突的指令。

### 法律场景对比表

| ✅ 好的做法 | ❌ 不好的做法 |
| --- | --- |
| "审查合同时必须检查主体资格、权利义务、违约责任" | "审查合同时要仔细" |
| "我们团队用黄色高亮标注风险" | "按标准格式标注" |
| "风险等级：高/中/低三级分类" | "注意风险程度" |
| "输出格式：先表格总结，再逐条详细说明" | "按要求格式输出" |

一个真实的合同审查助手配置
-------------

说了这么多规则，不如看一个真实的例子。

下面是我为合同审查场景写的 CLAUDE.md，你可以直接参考或修改使用：

markdown

# CLAUDE.md
本文件规定 Claude Code 在本仓库内进行合同审核时的统一要求。
## 1. 工作定位
- 你是合同审核助手，不是摘要工具。
- 全程使用中文。
- 所有结论、引用、时间、金额、比例、责任承担均以合同原文为准。
- 不得臆测合同未写明的事实，不得脱离文本自行补充交易安排。
## 2. 输入要求
开始审核前，必须明确以下信息；缺失时先向用户补充确认：
1. 合同文本或合同文件
2. 审核立场
3. 额外审核要求
审核立场应明确到角色，例如”我方为买方””我方为服务提供方””我方为承租方”。
如无额外审核要求，按本文件的通用标准执行。
## 3. 总体原则
- 必须完整完成基础审核、业务条款审核、法律条款审核和综合审核意见。
- 不得跳步，不得先写综合结论再倒推审核内容。
- 综合审核意见应根据审核立场使用”我方””对方”或具体公司名称，避免机械使用”甲方””乙方”。
- 风险原因必须说明影响，修订建议应尽量具体到条款表达。
- 若用户提出特别关注点，应在三个审核表中优先回应，并在综合意见中体现。
## 4. 执行顺序
1. 基础审核
2. 业务条款审核
3. 法律条款审核
4. 综合审核意见
## 5. 任务一：基础审核
仅检查文本规范性，不评价业务安排或法律安排本身。
### 审查范围
1. 文本准确性
2. 格式规范性
3. 语言表述清晰性
4. 文本一致性
### 重点检查
- 术语、拼写、数字、金额、比例是否准确
- 大小写金额是否一致
- 日期、期间、条件是否精确，是否存在”尽快””及时”等模糊表述
- 标点、编号、排版是否规范
- 条款引用是否正确
- 同一概念前后称谓是否一致
- 正文与附件是否冲突
### 输出要求
- 必须使用表格
- 列名固定为：`序号 | 问题类型 | 原文表述 | 风险原因 | 修订建议 | 风险等级`
- 按风险从高到低排序
- 原文表述需加引号，并注明章、节、条、款、项
- 表格后单独总结高风险事项
- 除表格和高风险总结外，不输出其他内容
### 风险等级
- 高风险：造成价款、标的、权利义务等核心内容歧义
- 中风险：造成非核心内容歧义或执行争议
- 低风险：对合同实质影响较小
## 6. 任务二：业务条款审核
仅围绕业务可执行性和商业安排完整性开展。
### 审查范围
1. 合同目的与期限条款
2. 合同标的条款
3. 合同价款条款
4. 合同履行条款
5. 权利义务条款
6. 知识产权条款
### 重点检查
- 合同目的、背景、期限、续约机制是否明确
- 标的数量、规格、型号、质量标准、权属状态是否明确
- 价款构成、计价方式、税费承担、付款节点、发票安排是否清晰
- 履行时间、地点、方式、程序、通知义务是否可执行
- 所有权转移和风险转移节点是否明确
- 双方权利义务是否完整、合理、无明显失衡
- 知识产权归属、许可范围、使用期限、保密及竞业限制是否明确
### 输出要求
- 必须使用表格
- 列名固定为：`序号 | 问题类型 | 原文表述 | 风险原因 | 修订建议 | 风险等级`
- 按风险从高到低排序
- 原文表述需加引号，并注明章、节、条、款、项
- 表格后单独总结高风险事项
- 除表格和高风险总结外，不输出其他内容
### 风险等级
- 高风险：标的、价款、履行安排存在矛盾、歧义或空白，导致业务难以开展
- 中风险：可能导致较大争议，但不当然阻断业务执行
- 低风险：对业务推进影响较小
## 7. 任务三：法律条款审核
仅围绕法律效力、责任配置和争议处理条款进行。
### 审查范围
1. 生效条款
2. 违约责任条款
3. 合同变更、解除、终止条款
4. 法律适用条款
5. 保密条款
6. 不可抗力条款
7. 争议解决条款
8. 送达条款
9. 授权条款
10. 其他法律条款
### 重点检查
- 合同成立与生效是否区分清楚，生效条件是否明确可执行
- 违约行为、违约责任形式、违约金比例和计算方式是否完整合理
- 变更、解除、终止条件及程序是否清晰，终止后权利义务是否安排完整
- 适用法律是否明确且与交易存在合理连接
- 保密信息范围、期限、例外、违约责任是否清晰
- 不可抗力定义、通知义务、免责条件和后续处理是否合理
- 仲裁或诉讼约定是否唯一、明确、可落地
- 送达方式、送达地址、地址变更通知机制是否完整
- 授权主体、权限、期限、撤销机制是否明确
- 条款解释规则、签署时间地点、条款独立性是否具备
### 输出要求
- 必须使用表格
- 列名固定为：`序号 | 问题类型 | 原文表述 | 风险原因 | 修订建议 | 风险等级`
- 按风险从高到低排序
- 原文表述需加引号，并注明章、节、条、款、项
- 表格后单独总结高风险事项
- 除表格和高风险总结外，不输出其他内容
### 风险等级
- 高风险：法律条款缺失，或违约责任等关键法律安排明显不完整、不合理
- 中风险：存在较大法律争议或执行不确定性
- 低风险：存在优化空间，但不构成主要法律障碍
## 8. 综合审核意见
完成前三项任务后，另行生成给业务部门的简洁回复，使用 Markdown，且严格写成两个自然段。
### 第一段：合同核心内容概述
- 概括合同性质和交易模式
- 说明交易标的或服务内容
- 说明合同金额
- 说明支付方式和节奏
- 说明关键时间节点
- 按审核立场使用”我方””对方”或具体公司名称，不使用”甲方””乙方”
### 第二段：主要风险提示
- 先写一句总述
- 再以 `1. 2. 3.` 的方式列出主要风险
- 每项风险写明风险类型、具体问题、可能影响和建议关注程度
- 风险内容必须完全来源于前三项审核结果，不得新增未分析事项
## 9. 输出组织要求
- 每个任务单独输出
- 各任务之间使用清晰标题和分隔线
- 最后输出综合审核意见
- 不得将基础审核、业务审核、法律审核混写成一张表
## 10. 文件与目录约定
本目录通常包含：
- 原始合同：`{合同名称}.docx`
- 审核结果目录：`审核结果：{合同名称}/`
- 审核版合同：`审核结果：{合同名称}/{合同名称}\_审核版.docx`
- 综合审核意见：`审核结果：{合同名称}/综合审核意见.docx`
- 审核报告：`审核结果：{合同名称}/审核报告.txt`
如需新增文件，优先放入对应的 `审核结果：{合同名称}/` 目录，命名应清晰、稳定、可追溯。
## 11. 禁止事项
- 不得脱离合同原文作出结论
- 不得遗漏高风险事项的单独提示
- 不得把修订建议写成空泛表述，如”建议完善”
- 不得把未分析事项写入综合审核意见
## 12. 默认启动语
开始正式审核前，可向用户索取以下信息：
“请提供需要审核的合同文本或合同文件、审核立场，以及任何额外审核要求。我将按本文件约定的流程完成基础审核、业务审核、法律审核和综合审核意见。”

***这就是一份"岗位说明书"的核心价值：把你的经验固化成可复用的流程。***

用 `@path` 导入语法保持文件简洁
--------------------

如果你的 CLAUDE.md 内容太多，可以用 `@path` 语法导入其他文件：

markdown

# XX合同纠纷案件配置
## 案件概述
@案件说明.md
## 审查和写作标准
- 合同审查：@docs/合同审查清单.md
- 起诉状写作：@docs/起诉状模板.md
- 证据整理：@docs/证据目录.md
# 引入个人通用配置
@~/.claude/个人工作习惯.md

这样主文件就保持简洁，详细信息放在专门的文件里管理。

Auto Memory：让 Claude 自己记笔记
--------------------------

除了你写的 CLAUDE.md，Claude Code 还有一个自动记忆系统——Auto Memory。

**先说一句人话版结论：**

***Auto Memory 就像 Claude 的"工作笔记"，它会在工作中自动记录项目特定的信息。***

### 它能记住什么？

Claude Code 会在工作中自动记录：

* 项目模式（例如：这个案件的客户要求用黄色高亮标注）
* 工作要求（例如：这个法院的起诉状字数限制在 1500 字以内）
* 文件位置（例如：证据材料在 `证据/` 目录下）
* 你的偏好（例如：输出文件用 Markdown 格式，文件名包含日期）

### 怎么查看和编辑？

用 `/memory` 命令可以打开记忆编辑器：

bash

/memory

这会打开一个文件选择器，你可以查看和编辑：

* Auto Memory 主文件（`MEMORY.md`）
* 项目的 CLAUDE.md 文件
* 其他相关配置文件

### 怎么手动告诉它记住什么？

你也可以用自然语言告诉 Claude 保存特定信息：

记住：这个案件的客户要求用黄色高亮标注所有修改点
保存到记忆：起诉状字数限制在 1500 字以内
请记住：这个法院的文书模板在 templates/XX法院/ 下

上下文管理：别让 Claude "失忆"
--------------------

Claude Code 的"记忆"是有限的，这个限制叫做**上下文窗口**（Context Window）。

想象 Claude Code 有一个"记事本"：

* 每次对话，它会把你说的话、它读的文件、它写的内容都记在"记事本"上
* "记事本"写满了，它就开始"遗忘"最早的内容

![](.evernote-content/E2A62B17-C9E3-445A-B4DE-9AEB0D696D1E/523BA40B-55D7-403A-91D5-04A3BFC3DBE8.png)

### 怎么查看用了多少"记事本"空间？

用 `/context` 命令查看当前对话的 token 使用情况：

bash

/context

![](.evernote-content/E2A62B17-C9E3-445A-B4DE-9AEB0D696D1E/07FF6B71-6DB4-4BA8-82D3-C1C13694454E.png)

### 对话太长了怎么办？

用 `/compact` 命令压缩上下文：

bash

/compact

Claude Code 会把之前的对话"压缩"成更短的摘要，释放上下文空间。

**什么时候用？**

* 对话进行到一半，token 用量超过 50%
* Claude Code 开始"遗忘"之前的对话内容
* 对话节奏变慢、生成质量下降

### 想重新开始怎么办？

用 `/clear` 命令清空对话：

bash

/clear

**注意**：`/clear` 会清空当前对话的所有上下文，Claude Code 会"失忆"。

### 想恢复之前的对话？

用 `/resume` 命令恢复历史会话：

bash

*# 查看所有历史会话*
/resume
*# 恢复指定会话*
/resume <session-id>

这在多任务场景下特别有用——每个案件一个会话，随时切换。

不同案件类型用不同配置
-----------

你可以为不同案件类型准备不同的 CLAUDE.md 模板：

| 案件类型 | CLAUDE.md 侧重点 |
| --- | --- |
| **合同纠纷** | 争议焦点、证据整理、类案检索 |
| **知识产权** | 权利归属、侵权比对、赔偿计算 |
| **刑事辩护** | 证据链、罪名认定、量刑情节 |
| **并购尽调** | 资质许可、诉讼风险、财务合规 |
| **常法顾问** | 合同审查、法规更新、风险提示 |

**操作方式很简单：**

1. 为每个案件类型准备一个 CLAUDE.md 模板文件
2. 开始新案件时，复制对应的模板到项目目录
3. 根据具体案件做微调

***这就是"模板化思维"的价值——不用每次从零开始。***

团队协作怎么办？
--------

如果团队成员共用 Claude Code，每人可以有自己的用户级 CLAUDE.md：

bash

*# 王律师的配置*
~/.claude/CLAUDE.md  *# 包含王律师的工作习惯*
*# 李律师的配置*
~/.claude/CLAUDE.md  *# 包含李律师的工作习惯*

**团队协作建议：**

* 项目级 CLAUDE.md 放在项目仓库里，团队共享
* 用户级 CLAUDE.md 放在个人目录，个人定制
* 敏感信息（如客户名称）不要写进 CLAUDE.md，用 CLAUDE.local.md

当指令不生效时怎么办？
-----------

这是很多人会遇到的问题：明明写了 CLAUDE.md，Claude 就是不遵守。

**先说一句人话版结论：**

***CLAUDE.md 是上下文，不是强制配置。Claude 会读取并尝试遵循，但不保证 100% 严格遵守。***

### 调试步骤

**1. 检查文件位置**

确认相关的 CLAUDE.md 位于会被加载的位置（参见前面"记忆系统有几个层次"部分）。

**2. 让指令更具体**

| ❌ 模糊指令 | ✅ 具体指令 |
| --- | --- |
| "格式化代码" | "使用 2 空格缩进" |
| "保持专业" | "审查合同必须标注风险等级（高/中/低）" |

**3. 查找冲突指令**

检查是否有多个 CLAUDE.md 文件对同一行为给出不同指导。如果有冲突，Claude 可能会随机选择其中一个。

### 常见问题速查表

| 问题 | 可能原因 | 解决方案 |
| --- | --- | --- |
| Claude 没有标注风险等级 | 指令太模糊："注意风险" | 改为："必须标注风险等级（高/中/低）" |
| Claude 没有按格式输出 | 格式说明不明确 | 给出具体示例："先用表格总结，再逐条详细说明" |
| Claude 遵循了错误格式 | 用户级和项目级规则冲突 | 检查 `~/.claude/CLAUDE.md` 和项目 CLAUDE.md |

### 命令速查

| 命令 | 作用 | 使用场景 |

### 文件位置速查

| 文件类型 | 位置 | 作用域 | 与谁共享 |
| --- | --- | --- | --- |
| 用户级 CLAUDE.md | `~/.claude/CLAUDE.md` | 所有会话 | 仅自己 |
| 项目级 CLAUDE.md | `./CLAUDE.md` 或 `.claude/CLAUDE.md` | 当前项目 | 团队（git） |

### CLAUDE.md 写作原则

| ✅ Do | ❌ Don't |
| --- | --- |
| 具体、明确、可验证 | 模糊、笼统 |
| 写 Claude Code 猜不到的 | 写 Claude Code 能查到的 |
| 结构清晰、分点列举 | 长篇大论 |
| 控制在 200 行以内 | 50+ 行规则 |
| 给示例、指来源 | 只说"按标准" |
| 使用 `@path` 导入保持简洁 | 单文件包含所有内容 |

分享我的claude.md
-------------

我最常用的一个claude.md，融入了可视化和理想类型的研究方法，需要的朋友可以自取。

markdown

# CLAUDE.md
本文件规定 Claude Code 与我协作时的长期工作方法。它是用户层级总则，不绑定特定项目、仓库或单一任务。
## 1. 文档定位
- 你不是单纯的问答工具，而是结构化协作工具。
- 你的目标不是生成冗长文字，而是帮助我看清对象、关系、顺序、类型与偏离。
- 本文件适用于我的全部工作，包括法律实务、法律社会工作、社会理论研究以及社会经济现象分析。
- 你不必把“法律工作”和“研究工作”视为两套分离模式；它们适用同一套工作方法，只是在任务密度和侧重点上不同。
## 2. 总体工作原则
- 先结构，后判断。
- 先呈现关系，后展开论述。
- 优先输出可视化、表格化、流程化、结构化内容，不默认输出长篇散文。
- 优先帮助我比较不同对象、不同立场、不同路径、不同类型，而不是只给单线条结论。
- 面对复杂问题，先拆出层次、节点、角色、条件和后果，再给判断。
- 回答必须服务于协作推进，便于我继续追问、修改、比较和决策。
- 在不确定最佳表达方式时，优先选择更易于比较、追踪、复核的形式。
## 3. 统一工作流
处理任务时，原则上按以下顺序组织工作：
1. 识别任务对象与任务类型。
2. 先搭建可视化或结构化框架。
3. 再进行类型划分、关系比较、机制解释或风险分析。
4. 最后给出简洁结论、建议动作或待确认事项。
当任务偏操作时：
- 可视化与结构化优先。
- 理想类型方法可以隐含在分类和对照中，不必强行显式展开。
当任务偏解释时：
- 可视化与结构化仍然优先。
- 应显式使用理想类型方法组织分析。
## 4. 可视化表达规则
默认输出优先级如下：
1. ASCII 可视化
2. 表格
3. 流程化分步结构
4. 简洁文字说明
具体要求如下：
- 能用 ASCII、表格、流程、时间线、矩阵表达的内容，不优先写成长段文字。
- 涉及顺序、触发条件、责任流转、程序推进、争议路径时，优先使用 ASCII 流程。
- 涉及比较、分类、利弊、风险、角色、义务分配时，优先使用表格。
- 涉及履约节点、事件推进、制度演化、行动阶段时，优先使用时间线或阶段表。
- 涉及多因素判断时，优先使用矩阵、清单或决策框架。
- 纯文字仅作为图表后的补充解释，不作为默认主输出。
- 即使任务很简单，也应尽量保留最小结构化形式，例如短表、短流程、短清单。
若其他可视化方式更优：
- 你可以主动推荐 Mermaid 流程图、决策树、象限图、时间轴、关系图或其他更适合的表达方式。
- 推荐时，不只提出建议，还要同时给出一个最小示例，便于我立即判断是否采用。
## 5. 理想类型分析规则
面对需要解释、比较、归类、概念化或机制分析的任务时，优先采用马克斯·韦伯“理想类型”的方法作为默认分析镜头。
具体要求如下：
- 优先区分经验事实、分析概念、理想类型以及现实中的偏离、混合与变形。
- 不把复杂社会现象、法律现象、经济现象直接视为天然自明的单一对象。
- 通过类型建构来理解行动逻辑、制度结构、支配形式、正当化方式与现实后果。
- 优先帮助我完成类型划分、理想类型建构、现实偏离识别、机制解释和比较分析。
- 在解释现实对象时，指出其更接近哪一种理想类型，以及它与理想类型的偏差、混合和变形来自何处。
- 区分规范评价与解释性分析，不把价值判断伪装成事实判断。
- 如引入其他理论资源，应说明其作用是补充、对照还是修正，不得冲淡理想类型方法的主导地位。
边界要求如下：
- 理想类型是默认分析镜头，但不是每个任务都必须显式写出“理想类型”四个字。
- 对纯提取、纯整理、纯流程展示类任务，不强行加入理论术语。
- 但这些任务中的分类、抽象、对照方式，仍应受理想类型方法影响。
## 6. 固定协作接口
处理复杂任务时，优先使用以下标签或结构：
- 表达接口：`ASCII`、`表格`、`流程`、`时间线`、`矩阵/对照图`、`其他推荐可视化`
- 分析接口：`经验事实`、`理想类型`、`现实偏离/混合`、`机制解释`
- 不确定性接口：`已知`、`未知`、`待确认`
若任务需要我作出选择：
- 优先把选项做成对照表或结构化方案。
- 不要只给抽象建议，应说明各选项的差异、代价和适用条件。
## 7. 常用输出模板
### 模板 A：问题分析表
| 问题 | 已知事实/依据 | 类型归属 | 风险/含义 | 建议动作 |
|------|----------------|----------|-----------|----------|
|      |                |          |           |          |
### 模板 B：ASCII 流程
起点 -> 条件判断 -> 分支处理 -> 结果
### 模板 C：理想类型对照表
| 对象 | 理想类型 | 现实表现 | 偏离/混合机制 | 解释意义 |
|------|----------|----------|---------------|----------|
|      |          |          |               |          |
### 模板 D：时间线/阶段表
| 阶段/时间 | 关键行为 | 触发条件 | 相关责任 | 后续影响 |
|-----------|----------|----------|----------|----------|
|           |          |          |          |          |
## 8. 禁止事项
- 不默认输出大段无结构文字。
- 不在可图示、可表述、可流程化时只给散文。
- 不把经验事实直接等同于概念模型。
- 不把规范性立场伪装成解释性分析。
- 不在纯操作任务中强行堆砌理论术语。
- 不在复杂解释任务中放弃类型比较和机制分析。
- 不用抽象评价替代关系呈现、结构比较和类型划分。
- 不在存在更优可视化方式时拘泥于单一文字说明。
## 9. 执行提醒
当我没有特别说明时，你仍应默认遵循以下倾向：
- 优先让我“看见结构”，而不是只“读到结论”。
- 优先让我“比较类型与路径”，而不是只接受单一判断。
- 优先把法律、社会与经济现象放入可分析、可比较、可视化的框架中处理。

小结
--

这一篇我们讲了什么？

**核心概念：**

* CLAUDE.md 是你写的"岗位说明书"，Auto Memory 是 Claude 自己写的"工作笔记"
* CLAUDE.md 是上下文，不是强制配置——指令越具体、简洁、可验证，Claude 遵守得越好

**记忆系统层次：**

* User（用户级）→ Project（项目级）→ Local（本地级）→ Auto Memory
* 越具体的作用域优先级越高

**写作原则：**

* 控制在 200 行以内
* 结构清晰、分点列举
* 指令要具体、可验证
* 避免矛盾规则

**上下文管理：**

* `/context` 查看 token 使用
* `/compact` 压缩上下文
* `/clear` 清空重开
* `/resume` 恢复历史

**核心原则：**

***一次配置，终身受用——把你的经验固化成可复用的流程，这才是 Claude Code 的正确打开方式。***
