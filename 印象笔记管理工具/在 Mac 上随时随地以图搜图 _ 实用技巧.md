图片搜索，是一种很酷的搜索方式，不用啰哩啰唆的文字描述，一张图片传上去，接下来只需等着返回搜索结果。

Google 的搜图功能

从浏览器、Windows、Android 到 iOS，多数平台上都不乏搜图小工具；就算在相对封闭的 iOS 上，Hum 也在 Workflow 时代就制作了 [搜图动作](https://www.icloud.com/shortcuts/05e3cfd6bfc943aabc139497b9d8181e "获取该捷径")。但是到了 macOS，搜图工具的日子一直很难过，最后有两件事促使我自己制作工具。

第一件就是 Google 网页版本移除了查看相似图片的功能，让 Google 图片搜索的能力大打折扣；第二件发生在近期，硕果仅存的几个 Safari 搜图插件也在升级到 macOS Mojave 后悉数归天。

![](.evernote-content/E3D031BD-9C97-4398-9785-8B2634FFBC64/FD549F89-71DF-44A4-8180-EAA9A82D3884.png)

Google 不再显示相似图片，只剩用处有限的「相关图片」

如果你不是 Chrome 或 Firefox 的用户，那基本只能依靠蹩脚的网页：翻网页时看到一张有趣的图片，就不得不先下载再上传进行搜索。若是在浏览器以外的地方遇到想搜的图片，Chrome 浏览器也帮不了你。

本文就来解决这些不爽，在 Mac 上打造一套呼之即来的搜图操作。

两种搜图方式
------

在 Mac 上搜图，不外乎两种情况：

1. 已经下载好了图片，需要进行搜索，属于**选图搜索**。
2. 在网页、电子书或别的地方看到一张图片，挺感兴趣，随手一搜，谓之**截图搜图**。

从实现的难度上来说，第一种更简单；而第二种，就要利用一些取巧的办法来自动清理截图文件。本文将分开讨论这两种情况，并各给出一套方案。

注：为求雨露均沾，本文搜索方法基于比较大众的图片搜索引擎（Google、百度之类）。它们原理相同，所以不分开单独介绍。文章以 Google 为例。

### 原理简析

如果你已经尝试过自定义搜索引擎，那你一定对于一段搜索 URL 的构成不陌生，它们不过是一段 URL 前缀和关键词的组合。比如在少数派搜 `minja`，搜索结果页面的链接就是两部分组成的。

![](.evernote-content/E3D031BD-9C97-4398-9785-8B2634FFBC64/BFE01080-CD47-48BA-B72C-76988EE53A8A.png)

搜索 URL 一般由前缀和关键词组成

**不管之前的步骤多么曲折，最后一步都是把和上图形式类似的 URL 交给浏览器打开。**那么，Google 图片搜索的 URL 又是怎样的？没玩过搜索引擎自定义的读者也不用担心，我们直接打开 Google 的官方文档，就可以清晰看到 URL 的组成：

1. 图片搜索的前缀
2. 图片的网址

![](.evernote-content/E3D031BD-9C97-4398-9785-8B2634FFBC64/7F09000F-84B9-4F66-BDF4-B225EEDB7B63.png)

Google 图片搜索的 URL 由前缀和图片网址组成

关键就在于，我们的图片素材往往是本地的，如何把它们上传到网上？有一个不花钱的方案，就是利用**图床服务**。其实，想到图床这一层后，选图搜索的步骤也就明晰了，直接来看一眼最后的脚本，顺着捋下来就是：

1. 红色部分：上传图片到图床，并获取图片地址
2. 蓝色部分：组合搜索链接
3. 绿色部分：在浏览器打开链接
4. 橙色部分（非核心作用）：一套批处理模板

![](.evernote-content/E3D031BD-9C97-4398-9785-8B2634FFBC64/78FB6368-6094-453C-97DD-AED88DF6164B.png)

选图搜图的代码

首先解决图床问题。我推荐的图床服务是 [SM.MS](https://sm.ms/)，免费不限流量，最重要的是**支持通过命令行进行交互**，方便我们打包成轻巧的自动化小工具。

![](.evernote-content/E3D031BD-9C97-4398-9785-8B2634FFBC64/57E7A036-9699-485D-9427-8BC8E6F1CC28.png)

SM.MS 的 API

SM.MS 的 API（Application Programming Interface, 应用程序编程接口）其实没有什么思路上的难点，基本就像查字典，愿意多翻几分钟的人自然就懂了，不想花力气的怎么都不可能学会。建议对 API 比较生疏的读者看看 Hum 的 Workflow 系列文章。捷径里有个高级模块「从 URL 获取内容」，可以用于上传图片，macOS 里有个同样功能的命令行工具 ——`curl`。写好的上传命令是：

```
curl -F smfile=@"本地图片地址" https://sm.ms/api/upload
```

先不急着做搜图小工具，我们在终端里试试上传命令的运行结果。

![](.evernote-content/E3D031BD-9C97-4398-9785-8B2634FFBC64/39AA9D73-F590-4743-ADA4-DE39BBF5B08E.png)

上传成功后的反馈结果

看到 `success` 了吗？上传成功。

不过，看到上传结果的兴奋劲儿还没过，我们就遇到一个新难题：**我们只需要一条图片地址链接，但是命令行大大方方给了一堆参数，怎样只提取出所需的部分**？提取文本可是命令行工具 `grep` 和 `sed` 的强项，不过里面细节太多，这里不便展开，大家直接拿我写好的命令去用就是。和刚才的上传命令组合在一起，生成链接的脚本就是：

```
curl -F smfile=@"$@" https://sm.ms/api/upload | grep -o -E 'https:\\/\\/i\.loli\.net\\.*\.png|https:\\/\\/i\.loli\.net\\.*\.jpg|https:\\/\\/i\.loli\.net\\.*\.jpeg' | sed 's/\\//g'
```

**知识点：如何在命令行里使用变量**

稍后我们要把链接和 Google 搜图的 URL 前缀拼接起来，这就涉及到了链接的**取用**问题，是时候搬出命令行里的**变量**概念了。变量就像带有名字的篮子，里面可以自由放入各种数据（当然包括刚才提取出来的图片链接），一会儿把需要的「篮子」拿过来，就能取用里面的数据。

在命令行里设置变量很轻松，一句 `变量名=数据` 就搞定。如果想把一条或一串命令的结果设置为变量，也只需要在命令前后套上括号、加上 `$` 符号，像这样：

```
var=$(curl -F smfile=@"$@" https://sm.ms/api/upload | grep -o -E ’https:\/\/i.loli.net\..png|https:\/\/i.loli.net\..jpg|https:\/\/i.loli.net\..jpeg’ | sed ‘s/\//g’)
```

`var` 里面「装着」的就是图片链接了。

链接有了，URL 前缀也知道了，剩下的步骤就势如破竹。继续利用刚才的变量知识，我们可以拼装出最终的图片搜索 URL。

```
var2=https://images.google.com/searchbyimage?image_url=$var
```

最后，在浏览器里打开上面的链接：

```
open -a Safari $var2
```

保存整段脚本，用你喜欢的自动化工具运行一下，我们就做出了简单的以图搜图小工具。比较好学的读者或许还牵挂着橙色的两行代码，它们是一组用于批处理的模板，也就是说，我们的搜图小工具其实是可以**一次性搜多张图片的**，不过估计用的场景不多，所以不再细说。

截图搜图
----

上一节的搜图动作，基本还是延续 iOS 上的思路：选图、上传、搜索。但我们用的毕竟是 macOS，理应拥有更大的自由度：

1. **自由选区**：不想上传整张图片，只想搜部分区域。比如看到别人晒工作环境，觉得他壁纸不错，此时就只需要截取显示器那部分进行搜索。
2. **实施搜索**：不用手动保存搜索素材，在当前页面就能进行搜索。
3. **自动清理**：搜完图片后，希望自动删除原图文件，达到「无残留」的体验。

### 原理简析

想要随时截图、搜完自动清理截图文件，可以依靠 macOS 的一大特色功能：**屏幕截图**。

截图时可以调整选区、自动保存图片，而且 macOS 下还有命令行版本的截图工具，给我们的深入定制提供了基础。顺着截屏的思路，搜图流程也有了变化：

1. 青色部分：截图保存到本地
2. 红色部分：上传图片、获取地址
3. 蓝色部分：组合搜索链接
4. 绿色部分：在浏览器打开链接
5. 紫色部分：删除截图文件

![](.evernote-content/E3D031BD-9C97-4398-9785-8B2634FFBC64/6DEE9E7E-02FC-44FF-8A0A-53C487B973D8.png)

截图搜图的代码

中间 3 步和上一节基本一致，这回也用不上前后的批处理模板（不然就和截图搜索的轻快相违背了），我们只需调整一下开头和结尾的步骤。

**知识点：macOS 的截图命令**

macOS 下有原生的截图快捷键，但它生成的文件名带有空格、还会根据当前时间而改变，在脚本里调用起来不方便，所以我们直接使用截图命令，**把截图临时放到指定的位置**：

```
screencapture -i 自定义的截图保存位置
```

其中的 `-i` 表示「自由模式」，全靠它，我们才能自己临时设置截图的保存位置，而且不会影响以后正常的截图。有兴趣的读者还可以在终端中运行一下 `screencapture help` 找找更多的命令。

本文所提供小工具中的截图位置默认在用户主文件夹（`~/图片名`）下，运行时一般看不到截图的出现和消失，整个过程很干净。接下来的搜索过程和上一节没啥区别，我们直接跳到最后一步：**清理截图文件**。

```
rm ~/googleimagesearch.png
```

在这个搜图工具里，`rm` 命令是比较合适的文件清除方式，但平时不要随意使用 `rm` 命令，它会直接把文件清掉而不经过废纸篓，操作不稳容易丢东西。

现在把脚本整合到自动化工具里运行一遍，理想的截图搜索就完成了。

小结
--

以图搜图并不是一个新奇的功能。

制作这套动作前，恰逢 Hum 的[捷径教程](https://sspai.com/series/68)更新到了套壳截图章节，我在帮他找 iPad Pro 的壳图素材。由于全面屏 iPad 刚发布不久，网上的壳图很难找，用了几轮关键词后都没结果，于是我换了个思路，随便找了张 iPad Pro 的截图，把中间部分涂黑，自己造了一张比较粗糙的「壳图」，然后用 Google 图片搜了一下 —— 一下子就有结果了，而且还是高清晰度的 Sketch 素材。

![](.evernote-content/E3D031BD-9C97-4398-9785-8B2634FFBC64/826095A8-4D5F-41BD-BEBA-5837571E7BBD.png)

通过以图搜图找到了壳图素材

这只是以图搜图的一次小小应用。想不好关键词时，用图片来搜索往往会有意想不到的答案。

具体到工具上，macOS 平台就比较匮乏，基本只能在几款浏览器（还不包括原生的 Safari）里搜；好在搜图原理并不复杂，摸清了搜索 URL 的构成后，再打通中间的几次数据传递，我们就自制了两种搜图小工具。不管你喜欢选图搜索还是偏好截图搜索，在 macOS 的搜图方面，我们都已经有了更称手的选择。
