| 摘要  本文是七层约束系列的第 2 篇，聚焦输入端三层——Hook、Skill 和 CLAUDE.md。这三层的核心逻辑是 Harness Engineering：不改变模型本身，而是在模型接收任务之前设定行为边界。文中完整拆解了 Hook 事件注入的配置方式与源码、Skill 的安装机制与三类使用场景、CLAUDE.md 三级加载体系及每级的写作原则。读完可直接复制配置落地。项目 GitHub 地址请见文章结尾。  关联阅读，7层约束系列第一篇：[让 Claude Code 成为稳定协作者：7 层约束系统设计与实践（已开源）](https://mp.weixin.qq.com/s?__biz=MzIwMDcwMjYwNg==&mid=2247484264&idx=1&sn=2bef4276e01b3f6c60972d5b5748c789&scene=21#wechat_redirect)  用 Claude Code 做过长期项目的人，大概都观察到过这个现象：开始阶段它配合得很好，几周或几个月之后，行为质量在悄悄下滑。不是突然崩溃，是缓慢退化——验证步骤开始被省略，回答越来越模糊，承诺了要做的事开始被跳过。  更让人费解的是，这种退化很难靠「加强提示」来解决。你叮嘱它「记得验证」，它说好的，下次又忘了。你在对话里反复强调规则，效果随着对话轮次增加而衰减。  原因是结构性的：用自然语言指令约束一个概率生成系统，等于把防线建在这个系统内部。指令本身也在概率处理的范围里，它和其他所有内容竞争注意力权重，在长对话中必然被稀释。  有效的约束必须在模型的概率处理之外。工程上有个对应的思路：Harness Engineering——不改变模型本身，而是设计模型的运行环境，在输入端就把行为边界设定好，在模型开始处理任务之前约束已经就位。  Claude Code 提供了三个输入端机制：Hook（事件注入）、Skill（流程锁定）、CLAUDE.md（分级规则体系）。这篇文章逐层拆解这三个机制的设计逻辑、完整配置方式和实际使用中的注意事项。    文章结构   1. CLAUDE.md 层：三级分层规则体系  1）解决什么问题  2）三层叠加结构与写法示例  3）加载机制  4）什么该写，什么不该写  5）冲突处理规则 4. 三层协同逻辑   一、Hook 层：上下文劫持这层解决什么问题  CLAUDE.md 里的规则在对话开始时加载，但它是静态的。随着对话轮次增加，早期写入的规则在注意力分配上被后续内容挤占，执行率逐渐下降。这不是模型「故意」忘记，是概率分布在长对话中自然的权重偏移。  Hook 的设计思路完全不同：不是让模型「记住」一条规则，而是让它每次处理用户消息之前都被迫重新读到这条规则。Claude Code 的事件系统支持在特定事件发生时执行脚本，将脚本输出注入模型的输入上下文。注入发生在模型开始生成回复之前，是系统级的强制介入，不依赖模型的「记忆力」。  Claude Code Hook 支持哪些事件  Claude Code 目前（截至 2026 年 5 月）支持 31 个 hook 事件，覆盖会话生命周期、用户输入、工具调用、子代理、任务管理、文件环境、上下文压缩等 9 大类。这里列出与约束工程最相关的核心事件：  用户输入1 个  UserPromptSubmit  用户发消息后、模型处理前  工具调用3 个  PreToolUse  任意工具调用执行前（含 Write/Edit/Bash）  PostToolUse  工具调用成功返回后  PostToolUseFailure  工具调用失败后  模型完成1 个  Stop  模型完成本轮回复时  会话生命周期1 个  SessionStart  会话启动或恢复时  上下文压缩1 个  PreCompact / PostCompact  上下文压缩前 / 后  这套约束系统实际配置的是 UserPromptSubmit 和 Stop 两个事件。前者在每条用户消息处理前强制注入纪律提醒和工作目录状态，后者在会话结束时触发归档脚本。  PreToolUse 和 PostToolUse 是可扩展的精细控制点——可以在文件写入前拦截、在工具返回后注入校验结果。「写入前自动触发 audit」在 hook 里做会让脚本过重，目前交给 Scripts 层处理（第四篇会讲）。  完整的 settings.json 配置  Hook 配置写在 ~/.claude/settings.json（全局生效）或 <项目目录>/.claude/settings.json（仅对当前项目生效）。两个位置可以同时存在，Claude Code 会合并加载。  当前使用的完整 hooks 配置：  {  "hooks":{  "UserPromptSubmit":[  {  "hooks":[  {  "type":"command",  "command":"node ~/.claude/scripts/hook-cwd-check.js",  "shell":"bash"  }  ]  },  {  "hooks":[  {  "type":"command",  "command":"node ~/.claude/scripts/hook-discipline.js",  "shell":"bash"  }  ]  }  ],  "Stop":[  {  "hooks":[  {  "type":"command",  "command":"python '~/.claude/scripts/export\_chat.py'",  "shell":"bash"  }  ]  }  ]  }  }    几点说明：   * UserPromptSubmit 下可以配置多个 hook 对象，按数组顺序依次执行，互相独立 * 每个 hook 对象的外层 hooks 数组是 Claude Code 的固定结构（不是多余的嵌套，这是设计如此） * "shell": "bash" 指定执行环境；Windows 用户需要有 WSL 或 Git Bash，或把命令改成 Windows 原生写法 * 路径用绝对路径比相对路径稳定，规避 cwd 变化导致脚本找不到的问题；也可以用 ~ 代替 home 目录   Hook 脚本的输出格式  Hook 脚本通过 stdout 向 Claude Code 传递数据。只有符合特定 JSON 格式的输出才会被 Claude Code 处理，其他 stdout 输出和全部 stderr 输出会被忽略（不会报错，只是无效）。  UserPromptSubmit 事件的注入格式：  {  "hookSpecificOutput":{  "hookEventName":"UserPromptSubmit",  "additionalContext":"这里是要注入到模型上下文的内容"  }  }    additionalContext 的内容会出现在模型的系统提示层，在用户消息之前被处理。无论用户发什么，模型在处理它之前都会先读到这段内容。这就是「上下文劫持」的含义——不是用户消息本身包含这些规则，是系统层面在用户消息旁边强制附加了约束。  实际运行的两个 Hook 脚本  脚本一：hook-discipline.js  每次用户发消息，把事实核实纪律框注入上下文。 additionalContext: disciplineReminder  }  };    process.stdout.write(JSON.stringify(output));    几个关键细节：   * 用 process.stdout.write 而不是 console.log：console.log 会在末尾追加换行符，某些情况下会导致 JSON 解析失败；stdout.write 输出原始字符串，更安全 * 带边框的视觉格式（╔╗╚╝ 等）让这段内容在模型的输入流里更突出，有助于提高遵守率，但不是必需的，去掉边框也能正常工作 * 脚本本身没有任何条件判断——每次都注入。如果你希望某些情况下不注入，在脚本里加 if 判断，不满足条件时调用 process.exit(0) 静默退出   脚本二：hook-cwd-check.js  检测当前工作目录，特定条件下触发提醒。  #!/usr/bin/env node    const cwd = process.cwd().replace(/\\/g,'/');    if(/\/\.claude$/.test(cwd)){  process.stdout.write(JSON.stringify({  hookSpecificOutput:{  hookEventName:'UserPromptSubmit',  additionalContext:'[警告] 强制提醒：当前 cwd 是 .claude 目录，必须立即执行新会话启动协议第0步——提示用户切换项目或确认后再继续'  }  }));  }  // 不满足条件时：不输出任何内容，脚本静默结束    这个脚本演示了「条件性注入」的模式：检测到特定状态才注入，否则完全静默。静默退出不会产生任何副作用——Claude Code 收到空输出时不做任何处理。  replace(/\\/g, '/') 是 Windows 路径兼容处理：Windows 的路径分隔符是 \，正则里需要先统一转成 / 才能正常匹配。  怎么写自定义 Hook  任何可以通过脚本自动检测、以文本形式表达的约束，都适合做成 hook。步骤如下：  第一步：确定注入内容  把你希望模型每次都看到的规则写成字符串。注意：   * 内容要简洁。太长的注入（超过 1000 字）会占用上下文窗口，稀释其他信息的权重 * 把最关键的规则写在前面，因为注入内容越靠前在注意力分配上越有优势 * 规则描述要可操作（「每条回复必须列出工具调用」比「要诚实」有效得多）   第二步：确定触发条件  Hook 是无条件执行的（每次用户发消息都跑）。如果你的约束只在特定情况下有意义，在脚本里加条件判断：  #!/usr/bin/env node    // 示例：只在 content 目录下的项目中注入写作规范  const cwd = process.cwd().replace(/\\/g,'/');  if(!cwd.includes('/content/')){  process.exit(0);// 静默退出，不注入  }    process.stdout.write(JSON.stringify({  hookSpecificOutput:{  hookEventName:"UserPromptSubmit",  additionalContext:"当前是内容创作项目，回复中涉及文章风格的建议必须参考 writing\_style\_v2.md"  }  }));    第三步：处理错误  脚本内部的错误不会暴露给用户，会静默失败（注入不生效，没有报错）。务必在脚本里做基本的错误处理：  try{  // 你的逻辑  process.stdout.write(JSON.stringify({...}));  }catch(err){  // 写到 stderr（被 Claude Code 忽略，但方便你 debug 时查看）  process.stderr.write('hook error: '+ err.message +'\n');  process.exit(0);// 出错时静默退出，不影响主流程  }    常见错误汇总：  错误后果修复方式用 console.log 代替 process.stdout.writeJSON 末尾有换行符，可能解析失败改用 process.stdout.write脚本内部抛出错误未捕获注入静默失败，无任何提示用 try/catch 包裹，错误写到 stderr路径用相对路径（./scripts/）cwd 变化时脚本找不到，静默失败改用绝对路径或 ~hookEventName 拼写错误Claude Code 无法识别，注入不生效严格用 "UserPromptSubmit" 或 "Stop"注入内容过长注入生效但挤压其他上下文的注意力权重精简注入内容，删掉冗余规则Hook 层的边界  Hook 能做到的：   * 每次对话强制注入固定纪律（消除遗忘） * 根据运行时状态（cwd、文件是否存在等）条件性注入提醒 * 在会话结束时触发外部脚本（导出、归档等）   Hook 做不到的：   * 拦截或修改模型输出：hook 不能阻止模型说某些内容，也不能在输出后自动修改它 * 在文件写入前执行检查：没有 PreWrite/PreToolCall 事件，无法在模型调用工具前介入 * 修改用户消息：只能在消息旁边注入额外上下文，不能替换或过滤用户消息的内容   这些限制是真实的技术边界，不是配置问题。需要在文件写入后自动触发检查的，靠 Scripts 层解决（第四篇）。  二、Skill 层：流程锁定这层解决什么问题  你在对话里告诉模型「遇到 bug 先分析根本原因，再形成假设，再测试修复」，它这次照做了。下次遇到 bug，没有特别提醒，它回到原来的方式——直接猜着改代码。  这是「一次性提示」的根本局限：方法论需要每次重新建立。每种任务类型都要重新交代工作方法，交代了也不稳定，在长对话中会退化。  Skill 的设计目标是把工作方法从「一次性指令」变成「持久化协议」。每种任务类型有对应的 skill，里面定义了「这类任务永远按这个流程做」。当任务被识别出来，skill 自动加载，方法论自动就位，不需要每次提醒。  安装：enabledPlugins 配置  Superpowers 是目前覆盖最完整的官方 skill 插件集合。在 ~/.claude/settings.json 中添加以下两个字段：  {  "enabledPlugins":{  "superpowers@claude-plugins-official":true  },  "extraKnownMarketplaces":{  "claude-plugins-official":{  "source":{  "source":"github",  "repo":"anthropics/claude-plugins-official"  }  }  }  }    字段说明：   * enabledPlugins：键是 插件名@marketplace名，值固定为 true * extraKnownMarketplaces：声明 marketplace 的源地址，Claude Code 据此拉取插件元数据；需要能访问 GitHub   配置完成后，重启 Claude Code（配置文件的更改需要重启才生效）。重启后输入 /skills，应该能看到已安装的 skill 列表。  如果 /skills 没有响应或列表为空：   1. 确认已重启 Claude Code（不是新建会话，是完全关掉再开） 2. 检查 settings.json 格式是否合法（JSON 语法错误会导致配置文件加载失败，不报错） 3. 确认网络能访问 GitHub   Skill 的自动激活机制  每个 skill 都有一个 description 字段，描述它的触发场景。当你的任务描述在语义上匹配这个字段时，Claude Code 会自动加载对应的 skill。   * 说「帮我给这个功能写单元测试」→ test-driven-development skill 自动激活 * 说「这个 bug 不知道怎么回事」→ systematic-debugging skill 自动激活 * 说「我要规划一下这个功能的实现」→ writing-plans 可能自动激活   自动激活是语义匹配，不是关键词精确匹配。如果没有自动触发（任务描述比较模糊时可能发生），可以手动输入 /skill名称 强制调用。  Superpowers 完整 Skill 列表  刚性3 个 Skill，写入 enabledPlugins 后一次性全部生效。  本地安装的约束型 Skill  除 Superpowers 插件外，还有几个本地安装的 Skill 承担专项约束职能，放在 ~/.claude/skills/ 目录下，由 SKILL.md 的 description 字段驱动触发：  Skill类型职责neat-freak（洁癖）审查型会话结束后对知识库做洁癖级检查：CLAUDE.md、docs/、Agent 记忆三层必须和代码保持同步，过期信息强制删除skill-vetter（安全审查）刚性型安装任何第三方 Skill 之前必须运行：检查危险操作（curl 到未知 URL、读取 ~/.ssh）、权限范围、可疑代码模式，有红旗直接拒绝安装  这两个 Skill 的约束方向和 Superpowers 不重叠：Superpowers 约束「怎么写代码、怎么推进任务」，neat-freak 约束「知识库要不要跟上代码变化」，skill-vetter 约束「新工具有没有安全问题」。三个方向各管一块，互不干扰。  三个关键刚性 Skill 的核心规则  Skill 1：verification-before-completion  核心原则：声明完成之前，必须有此刻刚刚执行的验证证据。  「新鲜」是关键词。不是「上次检查时没问题」，不是「通常这种情况不会出错」，是刚刚执行、刚刚拿到结果的验证。  验证的五步流程（固定，不可跳步骤）：   1. Identify：确定什么命令或检查能证明这个任务完成了（具体到可执行的命令，不是模糊描述） 2. Run：执行这个命令——完整执行，不能说「应该过了，就不跑了」 3. Read：读输出，检查退出码；退出码非 0 意味着失败，不能忽略 4. Verify：确认输出内容支持「完成」的结论，不只是「命令执行完了」 5. Only then：以上四步全部完成后，才允许声明「已完成」   触发违规的红线（出现任何一条）：   * 用 “should work”、”probably fine”、”seems to be working” 等表述代替实际验证 * 在没有执行任何检查的情况下表达满意或声明完成 * 信任另一个 agent 或上一步工具的「成功」报告，不亲自验证 * 只验证了代码能运行，没有验证行为符合预期   Skill 2：systematic-debugging  核心原则：找到根本原因之前，不许修改代码。  「修了这个又冒出那个」的根源几乎总是：在现象层打补丁，没有找到真正的原因。这个 skill 强制先理解后动手。  四阶段流程：  1  Root Cause  读完整的错误信息；尝试复现；检查近期所有相关改动。  ✓能用一句话说清楚「什么坏了，为什么坏了」  2  Pattern  找正常工作的类似案例；与参考实现对比；识别当前状态和预期状态的差异。  ✓明确是哪个地方、哪个假设出了问题  3  Hypothesis  形成单一、可验证的假设；设计最小改动来测试。  ✓假设被验证（对或错），结果决定下一步  4  Implementation  创建复现 bug 的失败测试；实施修复；验证测试通过。  ✓Bug 解决，测试通过，没有引入新问题  关键红线：   * 同一个 bug 改了 3 次以上还没解决 → 停止修复，质疑整体架构或问题理解 * 阶段 1 没完成就开始改代码（最常见的违规） * 同时测试两个假设（「我改了 A 和 B，不知道哪个修好的」） * 用注释掉报错的方式让代码「能跑」——这不是修复，是掩盖   Skill 3：test-driven-development  核心原则：没有失败的测试，不写生产代码。  Red-Green-Refactor 循环（每个功能点都要走一遍）：  RED 阶段：  1. 写一个描述期望行为的测试  2. 运行这个测试，确认它失败了（证明测试本身有效）  → 如果没有失败，说明测试没有在检测任何东西，重写测试    GREEN 阶段：  3. 写最少的代码让这个测试通过  → 不在这个阶段考虑代码质量，只要测试过就行  4. 运行测试，确认通过    REFACTOR 阶段：  5. 在测试通过的状态下清理代码（去重、重命名、优化结构）  → 重构过程中不加新功能  6. 每次重构后运行测试，确认还在通过    循环结束，开始下一个功能点的 RED。    每个循环结束时，代码库必须处于「全部测试通过」的状态。不允许在测试失败的情况下进入下一个功能点。  如何验证 Skill 已生效   1. 新会话启动时，系统提示区域出现 superpowers 的初始化信息 2. 说一个明确属于某个 skill 触发场景的任务，模型主动提出要走对应流程 3. 任务结束时，模型主动要求验证，不直接声明「已完成」 4. 手动输入 /verification-before-completion 等 skill 名，得到该 skill 的结构化响应   三、CLAUDE.md 层：分级规则体系这层解决什么问题  Hook 把约束注入每次对话，Skill 把工作方法锁定到任务类型。这两层都很好，但有一类约束它们处理不了：依赖大量背景上下文才有意义的规范。  比如：「这个项目用 TypeScript，类型不能用 any，因为我们的 API 层有严格的类型推断依赖」——这不是一条可以简短注入的纪律，是需要配合整个项目背景才能被正确执行的规范。  再比如：「新会话开始时先检查是否有未归档会话，再读项目注册表，再加载当前项目档案，最后回答用户问题」——这是一套完整的启动协议，内容太长不适合用 hook 注入，但又必须在每次对话开始时就执行。  CLAUDE.md 是这类规范的载体：Claude Code 在对话启动时，按目录层级自动加载所有适用的 CLAUDE.md 文件。规则持久存在于文件系统里，不需要手动粘贴，不需要每次提醒。  三层叠加结构  CLAUDE.md 按目录层级组织，形成三层叠加的规则体系。  ~/.claude/CLAUDE.md                 ← 个人哲学层（全局生效）  ↓  ~/workspace/CLAUDE.md               ← Workspace 层（当前工作区生效）  ↓  ~/workspace/project-name/CLAUDE.md  ← 项目层（仅这个项目生效）    下层规则补充上层，同类规则时下层更具体，优先级通常更高。  第一层：个人哲学层（~/.claude/CLAUDE.md）  覆盖所有对话、所有项目，无例外。  适合放什么：   * 认知方法和思维方式（「第一性原理」、「结论先行」、「精确理解问题范围」） * 跨项目通用的沟通风格（「默认中文，代码用英文」、「不复述用户说过的话」） * 跨项目通用的行为约束（「不自动 push 到远端」、「后端代码修改后完整重启」） * 会话启动和结束协议（固定的启动检查步骤、归档流程） * 全局纪律（「事实核实纪律」、「交付前强制步骤」）   示例片段：  ## 第一性原理    所有决策从问题本质出发，不因「惯例如此」照搬。方案有问题直接指出。  精确理解问题的范围和意图，不扩大、不缩小、不偷换。    ## 沟通风格    - 默认中文，代码/命令/变量名用英文  - 结论先行，再给理由和分析  - 不复述用户说过的话，不做无信息量的总结  - 遇到模糊需求，先给最合理的方案，再问要不要调整    ## 全局约束    - git push 仅用于跨设备同步，不自动执行，等用户说  - 后端代码修改后完整重启：杀旧进程→清缓存→重启→API 验证通过→再告知  - 不为了让代码跑起来而注释掉报错，找根本原因    ## 交付前强制步骤（不可跳过）    1. 冷却审查，切换红队角色  2. 代码检查，构建验证  3. 设计检查，9 维度审计  4. 体验检查，交互验证  5. 交付声明（强制附加检查结果）    缺任何一步 = 违规    第二层：Workspace 层（工作目录/CLAUDE.md）  覆盖这个工作目录下的所有项目，补充个人哲学层里没有定义的工作区特有规范。  适合放什么：   * 工作区的目录结构和分区规则（什么类型的项目放哪个子目录） * 新项目创建流程（「先建文件夹，再生成最小 CLAUDE.md，再开发」） * 跨项目共用的 Git 工作流（分支策略、commit 规范） * 跨项目共用的质量标准和工具配置   示例片段：  ## 目录分区    workspace/  ├── learning/     # 学习类项目  ├── content/      # 内容创作  ├── products/     # 产品开发  └── my/           # 实验性/一次性    ## 新项目协议    用户说「帮我建个项目」时，必须先问放哪个分区，提议 1-3 个命名候选，  建文件夹，生成最小 CLAUDE.md，然后才开始开发。不能跳过。    ## Git 工作流（全 workspace 统一）    - 统一用 main 分支，单人项目不需要 feature 分支  - commit message 格式：类型: 简短描述（英文）  - 不自动 push，等用户明确说    第三层：项目层（项目目录/CLAUDE.md）  仅对这一个项目生效，覆盖范围最窄，规则可以最具体。  适合放什么：   * 技术栈约定（语言版本、依赖选型、禁用的库） * 启动命令和部署流程（具体的命令行指令） * 领域特有的约束和规范（「这个接口只能用 Bearer Token 认证」） * 当前阶段的重点和搁置范围（「现在专注支付流程，不动认证模块」） * 子项目索引（各个功能模块在哪个目录）   一个实际的项目层 CLAUDE.md 示例：  # 知识库项目    ## 技术栈    - 后端：Python 3.12 + FastAPI  - 向量库：Chroma（本地部署，不用托管服务）  - 前端：原生 HTML/CSS，不引入任何框架  - 启动：source .venv/bin/activate && uvicorn app.main:app --reload --port 8000    ## 核心约束    - 不引入 SQLAlchemy，直接用 Chroma 的 Collection API  - API key 不能硬编码，统一从 .env 读取，.env 不提交 git  - 前端文件 Write/Edit 后必须执行 node ~/.claude/scripts/audit-html.js  - 不引入任何会增加打包体积的前端依赖（现在是零依赖）    ## 当前阶段    专注：文档摄入和语义检索核心功能  搁置：用户管理、权限控制——等核心功能稳定后再议  禁止动：现有的 API schema 定义，有变更需求先提出再讨论    ## 子项目结构    - app/       FastAPI 后端  - ingest/    文档处理和向量化流水线  - static/    前端页面  - scripts/   维护脚本    CLAUDE.md 的加载机制  Claude Code 在对话启动时，从当前工作目录开始向上遍历目录树，找到所有 CLAUDE.md 文件并全部加载。加载顺序：从最外层到当前目录（全局层最先，项目层最后），后加载的规则对同类规则有更高优先级。  几个需要知道的细节：   * 只加载 cwd 路径上的 CLAUDE.md，不加载 sibling 目录的规则 * 如果 cwd 是 /workspace/content/project-a，加载的是 ~/.claude/CLAUDE.md + /workspace/CLAUDE.md + /workspace/content/CLAUDE.md + /workspace/content/project-a/CLAUDE.md，不加载 /workspace/content/project-b/CLAUDE.md * 某一层 CLAUDE.md 不存在时，跳过那一层，继续加载其他层   什么该写，什么不该写  该写进 CLAUDE.md 的：   * 持久性的规范和约束（不随单次对话变化的） * 需要每次会话开始时就生效的信息 * 项目结构性事实（目录布局、技术选型、重要约定） * 启动和结束协议（每次都要执行的固定流程）   不该写进 CLAUDE.md 的：  内容类型问题正确的处理方式当前任务进度（「现在做到第3步」）任务结束后变成过时信息，污染后续会话用 Memory 层的项目档案记录临时性规则（「这次不用跑测试」）临时的事就在对话里说，写进文件等于永久生效直接在对话里说代码模式和架构细节代码本身是事实，CLAUDE.md 里的描述会过时让模型直接读代码频繁变更的内容每次变更都要同步更新文件，维护成本高评估是否真的需要在规范层体现  一个常见的错误：把「当前阶段正在做什么」写进项目 CLAUDE.md。这会导致下次会话（可能是几周后，当前阶段早就完成了）时，模型读到过时的阶段描述，产生错误的上下文。当前任务状态应该用 Memory 层的项目档案管理（第四篇）。  冲突处理规则  当设计原则与用户当次需求发生冲突时，模型正确的处理方式是：暂停执行 → 明确列出冲突点 → 说明遵守原则和满足需求各自的后果 → 等待用户决策。不能悄悄选一个执行，不能用「惯例如此」绕过。  如果发现模型在悄悄妥协，通常是规则描述不够清晰的信号——「全局强制」或「不可跳过」这类明确措辞有助于让模型判断这条规则是否真的不可绕过。  三层的协同逻辑  输入端三层不是独立运作的，是有分工的。   * Hook 处理「每次对话都需要激活的简短纪律」，靠强制注入确保遗忘不可能发生 * Skill 处理「特定任务类型的完整工作流程」，靠语义触发确保方法论自动就位 * CLAUDE.md 处理「需要背景上下文才能正确执行的规范」，靠分层组织确保规则覆盖精确   三层叠加的效果：在模型处理任务之前，它的认知环境里已经包含了行为纪律（Hook）、工作方法（Skill）、项目规范（CLAUDE.md）。这不是在限制它，是在帮它减少漂移的可能性。  *本文是「Claude Code 七层约束」系列的第 2 篇。系列共四篇，分别讲 AI 行为漂移的底层原因（第一篇）、输入端三层配置（本篇）、认知与质量两层（第三篇）、执行与记忆两层（第四篇）。完整七层约束系统配置已开源：https://github.com/lh180lh-sys/claude-code-constraints*  调研 & 撰写：AI（Claude）、墨染 主导 & 审校：墨染
