前特斯拉AI总监、OpenAI联创Andrej Karpathy在GitHub上发了一个知识库文件，叫 llm-wiki.md。75行markdown，不到2000字，一周拿了15000个star。很多人把它理解成"用AI做笔记的新方法"。不是。如果你做投资，它的意义远比"做笔记"大得多。

他到底说了什么
=======

Karpathy的核心观点是，不要每次都让AI临时翻资料回答你。让AI帮你持续维护一个知识库，新信息进来就整理好，旧信息自动更新，资料之间的关联自动建立。他管这个叫 LLM Wiki，说白了就是知识库。那这个和我们现在常见的"丢文档给AI问问题"（也就是RAG）有什么区别？打个比方：RAG 像你每次出门都现查地图。堵车了？查一次。换路了？再查一次。每次都从零开始。LLM Wiki 像有一个助手，每天帮你更新一张私人地图。哪条路修了，哪里新开了路，哪个口子最近特别堵——都标好了。你出门前扫一眼就行。区别在四个字：知识复利。

为什么这件事对炒股的人特别重要
===============

做投资的人，最大的损耗是什么？不是信息少。2026年了，信息多到喝水都能呛着。是研究不复利。你一定有过这种经历：去年认真研究过一只股票，写了好多笔记，今年再看，像没研究过一样买入时有5条理由，跌了30%以后一条都想不起来看了20份研报，关掉浏览器，脑子里只剩一个模糊的"好像挺看好的"卖完一只股票，从来没认真想过"这次到底学到了什么"问题不在于你不够努力。问题在于，没有人帮你把零散的研究整理成可以反复调用、持续更新的资产。这恰好是Karpathy解决的事。

从炒股角度来进行架构拆解
============

Karpathy的系统有三层，非常清晰：第一层：原始资料财报、研报、新闻、电话会纪要、行情数据、聊天记录。这些是"只读"的——进来了就不改，永远当原始证据保存。第二层：知识库（Wiki）这是核心。AI 读完原始资料后，不是简单做个摘要就完事。而是：给每家公司建一个"档案页"：基本面、看多逻辑、风险、催化剂、退出条件给每个投资主题建一个"主题页"：AI算力周期、存储景气度、港股高息、美联储政策给每个策略建一个"策略页"：动量策略适合什么市场、全天候组合怎么配给每个持仓建一个"论据页"：我为什么买的、证据是什么、什么情况下论据失效最关键的是——这些页面之间互相连接。美光的公司页链接到存储周期的主题页，存储周期链接到AI算力主题页，AI算力又链接到英伟达和台积电。新的财报出来，不只是更新一个页面，而是沿着这些连接，把相关的页面全部同步更新。Karpathy自己的原话：

"一个新资料进来，可能要触及10-15个Wiki页面。"

这才是"知识复利"的意思：不是存了多少，而是连接了多少。第三层：规则文件一份告诉AI"怎么维护这个知识库"的说明书。

* 什么格式
* 什么命名规范
* 新资料进来怎么处理
* 多久做一次"健康检查"
* 哪些矛盾需要标记

相当于给AI立了一套投研纪律。

三个最狠的操作
=======

他定义了三种核心操作，我觉得每一种都直接能用到投资上。操作一：Ingest（入库）新资料进来，不是往文件夹里一扔就完了。AI 会：

* 读完资料
* 提取关键信息
* 更新对应的公司页、主题页、论据页
* 标记新信息和旧结论之间的矛盾
* 记录日志

举个例子：一份美光季报出来了。AI 不只是写个"美光Q2收入beat"的摘要。它还会去检查——之前存储周期主题页里写的"库存正在去化"，和这份财报里的库存数据一致吗？之前的看多论据"DRAM价格趋势向上"，这次财报支持还是削弱了？这才是真正的研究助理该做的事。操作二：Query（查询）你问问题，AI 不是从零开始翻资料。它先看索引，找到相关的Wiki页面，读完之后综合回答。更重要的是——好的回答可以存回知识库。你问了一个"美光现在的看多逻辑还成立吗？"，AI 给了一个高质量分析，这个分析本身就变成知识库的一部分，下次不用重新推导。这就像你的研究在"滚雪球"。操作三：Lint（健康检查）定期让AI扫描整个知识库，找问题：有没有页面超过30天没更新？有没有论据已经被新数据推翻但还没标记？有没有公司页之间该有的关联没建上？有没有该关注但还没建页面的主题？这一步是投资者最容易忽略、也最需要的。因为投资里最贵的错误，往往不是"买错了"，而是"该更新判断的时候没更新"。你还在拿着一个半年前的看多逻辑，但世界已经变了。

他真正说对了什么
========

Karpathy最后说了一句很到位的话："人类放弃维护知识库，是因为维护成本增长快于知识价值。LLM改变了这个等式。"这句话值得再读一遍。为什么你的Notion笔记最后都荒了？为什么你的读书笔记从来不翻第二遍？为什么你明明研究过的公司，两个月后像没看过一样？不是你懒。是维护知识库这件事，对人类来说成本太高了。更新交叉引用、保持信息同步、标记矛盾、补充缺漏——这些事极其重要，但极其无聊。LLM 不觉得无聊。它一次能改15个文件，不会漏掉交叉引用，不会忘记标记矛盾。当维护成本接近零的时候，知识库才真正能复利。

从"做研究"到"养资产"
============

不要只做研究，要养研究资产。"做研究"是一次性的——看了一份报告，做了一个判断，然后这个判断就埋在聊天记录里了。"养资产"是持续的——每一份报告都变成知识库的一部分，每一个判断都有证据链，每一次市场验证都更新了你对这家公司的理解。具体来说：做研究｜养研究资产看完研报，记个要点｜更新到公司页，关联到主题页买入时心里有想法｜写成thesis，列出证据和反证跌了，"拿着吧" ｜检查论据是被强化了还是被削弱了卖了，结束 ｜复盘页：为什么买、为什么卖、学到什么再看同一只股票，从头来 ｜打开公司页，所有历史研究都在前者是消耗，后者是积累。

这和量化有什么关系
=========

你可能觉得这是主观投资者的事。做量化的人不需要这些。恰恰相反。量化最怕的就是叙事漂移——你用一套逻辑建了模型，回测很好，上线跑。跑了三个月，回撤了。你开始讲一个新故事来解释为什么回撤是合理的。再过三个月，你已经忘了最初那套逻辑到底是什么了。知识库能干什么？锁住原始逻辑：建仓时的依据、参数选择的原因、回测里哪些假设是关键的追踪演变：什么时候市场环境变了、什么时候你调了参数、为什么调的防止自欺：当初说好的退出条件，到底是否触发了主观研究给方向，量化模型给纪律，知识库给记忆。三者缺一不可。

我正在做的方向
=======

说到这里，其实可以分享一下我们在 Alpha GBM 上的一些思考。我们做Alpha GBM，不是想用模型替代投资者思考。而是想做一件事：把信息、论据、量化验证、风险管理和执行建议连成一个闭环。信息层：新闻、财报、行情数据的结构化整理论据层：为什么看多/看空，证据是什么，证据有没有过期验证层：历史统计、因子回测、风险暴露分析执行层：在什么位置买、仓位多大、什么条件退出复盘层：事后追踪，判断有没有失真这不是一个"选股工具"。这是一个让投资判断从"拍脑袋"变成"可追踪、可校验、可迭代"的系统。和Karpathy说的LLM Wiki，底层逻辑是一样的：让认知可以复利。

最后
==

Karpathy那75行markdown知识库之所以火，不是因为技术多高深。是因为它击中了一个所有人都有、但很少有人说清楚的痛点：我们不缺信息，我们缺的是把信息变成可复用判断的能力。对做投资的人来说，这个痛点还要再加一句：在市场里，过期的判断比没有判断更危险。知识库不是用来存资料的。是用来确保你的判断永远跟得上最新事实的。

### 🔗 核心文件地址

* **原始 Gist（想法文件）**：<https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f>
* **文件内容**：一个约 75 行、不到 2000 字的 Markdown 文件，旨在被复制粘贴到任何 AI 代理（如 Claude Code、Cursor、OpenAI Codex 等）中，作为一套“操作说明书”。

### 💡 核心思想：从“检索”到“编译”

Karpathy 提出的不是一个新的软件，而是一个模式（Pattern）。其核心是颠覆传统的 RAG（检索增强生成）思路：

* **传统 RAG**：每次提问，LLM 都从零开始重新检索、阅读原始文档并拼凑答案，没有积累。
* **LLM Wiki 模式**：让 LLM 扮演“**持续整理的知识管理员**”角色。你只需将新的原始资料（文章、PDF、笔记）放入一个文件夹，LLM 会主动阅读、提取关键信息，并**增量式地构建和维护一个结构化的、相互链接的 Wiki（Markdown 文件集合）**。知识被“编译”一次，之后持续更新，而非每次查询都重新推导。

### 🏗️ 三层架构

1. **原始资料层（Raw Sources）**：你收集的不可变文档（PDF、文章、图片等），LLM 只读不写。
2. **Wiki 层**：由 LLM 全权创建和维护的 Markdown 文件目录，包含摘要页、实体页、概念页、索引等。
3. **模式层（Schema）**：一个配置文件（如 [`CLAUDE.md`](http://CLAUDE.md)或 [`AGENTS.md`](http://AGENTS.md)），它定义了 Wiki 的组织规则、工作流和惯例，将通用 AI 代理转变为有纪律的 Wiki 维护者。

### 🛠️ 如何与你的工具链结合？

这个模式与你之前关注的工具完美契合：

* **与 OpenClaw 集成**：你可以将 [`llm-wiki.md`](http://llm-wiki.md)或基于它定制的 [`AGENTS.md`](http://AGENTS.md)文件作为 **Skill** 加载到 OpenClaw 中，让它成为你的“知识库管理员”。
* **与 Obsidian 配合**：生成的 Wiki 就是纯 Markdown 文件，可以直接用 Obsidian 打开、浏览和双向链接，形成可视化知识图谱。
* **与知乎 CLI 联动**：用知乎 CLI 抓取的文章，可以直接作为“原始资料”放入 `raw/`文件夹，由 LLM 自动整理到 Wiki 中。

### 🚀 社区实现与衍生项目

由于 Karpathy 只提供了“模式”而非具体产品，社区已涌现出多种实现：

* **轻量提示词版**：一份约 200 行的规则文件，任何支持 Agent Skills 协议的 AI 工具都能调用，无代码依赖。
* **完整桌面应用版**：带有知识图谱可视化、多格式文档解析、浏览器剪藏等功能的独立应用。
* **开箱即用模板**：一个受欢迎的 GitHub 仓库 [**balukosuri/llm-wiki-karpathy**](https://github.com/balukosuri/llm-wiki-karpathy) 提供了预配置的 Obsidian 库、定制化的 [`CLAUDE.md`](http://CLAUDE.md)和文件夹结构，让你在 30 分钟内就能拥有一个运行中的个人知识库。

**总结**：[`llm-wiki.md`](http://llm-wiki.md)之所以能迅速获得海量关注，正是因为它用一个极其简洁的文档，指出了一个根本性的范式转变——**让 LLM 从被动的“检索员”变为主动的“知识编译与维护者”**。这为构建真正具有复利效应的个人知识系统提供了清晰、可落地的蓝图。
