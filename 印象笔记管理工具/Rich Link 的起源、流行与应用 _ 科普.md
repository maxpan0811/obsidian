这就是 Rich Link：**与其使用字符来单调地展示一个链接，不如把链接展示成图文丰富的形式。**Rich Link 正在变得越来越普及，无论是微博、Facebook 中的 Feeds 流，还是 IM 软件中的链接预览，本质上都是一种 Rich Link。

就连苹果也开始在系统中更广泛地使用 Rich Link。在今年的 WWDC 2019 当中，苹果在一个 Session 里介绍了《[Embedding and Sharing Visually Rich Links](https://developer.apple.com/videos/play/wwdc2019/262/)》，在即将到来的 iOS 13 中，苹果将 Rich Link 开放成为了一个系统级别的能力，方便开发者直接调用系统级 API，在自己的 app 里也实现这样的效果。

Rich Link 的各种展现形式

那么这篇文章，我们就从 Rich Link 展开讲讲，它的起源、流行和应用。

Rich Link 的起源
-------------

在说 Rich Link 之前，我们先回归本源，看看从一串简单的链接文本，是如何逐渐演变到形式丰富的 Rich Link 的。这件事情的核心本质是，我们希望链接不仅仅是链接，还能从它最终所指向的目的地中，**提取出更多有用的信息，让人们在点击之前，能掌握更多的信息。**

链接本身就像是门牌号，带领我们通往互联网的各个地方，在 app 兴起前的 PC 时代尤其如此。不过，就像我们拿到经纬度坐标一样，它虽然代表了一个地理位置，但却让人摸不着头脑。单独的链接也是如此，为了增加链接的可理解性，最初的努力，就是在 HTML 中定义了 `<a>` 标签。按照 `<a>` 标签的语法定义，除了声明实际要前往的链接外，你还可以额外添加文字说明，告诉大家跳往的内容是什么。

```
- 纯链接形式：https://sspai.com/post/54816

- <a> 标签形式：<a href="https://sspai.com/post/54816">基于文件夹的图片素材管理思路</a>
```

有了它之后，你看到的不再是一个个毫无意义的字符串，而是配上文字说明的入口。这样，至少让人一眼看了，知道自己会去哪。我们可能都已经习惯了这样的形式，但想像一下，如果我们不使用 `<a>` 标签，而把链接直接放在网页上，例如 Power+ 的文章目录，就完全让人看不懂了。

![](.evernote-content/DFF916BE-0D57-4791-A5FA-E84B53321EA1/A86FD9D7-8713-46B3-828D-BDA730915836.png)

想像一下如果链接展现成右边那样

不过，`<a>` 标签只是试图让链接更容易被人理解的一个最早的尝试，如果说链接就像经纬度坐标，那么 `<a>` 标签则像报出了一个地名。而显然，还存在着更直观的方式，例如展现出一幅地图，用来指路会更加清晰。

同样的道理，许多公司在他们的产品里，为了能向用户传递更多关于要点击的链接信息，也做过类似的努力。例如，Google 在搜索结果中，不仅展示网页的标题，还会摘要出一小段网页内容。在 2009 年，还推出过 [Rich Snippets](https://webmasters.googleblog.com/2009/05/introducing-rich-snippets.html) 特性，在搜索结果中，可以拉取网页中开发者事先声明好的一些特定信息。例如当搜索餐馆时，Yelp 的结果会额外显示星级、评价人数与客单价等信息。

![](.evernote-content/DFF916BE-0D57-4791-A5FA-E84B53321EA1/AA821C1F-D857-4936-A54E-7D68F4DCF8FE.png)

Google 搜索结果中额外展示信息

不过在我看来，直到这一步各个公司的意图还是希望能让链接传递出更多有用的信息，方便人们决策是否需要进一步的点击。那么事情，是从哪里开始发生了变化呢？

Rich Link 的流行
-------------

真正让 Rich Link 流行起来，乃至成为一种行业标准的，则是 Facebook。

如果要选出一项让 Facebook 大获成功的产品创新，这个答案当之无疑就是 Feeds 流。如今我们已经习惯了刷微博、刷知乎、刷朋友圈，但你可能很难想像，在 Facebook 诞生之前，并没有产品提供 Feeds 流，无论是彼时国外火爆的 MySpace，还是国内曾经一度流行的 QQ 空间。如果你想看哪位好友有最新动态时，你必须点进那个人的主页。

Feeds 流的出现，成为了人们消磨时间的最佳方式。每一次下拉刷新，就像打开水龙头开关，总有源源不断的新内容推送到你的面前。对于这些公司来说，一方面要生产出源源不断的 Feeds 流，而另一方面，一个重要的工作，就是不断优化 Feeds 中的展示形式，让你更有欲望打开。

Facebook 早年在 Feeds 流的展现形式上，投入了巨大的精力，尝试各种样式与设计，进行各种 A/B 测试，以找到让用户最有可能打开的展现形式。即使到后期，Facebook 每年也会对 Feeds 形态不断迭代更新。在今年公布全新的 Facebook 设计之前，上一次较大的设计调整是 2017 年。尽管第一眼看来，似乎区别不大，但变化比较明显的是 Link Share 的样式，网页图片从原来的卡片，变成了和用户分享照片一样，占据了整个屏幕的宽度，视觉上更加醒目。

![](.evernote-content/DFF916BE-0D57-4791-A5FA-E84B53321EA1/EB300453-5465-4B97-A583-C17BA47FA9A6.png)

2017 Facebook 设计改版前

![](.evernote-content/DFF916BE-0D57-4791-A5FA-E84B53321EA1/84EB4BBB-6A1C-4E0F-AAB7-76A00F4C0D34.png)

2017 Facebook 设计改版后

可以看到，通过突显出图片，Rich Link 在视觉上更加吸引人们的眼球，仿佛就像是用户发布了新照片一样。在某种程度上，这些设计的出发点，就是希望诱导你点击更多 Feeds 流中的内容。更多的点击，意味着更长的用户停留时间，意味着更多的广告收入。

现如今，不仅仅是 Feeds 流已经行为了各种产品的标配，很多人没有意识到的是，Rich Link 也在这个过程中，成为了一种标配。无论是文章开头的少数派微博截图，还是你在用的各类产品，链接都绝不再仅仅只是一串单调的字符，或是一个简单的文本，它一定是图文并茂的形式。甚至，像在微博中，为了吸引你的注意力，这一类的链接预览图片，还会自动轮播变换。

微博中的链接预览甚至会轮播图片

而且，这种 Rich Link 生成的门槛非常之低。在微博、Facebook 中发布内容时，只要你填写了一个链接，它就会自动去获取链接的相关信息，从而生成一个链接预览。

![](.evernote-content/DFF916BE-0D57-4791-A5FA-E84B53321EA1/DB03F6FA-5E56-418B-B787-4E85775FC530.png)

Facebook 中发布链接会自动生成预览

到了这一步，Rich Link 多少显得有些本末倒置。

从最初只是希望为枯燥的链接，提供更多的额外信息，方便人们判断是否需要进一步点开，到现在各个平台上 Rich Link 已经变成了一种标配性的装饰，诱导人们更多地打开和消费内容，从而增加停留在平台上的时间和相关的广告收入。这样的产品形态和展现设计，也在一定程度上，倒逼着内容生产者，必须向其靠拢。就和标题党盛行一样，如今一篇内容，花在标题和题图上的时间，很多时候甚至超过了文章内容本身。

不过，这样的反思，在点击量、转化率、广告收入等等可以量化的商业指标面前，显得多少有些微不足道。Facebook 不仅仅让 Rich Link 流行起来，甚至还将其定义成为了一种事实上的标准规范。

Rich Link 成为一种标准
----------------

在 Facebook 还不够壮大时，生成链接预览这件事，更依赖于 Facebook 针对每一个站点和网页的优化，难免会有站点的链接预览效果不佳。不过，当 Facebook 逐渐成为一个通用的社交平台，话语权也越来越大时，在 2010 年，Facebook 开放了一个叫做 [Open Graph protocol](http://ogp.me/) 的标准，任何遵循这个标准的网页，当被分享在 Facebook 的时间线中，都可以获得更规范的展示效果。网页开发者只要按照 Open Graph 定义的标准，其展现效果就有了保证。

自此，Open Graph 已经成为了一种通用的标准。当前大多数的平台和 IM 软件，其链接预览的特性，都是遵循 Open Graph 标准的。例如，常见的 Telegram、Slack，以及 Facebook 自身旗下的 IM 软件，都遵循了这一标准。Twitter 虽然也有一套自身类似的标准，叫做 [Twitter Cards](https://developer.twitter.com/en/docs/tweets/optimize-with-cards/guides/getting-started)，但它也是兼容 Open Graph 标准的。

其实，Open Graph 这套标准并不复杂，它的实现原理非常简单。如果你查看一个网页的源代码，你会发现从整体结构上，每个网页都分为 `<head>` 和 `<body>` 两部分。`<body>` 中是这个网页上各种元素和内容的代码，而 `<head>` 中往往不对应网页上的具体内容，在这里可以用来声明一些网页的属性，如页面标题、图标、使用的页面编码等等。

![](.evernote-content/DFF916BE-0D57-4791-A5FA-E84B53321EA1/69CA65B6-AC27-4D06-9129-DF3029D6F59E.png)

少数派网站也声明了 open graph 标准中的标题和描述

Open Graph 标准，只需要你在 `<head>` 中声明网页的标题、类型、展示图和指向地址。除此之外，作为可选的内容还有音频、视频、描述文字等信息。当第三方网站或 IM 在生成链接预览时，会读取这些信息来生成展示。

不过，Open Graph 在国内的普及程度并不高，许多网站并没有遵循 Open Graph 协议。作为平台来说，也不像国外的站点或 IM 工具，统一使用 Open Graph 协议来生成链接预览，而是往往自己独立一套标准，要求第三方来适配。

以国内最大的 IM 软件微信来说，它就没有遵循 Open Graph 协议。如果你想让自己的网页在微信中分享获得更好的展示效果，就需要使用微信平台自己的[一套标准](https://mp.weixin.qq.com/wiki?action=doc&id=mp1421141115#10)。

![](.evernote-content/DFF916BE-0D57-4791-A5FA-E84B53321EA1/F033B8B5-37E0-41C1-8F00-621D28B60ED8.jpg)

微信自己定义的一套标准

由于国内平台都各自搞各自的标准，网站本身也没有积极使用 Open Graph 这套协议，导致国内的站点在国外平台分享时，获得的预览效果往往较差。而在国内站点分享时，则根据不同平台对特定站点的优化程度，显示的效果差异也较大。

Rich Link 在 iOS 中的应用
--------------------

在 iOS 10 推出时，苹果针对 Message 应用做出了重大更新，其中一个新特性就是链接预览（Link Preview）：如果消息的内容是一个链接，那么，Message 会自动将它转换为一个丰富的图文形式展示，配上图片甚至是视频，让你能够更方便地大致了解网页所要展示的内容。自然，苹果的这套 Rich Link 机制也不例外，按照苹果的[开发者文档](https://developer.apple.com/library/archive/technotes/tn2444/_index.html)显示，它同样是遵循 Open Graph 标准的。

![](.evernote-content/DFF916BE-0D57-4791-A5FA-E84B53321EA1/DAA9611F-47DC-4DCE-832D-4EDB3D4D7F2F.png)

iOS Messages 中的链接预览

不过，在 iOS 13 之前，只有原生的 Messages 和 Notes 应用，才能享受到 Rich Link 的体验，而且两者的支持程度也不尽相同：

* **Messages：**有四种展现形式，仅展示 favicon，展示大图，展示需要手动播放的视频，展示自动播放的视频。且只需要在消息中输入链接，系统会自动转换成 Rich Link。
* **Notes：**支持两种展现形式，仅展示 favicon，以及展示大图。如果你直接在 Notes 中输入链接，是不会自动转换生成 Rich Link 的，只有在第三方应用中，分享到 Notes 的链接，才会展现成 Rich Link 的样式。

而到了今年 9 月即将发布的 iOS 13 后，Rich Link 的原生支持应用，上周的 [Power+ 文章](https://sspai.com/post/55237)也提到了，还多了一个「提醒事项」。并且，Rich Link 还成为了一项**系统级的能力**，第三方开发者在自己的 app 中也可以轻易调用。

例如，在苹果官方演示的一个案例中，如果有一款 app，可以在网上浏览到喜欢的食谱时，将链接收藏下来。现在，你可以在 app 内看到更丰富的内容与图片展示，而不是一连串密密麻麻的链接。当然，在 iOS 13 之前，开发者想要实现这样的效果，花点功夫也可以做到。不过在成为一种系统能力之后，要实现这样的效果就简单多了。

![](.evernote-content/DFF916BE-0D57-4791-A5FA-E84B53321EA1/468A7EA1-0C4B-41EC-916B-324907B6B375.png)

传统展现链接方式 vs. Rich Link 的展现方式

但是，并不是所有人都喜欢 iOS 中的 Rich Link。以在 Messages 应用为例，当你发送的链接转为 Rich Link 的大图或视频显示时，往往会需要较长时间的加载（loading）等待，甚至在 2018 年还因为漏洞问题引发过 iOS 系统的[崩溃](https://www.buzzfeednews.com/article/nicolenguyen/chaios-imessage-bug)。不过，苹果并没有提供选项让你能主动关闭 Rich Link 特性。唯一的例外，是在 Messages 应用中，可以通过特定的方式，强制发送链接的原始形态。

你只需要在发送的链接前后，各加上一个点，Messages 应用就不会对其进行 Rich Link 转换，而是保持链接的文本形态。例如，发送「apple.com」会被强制转换成 Rich Link，而发送「.apple.com.」，则最终发出的内容，就是「apple.com」。

![](.evernote-content/DFF916BE-0D57-4791-A5FA-E84B53321EA1/9EFE261E-BD6B-4885-A0ED-5C592F1920B1.png)

强制保持链接的文本形态

以上，就是关于 Rich Link 的起源、流行、标准制定，以及在 iOS 系统中的相关展现。你对现在各个平台，越来越丰富多彩的链接展现形式，和诱导人们点击的做法，有什么想法和思考，也欢迎在评论中和大家一起分享。
