信号1：长Spec死了，10个bullet就能进开发
--------------------------

图里的原话：「以前写详细需求文档，现在很多场景下，写10个bullet points就能进开发。」

这话外行听不出威力。在公司写过PRD的人听到，会愣一下。

因为传统软件工程的入门姿势是这样的：产品经理写10-30页PRD → 拉评审 → 设计师出原型 → 技术leader写设计文档 → 拆成50个ticket → 工程师领ticket → 开发 → 测试 → 上线。这一套跑下来，第一行代码落键之前，少说一周。

Codex团队说的「10个bullet进开发」，不是说**Spec不重要了**，而是说——**「描述过程」的那部分Spec不重要了**。

因为模型自己会拆步骤。你跟它说目标态，它能反推出该改哪几个文件、该跑哪几条命令、该按什么顺序来。这部分以前是高级工程师的活儿，现在被模型吃掉了。

那留给人写的是什么？三件事：

· **目标态**——做完之后世界应该是什么样

· **不碰的边界**——哪些代码 / 数据 / 模块不许动

· **验收标准**——满足什么条件算完成

这三件凑起来，确实就10个bullet。

给你一个我自己天天在用的模板，可以直接抄：

# 任务 bullet 模板（替代 PRD）

【目标态】（1-2 行）

· 做完后用户能 \_\_\_

· 数据 / 界面 / 行为变成 \_\_\_

【输入输出】（3-4 行）

· 输入：来自 \_\_\_ 的 \_\_\_ 字段

· 输出：写到 \_\_\_ 表 / 文件 / 接口

【不许动】（2-3 行）

· 不改 \_\_\_ 表结构

· 不动 \_\_\_ 鉴权模块

· 不引新依赖

【验收】（2-3 行）

· \_\_\_ 命令跑通

· \_\_\_ 页面显示 \_\_\_

· 没引入新 lint warning

这个模板我贴给Claude，它能拆出来的task数量，平均是**12-18个**。比我手动拆要多，而且大多数我能直接ack。

八个月下来，我没写过一份正经PRD。

信号2：Skills比代码值钱
---------------

图里的原话：「把常见任务沉淀成可直接调用的能力，让模型去跑。」

这一条是四个信号里我个人觉得最深的。

在传统软件公司，你的核心资产是什么？是产品代码。一个产品的代码库，是公司估值的一部分。工程师离职带不走代码，但能带走的是"知道这套代码怎么用"的隐性知识。

AI coding时代，核心资产换了——换成了**skill库**。

什么是skill？在Claude Code里，一个skill就是一个SKILL.md文件。文件里写：

· 这个能力叫什么（name）

· 什么场景触发（description + 触发词）

· 干这件事的标准动作清单

给你一个最小骨架：

---

name: client-weekly-report

description: 客户周报生成器。当用户说「写周报 / weekly / 给客户汇报」时触发。

---

## 触发词

写周报 / 客户周报 / 项目进展

## 标准动作

1. 跑 `git log --since='7 days ago'` 拉本周提交

2. 读项目 .handoff-latest.md 拿当前状态

3. 用模板 templates/weekly.html 渲染

4. 输出到 reports/YYYY-MM-DD-weekly.html

5. 不发邮件，等 PM 确认

## 输出格式

客户视角周报：本周完成 / 下周计划 / 风险 / 等客户决策事项

就这么20行，AI学会一个新能力。下次任何项目里你说"写周报"，这套动作自动跑。

我自己有**200+ 个skill**：拉项目、接手老代码、写客户邮件、跑数、巡检服务、出会议纪要、改bug……每一个都是一次性写、永久复利。

划重点：**写10分钟skill，省下接下来100次重复**。这是复利。

而代码不是复利。明天你换一个项目，你的业务代码不能直接搬过去；skill可以。

Codex团队说"Skills优先"，意思就是——把功夫花在沉淀能力上，不是沉淀代码本身。

信号3：组织 > 描述，三件套定项目
------------------

图里的原话：「重点不再是把过程描述得特别细，而是把能力、约束和目标组织清楚。」

这一条是前两条的总结，也是落地最关键的一条。

「组织清楚」这三个字，听着像废话，但具体是什么？我用过半年才摸出来——是三件套：

· **能力**：用skill来承载（信号2解决了）

· **约束**：用contracts来承载（什么不能做 / 客户铁律 / 雷区）

· **目标**：用phase + handoff来承载（现在做到哪 / 下一步做什么）

![](.evernote-content/2C42D03D-159B-44C1-BE34-D1F37C96A3FE/2A52F919-95C2-49D7-981C-1F63D17B358E.jpg)

把四个信号变成四种文件

我现在每个项目根目录长这样：

project/

AGENTS.md           # 项目完整上下文（客户/进展/技术）

contracts.md        # 铁律红线（违反 = 事故）

phase.txt           # 当前阶段，一行话

.handoff-latest.md  # 上次交接 + 下次进来第一件事

skills/             # 项目专属能力

src/                # 业务代码

上面四个markdown文件，加一起不超过300行。但AI每次进项目，读这四个文件5秒钟就回到了上次的状态。

为什么管用？因为这四个文件分别回答了项目协作里最致命的四个问题：

· **我是谁？我在哪？** → AGENTS.md

· **什么不能碰？** → contracts.md

· **现在跑到哪了？** → phase.txt

· **接下来第一件事是什么？** → .handoff-latest.md

问题答完，AI就能动手。

对比一下旧范式：你要靠会议 + 文档 + Slack历史 + 同事口口相传，才能把这四个问题答清楚。三天能上手就算快的。

---

信号4：模型进入连续协作，不是一次性执行
--------------------

图里的原话：「从一次性执行，变成能理解项目状态、一起brainstorm、持续推进。」

这是"把AI当工具"和"把AI当同事"的分水岭。

一次性执行长什么样：你给prompt，它干完一件事，下次进来重头开始。它不记得上次跟你聊过什么、上次否决了什么、客户最在意什么。

连续协作长什么样：它知道这个项目跑到哪了，知道你上次为什么把那个方案毙掉，知道客户最近三周关心的是什么。

这事怎么落地？三件套：

· **handoff.md**：每次会话结束写一段，长期追加，做"项目日记"

· **memory**：跨会话沉淀偏好 / 反馈 / 雷区，AI下次进来自动读

· **checkpoint**：长会话主动把进度落盘，然后再压缩上下文

给你一段我每次会话结束都会让AI写的 .handoff-latest.md模板：

# 交接 — {项目名} ({日期})

## 状态

{等客户反馈 / 开发中 / 待验证 / 待发送}

## 下次进来第一件事

{具体行动指令，不是描述。例如：「检查 reports/ 有没有新文件」}

## 等谁

{谁的什么反馈，预计什么时候。"无"也要写}

## 上下文（不读代码就不知道的事实）

· {事实 1}

· {事实 2}

## 试过没用的（避免下次重复踩坑）

· {方法 1}: {为什么失败}

## 危险区

· {千万别做的事 / 已知的坑}

六个固定字段，每次结束写一遍。下次任何AI进来，读这一个文件就能续上。

真实数字：本月我有**11个项目并行**，每个项目平均**8个会话**，靠这套handoff系统串起来。切换项目时，5秒内回到上次的位置。

对比旧范式：你换一个ticket，重新读代码、读注释、回想上下文，30分钟起步。

两套打法并排放一起

| 维度 | 旧范式：长Spec | 新范式：短bullet + 模型 |

| 入门产物 | 10-30页PRD + 设计文档 | 10行目标态 + 验收清单 |

一个反例：什么场景Spec反而该留长
------------------

为了避免把话说满，得讲清楚：不是所有场景都该砍Spec。

以下三种场景，长Spec仍然有不可替代的价值：

· **涉及合规 / 审计 / 法律责任**：金融、医疗、政府类项目，Spec本身就是审计材料的一部分，不写不行。

· **跨季度交付 / 多团队协作**：5个团队20个人做的大项目，Spec是同步语言，不靠它没办法对齐。

· **产品定义阶段**：一个新产品从0到1，你自己都没想清楚要做什么的时候，写长Spec的过程就是逼自己想清楚的过程。这步省不掉。

但除了这三类，剩下的80% 场景——一个人 + AI / 小团队 + AI / 内部工具 / 个人项目——bullet就够了。

判断方法很简单：**你写的那段东西，模型读完是更清楚，还是更混乱？**更清楚就留，更混乱就砍。

小团队 / 一个人怎么抄
------------

讲了一圈道理，落到地上是一份清单。

| 步骤 | 动作 | 时间成本 |
| --- | --- | --- |
| 第1步 | **项目根写AGENTS.md**  上下文 + 约束 + 目标三件套 | 30分钟 |
| 第2步 | **常重复动作沉淀成skill**  一个SKILL.md文件 = 一项能力 | 每个10分钟 |
| 第3步 | **装 .handoff-latest.md模板**  6个固定字段：状态/下次第一件事/等谁/上下文/试过没用的/危险区 | 5分钟 |
| 第4步 | **建contracts.md收口铁律**  客户口头约定一句不落 | 实时追加 |
| 第5步 | **每周看skill命中率**  0命中的直接删，留下的就是真复利 | 每周10分钟 |

表：小团队 / 一个人抄Codex范式的5步清单

五步加起来，第一周大概4-5小时投入，之后每周维护1小时。

我自己从0到200 skill，用了6个月。但前30个skill写出来时，我已经能感觉到——做事比以前快3倍不止。

一个心理障碍要先破：很多人卡在"我先把项目做完，再来沉淀"。错。沉淀和做事是一回事——你做这个任务的同时把它写成skill，下次同类任务直接调，越用越快。先做项目再沉淀，等于永远没空沉淀。

---

写在最后
----

「**君子务本，本立而道生**」——《论语·学而》

软件工程走了50年。从瀑布到敏捷，从敏捷到DevOps，每一次范式转移，本质都是把"本"换了。

瀑布时代的本，是"计划周密"；敏捷时代的本，是"快速迭代"；DevOps时代的本，是"持续交付"。

AI coding时代的本，是什么？听完Codex团队那期播客之后，我心里这个答案变得很清楚——**是「能力沉淀」**。

你的核心资产不再是代码，不再是文档，不再是某一个ticket是否做完。

而是——你身边这个AI，在跟你做完一件事之后，比之前强了多少。下次类似的事再来，它能不能自己跑完。

这个东西如果累积起来，半年后看复利惊人；如果不累积，一年之后跟今天没区别。

Codex团队亲口讲出来的这四个信号，是你提前6个月看到的山顶。

我把山路画下来了。你按这条路走。
