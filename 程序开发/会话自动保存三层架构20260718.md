# 会话自动保存到 Obsidian vault 三层架构

> 序列号：`session-1784378253-MacBook-Pro-16-M4`

## 背景

日常 Claude Code 会话中产生大量有价值信息（数据口径定义、新事实、用户纠错、工作流偏好），之前只保存到 `memory/`（Claude Code 内部记忆），用户 Obsidian 里看不到，会话关闭就"蒸发"。而且 `/compact` 和 `/clear` 命令会导致 pending 的记忆丢失。

## 方案设计

三级兜底，层层独立：

```
L1: CLAUDE.md 双写规则（我主动执行）
    ├─ 判断到有价值信息 → memory/ + wiki/capture/ 同时写入
    └─ 记不住时由 L2/L3 兜底

L2: PostToolUse hook（每步工具调用后自动）
    └─ sync-memory-to-capture.sh → idempotent 同步

L3: Stop hook（会话正常结束时）
    └─ 同上脚本，二次确认
```

### 关键决策

| 决策 | 为什么 |
|------|--------|
| 目标目录 `wiki/capture/` | 自动抓取临时区，不跟人工整理的 `cards/` 混在一起。定期筛选后迁入 `cards/analyses/concepts` |
| PostToolUse 为主力 | Stop 在 `/compact` `/clear` 时不触发。PostToolUse 每步都跑，覆盖 compact/clear/崩溃 |
| Idempotent（同名跳过） | L2 和 L3 可共存，不会重复写入 |
| 历史 memory 一次性批量同步 | 210 条旧条目首次运行时全量迁入，后续只同步新条目 |

## Hook 脚本细节

```bash
# sync-memory-to-capture.sh 核心逻辑
for f in memory/*.md（排除 MEMORY.md）:
    提取 frontmatter name/topic/title → 作为 capture 文件名
    提取 type → 作为标签
    提取 frontmatter 之后 body → 作为 capture 正文
    检查 wiki/capture/<name>.md 是否已存在 → 跳过 or 写入
```

注意：`set -e` 与 `grep` 无匹配冲突 → 脚本需用 `|| true` 兜底。

## 需要记录的经验

1. Hook 脚本必须 100% 容错——`memory/` 里有旧格式文件（无 `name:` 字段，用 `topic:`/`title:`），要 fallback 兼容
2. PostToolUse 比 Stop 更适合作为主力兜底——Stop 在 `/compact` `/clear` 时不触发
3. Idempotent 设计（同名跳过）让多层 hook 共存不冲突
4. 传统 `set -euo pipefail` + `grep` 无匹配冲突 → 改为 `grep ... || true` 模式
5. 自动记忆的"安静写入"原则（不在界面报"已保存"）依然适用，hook 脚本例外（报一次即可）
