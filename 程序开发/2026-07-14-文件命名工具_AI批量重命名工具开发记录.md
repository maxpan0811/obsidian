# 文件命名工具 - AI批量重命名工具开发记录

## 背景

需要从一堆 UUID 命名的 `.doc` 文件中，一键生成"看一眼就知道内容"的中文文件名。要求免费、本地运行、支持 8 种以上文件格式，且能配合 Codex/Claude Code 使用。

## 方案评估

| 方案 | 结论 |
|------|------|
| **smart-rename** | ❌ macOS 有 3 个致命 bug，无法使用 |
| **ai-renamer** | ⚠️ 图片为主，对表格文档弱，未实测 |
| **gemini_file_renamer** | ❌ 偏文献，不覆盖全部格式 |
| **自写 Python + Ollama** | ✅ 最终采用 |

## smart-rename macOS 兼容问题

1. **`timeout` 命令缺失**：smart-rename 使用 Linux 的 `timeout`，macOS 需 `brew install coreutils` 后用 `gtimeout`（或加 gnubin 到 PATH）
2. **模型拉取逻辑有 bug**：创建本地自定义模型后仍尝试远程 `ollama pull`，后者必然失败
3. **macOS sed 无法处理中文**：`sed: RE error: illegal byte sequence`，导致生成的含中文字符的文件名被清空只剩扩展名

## 最终方案：自定义 Python 脚本

### 工作流

```python
# 三步走：
1. textutil（macOS 自带）→ 提取老式 .doc 内容
2. pandoc → 提取其他格式（.docx/.pdf/.xlsx 等）
3. Ollama API（qwen2.5:7b）→ 两步生成：日期提取 + 内容描述
4. 代码拼合：{日期}_{描述}.ext
```

### 命名规范

- **日期在最前面**（如有明确日期），格式 `YYYY年MM月DD日`
- 所有分隔符用 **下划线 `_`**，避免与汉字"一"混淆
- 保留精确人名、地名、机构名
- 不编造日期（仅提取文档中出现的日期）

### 脚本位置

`~/.claude/skills/文件命名工具/scripts/rename_by_content.py`

### 前置依赖

```bash
brew install pandoc
brew install ollama
ollama pull qwen2.5:7b
```

### 用法

```bash
# 预览
python3 scripts/rename_by_content.py 目录/

# 执行改名
python3 scripts/rename_by_content.py 目录/ --execute

# 用 DeepSeek V4（可选）
DEEPSEEK_API_KEY="sk-xxx" python3 scripts/rename_by_content.py 目录/ --deepseek --execute
```

## 模型对比

| 模型 | 本机可用？ | 质量 | 速度 | 费用 |
|------|-----------|------|------|------|
| qwen2.5:7b (Ollama) | ✅ | 良好 | ~3s/文件 | 免费 |
| deepseek-r1:70b (Ollama) | ❌ OOM 崩溃 | - | - | - |
| DeepSeek V4 (API) | ✅ 有 key | 更好（但有余额） | ~1s/文件 | 按量付费 |

## 关键经验

- **文件名生成是"短输出"任务**，7B 模型反而比大模型更合适——大模型容易过度思考、输出冗余
- **格式代码控制**：让 LLM 只做提取/概括，Python 拼合格式，大幅提高一致性
- **品牌名校正**：模型混淆"祥龙万里行"/"翔龙万里行"，用 `BRAND_CORRECTIONS` 字典做后处理
- **不要重复跑已有好名字的文件**：LLM 每次结果有采样波动

## 后续迭代（2026-07-14 第二轮）

### 30 字截断限制

在多次实测后确认：文件名总长度不超过 **30 个汉字**（含日期前缀），超出自动截断末尾。

### 截断逻辑的 bug 修复

初次实现的截断条件为 `while date_str and cjk_chars > 30:`，导致**没有日期前缀的文件**（如 `6.1讨论问题.docx`）永远不会触发截断。修复后改为 `while cjk_chars > 30:`，所有文件统一执行。

### 符号统一为下划线

最终规范：所有分隔符用下划线 `_`，包括：

| 原符号 | 处理方式 |
|--------|----------|
| `-` 连字符 | 替换为 `_` |
| 空格 | 替换为 `_` |
| `,` `，` 逗号 | 替换为 `_` |
| `、` 顿号 | 替换为 `_` |
| `/` 斜杠 | 替换为 `_` |
| `;` `：` 等 | 替换为 `_` |

### 实测数据

| 批次 | 文件数 | 格式 | 原始命名 | 处理后 |
|------|--------|------|----------|--------|
| 测试 | 6 | .doc | UUID | 语义化中文名 |
| 第1批 | 44 | .doc/.docx | 编号+UUID | 课程名+日期前缀 |
| 第2批 | 23 | .doc/.docx | UUID | 业务文档名+日期前缀 |

### 已知局限

- **模型采样波动**：同一文件多次运行可能产生不同文件名，已有好名字的文件不要重跑
- **长清单文档**：签证材料、定价规则等清单式文档，模型倾向把条款全列进去，30 字截断会导致信息丢失
- **空格问题**：LLM 有时在描述中插入空格，`sed` 替换 `_` 后可能产生双下划线，脚本已通过多重清理链处理
