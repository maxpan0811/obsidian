## 2026-07-02 — 💡 关键发现：本地数据库直接同步（攻克限流）

### 背景

前面所有管线（sync_favorites.py、fast_export.py、rest_export.py）都依赖 Evernote 云 API。Thrift API 每次调用间隔至少 4s，每 ~700 次限流 15 分钟，17k 篇笔记全量导出需要 24-40 小时。

灵感来自用户的提问：**"我的印象笔记内容都是保存在本地的，为什么不直接从本地同步？"**

### 发现

检查本地数据目录发现的是：

```
~/Library/Application Support/com.yinxiang.Mac/accounts/app.yinxiang.com/207087/
├── localNoteStore/LocalNoteStore.sqlite  ← CoreData 数据库，252MB
│   ├── ZENNOTE       — 17,183 条活跃笔记
│   ├── ZENNOTEBOOK   — 4 个笔记本
│   └── ZENRESOURCE   — 全部资源元数据
└── content/  ← 49,808 个目录
    └── {ZLOCALUUID}/
        ├── content.enml         ← 完整 ENML 正文
        ├── {resource_guid}.jpg  ← 原始图片文件
        └── snippet.txt / card.png / quickLook.png
```

### 映射关系

| 概念 | Evernote API | 本地数据库 |
|------|-------------|-----------|
| 笔记 ID | GUID | `ZENNOTE.ZGUID` |
| 本地路径 | — | `ZENNOTE.ZLOCALUUID` → `content/{uuid}/` |
| 正文 | getNote().content | `content/{uuid}/content.enml` |
| 图片 | getNote().resources[].data.body | `content/{uuid}/{resource_guid}.jpg` |
| hash 桥接 | `<en-media hash="...">` | `ZENRESOURCE.ZDATAHASH`（hex 小写匹配） |
| 笔记本 | notebookGuid | `ZNOTEBOOK → ZENNOTEBOOK.ZGUID` |

已验证：en-media hash（小写）与 `hex(ZDATAHASH)`（大写）做了大小写无关匹配，5 篇测试中的 hash 全部成功对应到本地资源文件。

### 关键数据

| 来源 | Favorites 笔记数 |
|------|:---------------:|
| Thrift API findNotesMetadata | 17,421 |
| 本地 SQLite（ZACTIVE=1） | 17,183 |
| content/ 目录 | 49,808 |

17,421 vs 17,183 的差值（238 篇）是已在云端但未完全同步到本地的笔记（如近期新增）。

### local_sync.py 设计

| 笔记类型 | 处理方式 |

| 纯文本（无 en-media） | `content.enml` → ENML→Markdownify → `.md` |
| 含图片 | 同上 + 从 `content/{uuid}/` 复制图片到 `attachments/` |
| 含 PDF/Word 附件 | **不导出 .md**，已有 .md 移入废纸篓 |
| 重复标题 | 暂未处理（后续对齐 reconcile 逻辑） |

**性能：** 无网络调用，无限流，纯磁盘 IO。预计 17,000 篇 + 图片在 10-30 分钟内完成。

## 2026-07-04 — Obsidian 白屏排查：非点号符号链接导致渲染崩溃

### 问题

Obsidian 1.12.7 打开 vault 后一片白色空白（无侧边栏、无按钮），即使删除了 71GB attachments 后仍出现。

### 排查过程

| 步骤 | 操作 | 结果 |
|------|------|:----:|
| 1 | 新建空库测试 | ✅ 正常 → 确认不是 Obsidian 安装问题 |
| 2 | 删除 `.obsidian` 配置 | ❌ 白屏依旧 |
| 3 | 删除 `~/Library/Application Support/obsidian/` 应用缓存 | ❌ 白屏依旧 |
| 4 | `--disable-gpu` 启动 | ❌ 白屏依旧 |
| 5 | 重装 Obsidian | ❌ 白屏依旧 |
| 6 | 移走 `印象笔记管理工具.bak` 目录 | ✅ 正常 → 锁定问题在此目录 |
| 7 | 17K .md 二分法（4300→2150 逐半排除） | ❌ 每次都白屏 → 排除单篇文件损坏 |
| 8 | **移走 `_evernote-content.bak` 符号链接** | ✅ **秒开** → 元凶确认 |

### 根因

`_evernote-content.bak`（**非点号开头**）符号链接指向 Evernote 本地 content 目录（约 **50,000 个子目录**）。Obsidian 在启动渲染时遍历这个符号链接目标，5 万目录导致 Electron 渲染进程崩溃，整窗白屏。

Obsidian **不遍历**点号开头的文件/目录，但**会遍历**非点号开头的符号链接及其目标路径。

### 修复

将符号链接名改为**点号开头**（`.evernote-content`），Obsidian 跳过点号开头的文件/目录遍历。vault 恢复正常秒开。

**关键教训：**
- Obsidian 白屏不要只怀疑 vault 内容损坏。**符号链接目标遍历也会导致渲染崩溃**
- 点号开头的文件/目录在 Obsidian 中既是"文件树隐藏"，也是"**索引跳过**"
- 17,220 个 .md + 点号符号链接 = **秒开** ✅
- 旧 `local_sync.py` 生成符号链接时命名正确（`.evernote-content`），但手动备份时重命名为 `_evernote-content.bak`（去掉点号）触发了崩溃
- Obsidian（Electron）渲染进程在启动时遍历 vault 内所有**非点号开头**的文件和目录。符号链接被当作普通目录对待，其目标也被遍历
