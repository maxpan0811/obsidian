## 关键发现

### 1. qwen3-vl:30b 是 MoE 架构，非稠密 30B

- `ollama show qwen3-vl:30b` → `architecture: qwen3vlmoe, 31.1B total, Q4_K_M`
- 实际激活参数仅 ~3B（A3B 规格），M4 Max 64G 推理速度 88.5 tok/s
- DeepSeek 按稠密 30B 估算 15-20s 偏悲观，实测小图全流程 15s
- **结论：** 不需要补轻量副驾模型——30B MoE 本身就是甜点

### 2. 加速三板斧

| 方法 | 效果 | 实现 |
|------|------|------|
| ThreadPoolExecutor 并发 4 | ~2.4× 加速 | Ollama 共享权重，4 请求并行 |
| 大图 sips 缩放 | ~30% | 1MB+ 图片缩到 2248px 短边 |
| Checkpoint + mapping | 断点秒续跑 | 存 old→new 映射，--resume --execute 直接 rename |

### 3. 实战踩坑

**坑 1：两个 batch 同时跑撞车**
第一次测速手滑跑了两个 4-worker batch，8 个进程抢同一模型，单张从 15s 拖到 40-180s。
→ **教训：** 只跑一个 batch，不重复提交。

**坑 2：后台任务输出被缓冲**
`run_in_background` 无 TTY，Python print 不 flush。日志延迟显示，看不到实时进度。
→ **修复：** `PYTHONUNBUFFERED=1` 或 `python3 -u`。

**坑 3：已命名的文件被重复改名**
用户把上一轮改好的文件又放进了 temp。脚本没识别已命名文件，重新分析产生略不同的名字，和原文件冲突后自动加 `(2)` 后缀。
→ **修复：** 新增 `ALREADY_NAMED_PATTERN` 正则匹配 `^\d{4}年.*[一-鿿]`，匹配的跳过不改。
→ **待改进：** 当前只匹配 `YYYY年MM月DD日_` 完整格式，`YYYY年_` 短格式漏掉了。

**坑 4：Checkpoint mapping 存反了**
第一次 checkpoint 存的是 `new→old`，续跑时以为已完成直接跳过。
→ **修复：** 改为 `old→new` 映射，续跑直接 os.rename。

**坑 5：文件列表锁定后源文件消失**
脚本启动时收集文件列表，等排到执行时源文件已被前一轮 rename 改掉/删掉，`os.rename` 崩溃。
→ **修复：** 加 `try/except FileNotFoundError` 跳过不存在的文件。

**坑 6：`qwen3-vl:14b` 标签不存在**
Ollama registry 上 `qwen3-vl` 系列只有 `:30b` 一个版本，所有其他 tag（14b/8b/7b/32b/70b）全 404。
→ **结论：** Qwen3-VL 只出了 MoE 单一大尺寸。

## 改进方向

1. 已命名跳过正则放宽到 `^\d{4}年` 开头即跳过（不只完整年月日）
2. 文件列表锁定改为动态扫描——每次取 next pending file 而不是启动时锁死
3. 考虑加 `--max-results` 限制输出文件名长度，减少 tok 消耗
4. batch rename 支持 `--revert` 从 checkpoint mapping 恢复原名
